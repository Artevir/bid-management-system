# 中文标题-20260413-V1.0

> 文档日期：2026-04-13  
> 主题：Linux 生产构建与依赖安装说明；Coze S3 相关 SDK 的按需加载与 Next 服务端外部化配置

---

## 1. 背景与目标

在 Linux 服务器（如 `/www/wwwroot/bid-management-system`）执行 `pnpm db:push` 与 `pnpm build` 时，曾出现两类典型问题：

1. **esbuild 平台不匹配**：提示在错误平台安装了 `esbuild` 的二进制（常见于从 Windows 拷贝 `node_modules` 或在非目标环境预装依赖）。
2. **Next.js 构建阶段报错**：`TypeError: Class extends value undefined`，且指向收集页面数据时的 API 路由（例如 `/api/interpretations/[id]/checklist`）。

本文说明根因与已在代码侧采取的缓解措施，并给出部署侧的标准操作流程。

---

## 2. 根因摘要

| 现象 | 根因 |
|------|------|
| `You installed esbuild for another platform...` | `node_modules` 未在**当前运行系统**上安装，或跨系统复制了依赖目录。 |
| `Class extends value undefined`（构建期） | `coze-coding-dev-sdk` 在**模块顶层**被加载/实例化，经 Next 打包后在服务端预检阶段触发 SDK 内部不兼容行为。 |

---

## 3. 代码变更说明（2026-04-13）

### 3.1 按需动态加载 Coze `S3Storage`

- **`src/lib/interpretation/service.ts`**：移除顶层 `import { S3Storage } from 'coze-coding-dev-sdk'`；仅在 `loadDocumentBuffer` 中且已配置 `COZE_BUCKET_*` 时执行 `await import('coze-coding-dev-sdk')` 再实例化。
- **`src/lib/file/service.ts`**：移除模块加载时即 `new S3Storage(...)` 的逻辑；改为 `getCozeBucketClient()`，内部对 `import('coze-coding-dev-sdk')` 做 Promise 级缓存，各业务函数在需要时再 `await getCozeBucketClient()`。

目的：避免在 **Next「Collecting page data」** 等阶段仅因加载路由依赖树就执行有问题的 SDK 顶层副作用。

### 3.2 Next 配置：`serverExternalPackages`

- **`next.config.ts`**：增加 `serverExternalPackages: ['coze-coding-dev-sdk']`，让该包尽量以 Node 原生方式在服务端解析，降低与 Webpack 打包链的互操作问题。

---

## 4. 服务器部署推荐步骤（Linux）

在**目标服务器**项目根目录执行（勿从本机复制 `node_modules`）：

```bash
cd /www/wwwroot/bid-management-system   # 按实际路径调整
git pull origin main
rm -rf node_modules .next
pnpm install
pnpm db:push
pnpm build
```

说明：

- **`rm -rf node_modules`** 后在本机执行 **`pnpm install`**，可确保 `@esbuild/linux-x64` 等与当前内核一致。
- 清理 **`.next`** 可避免旧产物与当前依赖不一致导致的偶发错误。

---

## 5. 环境变量提示（与 S3 相关）

若使用 Coze 兼容桶读取/上传，需配置（示例名称以项目为准）：

- `COZE_BUCKET_ENDPOINT_URL`
- `COZE_BUCKET_NAME`

未配置时，解读与文件服务应走本地 `uploads` 等路径逻辑，不应在启动或构建阶段强制加载 SDK。
