# 交付整改计划-2026-04-14-V1.0

## 目标
- 以交付为目标，分阶段完成安全封堵、业务一致性修复、工程治理。
- 先修复 P0 阻断项，再推进 P1 高风险项，最后进行 P2 治理。

## 已完成（本轮）
- 上传接口接入鉴权，分片合并路径限制到 `uploads/chunked` 白名单目录。
- 管理接口加固：`/api/admin/rbac/init`、`/api/admin/db/indexes` 改为管理员权限。
- 电子签章回调增加签名校验（生产环境强制验证 `E_SIGN_CALLBACK_SECRET`）。
- 审批执行逻辑加固：禁止越级审批、修复终审无法完结问题。
- 审批前后端契约修复：前端执行审批时携带 `level`，后端支持缺省自动解析当前级别。
- 项目与解读数据权限收敛：仅可访问自己参与/上传的数据。
- 回收站详情与操作按删除人隔离，避免跨用户读取与删除。

## P0（立即完成，阻断上线）
1. 接口安全封堵
   - 所有高危管理接口统一接入管理员权限校验。
   - 所有文件上传链路禁止客户端指定任意落盘路径。
2. 审批主流程一致性
   - 审批状态机强约束：仅允许当前节点审批，前置节点必须已通过。
   - 终审节点自动收敛到文档 `approved`，避免流程卡死。
3. 数据横向越权
   - 项目、解读、回收站统一加用户归属过滤。

## P1（1 周内完成）
1. API 默认拒绝策略
   - 对 `/api/*` 引入统一门禁策略（公开白名单 + 默认拒绝）。
2. 事务与幂等
   - 工作流、文档生成、审批执行等多步写操作补齐事务边界与幂等键。
3. 软删除语义统一
   - 统一 `isDeleted/deletedAt/deletedBy` 语义，避免状态字段混用删除语义。
4. 契约统一
   - 前后端统一返回结构（`success + data + pagination`），清理 `items/data.items` 混用。
5. 观测与错误治理
   - 统一错误处理器与结构化日志（requestId/userId/path/errorCode）。

## P2（版本内治理）
1. 类型安全治理：逐步消除关键链路 `any/as any`。
2. 双轨模块收敛：统一 `e-sign/esign`、`audit`、`search` 单一真源。
3. 测试门禁提升：E2E 主链路纳入 CI 强制门禁。
4. UI 一致性：统一空态/错态组件，补齐 icon 按钮 `aria-label`。
5. 环境治理：生产环境配置 fail-fast，禁止敏感默认值回落。

## 角色与责任
- 架构师：统一鉴权网关、状态机与服务边界。
- 研发经理：任务拆解、代码规范与提测节奏。
- DBA：并发一致性、索引与软删除策略校准。
- 测试负责人：端到端回归矩阵与发布门禁。
- 产品经理：冻结交付范围与接口契约标准。
- UI 负责人：交互一致性与可访问性验收。
- 实施经理：灰度/回滚/上线核验与环境治理。
