1）res.data.xxx → res.xxx
这个是高频且相对安全的一类，但仍建议先 grep 再替换。你已经在 bid/page.tsx 上碰到过。
先查
grep -R "res\.data\." -n src/app src/lib
批量替换
find src/app src/lib -type f \( -name "*.ts" -o -name "*.tsx" \) -print0 | xargs -0 perl -pi -e 's/\bres\.data\./res./g'
风险说明
这个替换只适合你已经确认 service 返回值是已解包对象的仓库。如果某些地方 res 真的是原始 axios/fetch 响应，这一刀会误伤，所以务必先看 grep 结果。
2）旧 schema 路径 → 统一 schema 导入
你前面已经确认真实表定义集中在 src/db/schema.ts，而 audit-logs 在 src/lib/db/schema/audit-logs.ts。
单个路径替换：audit-logs
grep -R "@/db/schema/audit-logs" -n srcfind src -type f \( -name "*.ts" -o -name "*.tsx" \) -print0 | xargs -0 perl -pi -e "s#@/db/schema/audit-logs#@/lib/db/schema/audit-logs#g"
单个路径替换：projects/documents/companies 拆分文件
先查：
grep -R "@/db/schema/projects\|@/db/schema/documents\|@/db/schema/companies" -n src
这些不建议“一刀切自动改成同一个 import 行”，因为：
documents 实际要改成 bidDocuments 
每个文件原来 import 组合不同 
相对安全的做法：先删旧 import，再人工补新 import
find src -type f \( -name "*.ts" -o -name "*.tsx" \) -print0 | xargs -0 sed -i "/@\/db\/schema\/projects/d"find src -type f \( -name "*.ts" -o -name "*.tsx" \) -print0 | xargs -0 sed -i "/@\/db\/schema\/documents/d"find src -type f \( -name "*.ts" -o -name "*.tsx" \) -print0 | xargs -0 sed -i "/@\/db\/schema\/companies/d"
然后再按文件补：
import { projects, bidDocuments, companies } from '@/db/schema';
3）request.ip 清理
这是非常适合批量改的一类。你已经在登录 API 上遇到过。
先查
grep -R "request\.ip" -n src
粗暴删掉 request.ip ||
find src -type f \( -name "*.ts" -o -name "*.tsx" \) -print0 | xargs -0 perl -pi -e "s/\s*request\.ip\s*\|\|\s*//g"
更推荐：人工改成标准 IP 获取模板
const ipAddress =  request.headers.get('x-forwarded-for')?.split(',')[0]?.trim() ||  request.headers.get('x-real-ip') ||  request.headers.get('cf-connecting-ip') ||  'unknown';
这类建议不要完全自动替换，因为每个文件变量名未必都叫 ipAddress。
4）import { cache } from '@/lib/cache' → redis-client
这个是高频且相对适合批量替换。你已经查到很多文件这么写。
先查
grep -R "import { cache } from '@/lib/cache'" -n src
批量替换
find src -type f \( -name "*.ts" -o -name "*.tsx" \) -print0 | xargs -0 perl -pi -e "s#import \{ cache \} from '@/lib/cache';#import { redis as cache } from '@/lib/cache/redis-client';#g"
处理默认 redis 导入
先查：
grep -R "import redis, .*from '@/lib/cache'" -n src
再替换：
find src -type f \( -name "*.ts" -o -name "*.tsx" \) -print0 | xargs -0 perl -pi -e "s#from '@/lib/cache';#from '@/lib/cache/redis-client';#g"
风险说明
这条会把所有 from '@/lib/cache' 改成 redis-client，包括不该改的聚合 API 导入。所以更稳是先 grep 精确命中后，按文件替换。
5）旧 LLM 直接调用 → service 层调用

6）Next.js 15 动态路由 params
你目前这类已经基本清掉了，但如果以后再来一轮，很适合半自动处理。
先查
grep -R "params }: { params: {" -n src/app/api
第一刀：把类型包装成 Promise
find src/app/api -type f -name "route.ts" -print0 | xargs -0 perl -pi -e 's/\{ params \}: \{ params: \{ ([^}]+) \} \}/{ params }: { params: Promise<{ $1 }> }/g'
第二刀：const { id } = params → await params
这类变量名不统一，不建议全自动。先查：
grep -R "const { .* } = params" -n src/app/api
再人工逐个改。
