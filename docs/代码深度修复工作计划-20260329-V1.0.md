# 代码深度修复工作计划-20260329-V1.0

**版本**: V1.0  
**日期**: 2026-03-29  
**目标**: 基于 `docs/代码修复-20260329-V1.0` 的批量修复方法论，对仓库做系统性、可持续的全盘修复，避免“见一个错修一个”的反复返工。

## 一、验收口径（Definition of Done）

1. **构建全绿**：`pnpm build` 通过（包含 TypeScript 类型检查阶段）。
2. **关键链路可用（冒烟）**：
   - 登录页可打开并可登录
   - 权限相关接口可用（至少不因导入/类型问题崩溃）
   - 标书列表 / 创建文档 / 编辑页可进入
   - 章节生成接口（流式/非流式）可调用
3. **漂移源被收口**：auth/cache/schema/llm 的导入与调用口径统一，不再出现“引用不存在模块/接口漂移”类问题。
4. **修复可复现**：每一轮修复都能用 grep 定位命中 → 按规则替换 → build 验证。

## 二、顶尖团队协作分工（职责清晰、一次只做一类）

### 1）产品经理（PM）
* 统一优先级：P0（阻断构建/登录/核心业务）优先，P1（体验/性能）后置。
* 定义冒烟验收清单与“可以上线”的最低标准。

### 2）测试经理（QA Lead）
* 为每个修复类别给出最小可执行验证步骤（构建 + 冒烟 API/页面）。
* 在“每轮只修一类”的节奏下，维护回归点列表（避免旧问题反复出现）。

### 3）研发经理（Eng Manager）
* 严格控制范围：每轮只修一个类别，禁止跨类别混修。
* 统一修复节奏：grep → 修改 → build → 记录下一类。

### 4）技术负责人（Tech Lead）
* 制定“工程契约”：Next.js 15 路由签名、中间件返回类型、模块导出入口。
* 决策“收口策略”：优先选择 `Response` 作为统一返回类型以兼容 `NextResponse` 和流式 `Response`。

### 5）后端负责人（Backend Lead）
* auth/cache/schema 导入漂移收口，保障 API 路由构建稳定。
* 审计日志、反代头（IP/UA）等生产可观测字段稳定。

### 6）前端负责人（Frontend Lead）
* 处理 TypeScript 类型稳定性（`never[]`、`res.data` 漂移、缺失 hook 等）。
* 运行时白屏链路排查（UI 组件依赖/Server-Client 边界）。

### 7）LLM 负责人（LLM Lead）
* 统一 LLM 调用口径：**流式走 `streamChat`，非流式走 `invokeChat`**。
* 禁止直接依赖 adapter 实例上的 `chat/chatStream` 这类易漂移接口。

### 8）DevOps/SRE
* 保证生产环境与构建环境一致（Node/pnpm/环境变量/反代头）。
* 固化部署步骤：`git pull` → `pnpm install` → `pnpm build` → `pnpm start`。

## 三、修复节奏（每轮固定流程）

1. **按类别 grep 扫描**
2. **只修这一类的所有命中点**
3. **执行 `pnpm build`**
4. **记录下一轮出现的新类别错误**
5. **写修复报告（存入 docs，按版本递增）**

## 四、按类别的修复路线图（一次只做一个类别）

### 类别 1：框架契约（P0）
**范围**：
* Next.js 15 动态路由参数签名（`params` 使用 Promise）
* 中间件链返回类型统一为 `Response`

**扫描命令**：
```bash
grep -R "params }: { params: {" -n src/app/api
grep -R "Promise<NextResponse" -n src/lib/auth src/app/api src/lib
```

**修复规则**：
* `params: { id: string }` → `params: Promise<{ id: string }>`，并 `await params`
* 中间件 handler 返回类型：`Promise<NextResponse>` → `Promise<Response>`

### 类别 2：中间件调用方式（P0）
**范围**：`withDocumentPermission/withChapterPermission/withResourcePermission` 等 async 工厂函数的调用方式。

**扫描命令**：
```bash
grep -R "withDocumentPermission(" -n src
grep -R "withChapterPermission(" -n src
grep -R "withResourcePermission(" -n src
```

**修复规则**：
* 错误：`withDocumentPermission(...)(request, handler)`
* 正确：
  - `const middleware = await withDocumentPermission(...)`
  - `return middleware(request, handler)`

### 类别 3：模块导出漂移收口（P0）
**范围**：auth/cache/schema/llm 导入口径统一，消灭 `Module not found` 与“接口漂移”。

**扫描命令**：
```bash
grep -R "requireAuth\\|withAdmin\\|clearPermissionCache" -n src
grep -R "from '@/lib/cache'\\|from \"@/lib/cache\"" -n src
grep -R "@/db/schema/" -n src
grep -R "\\.chatStream(\\|\\.chat(" -n src/app/api src/lib | head -n 200
```

### 类别 4：LLM 口径统一（P0）
**规则**：
* 流式：`streamChat({ messages }, forwardHeaders)`
* 非流式：`invokeChat({ messages }, forwardHeaders)`

### 类别 5：前端类型稳定性（P1）
**范围**：`res.data` 漂移、`never[]`、缺失 hook/变量等。

**扫描命令**：
```bash
grep -R "res\\.data\\." -n src/app src/lib
grep -R "queryClient\\." -n src
```

### 类别 6：运行时稳定性（P1）
**范围**：UI 组件依赖链、server/client 边界、webpack module factory 异常等。

### 类别 7：工具链/文档（P2）
**范围**：OpenAPI 生成类型、脚本固化、排查命令包等。
