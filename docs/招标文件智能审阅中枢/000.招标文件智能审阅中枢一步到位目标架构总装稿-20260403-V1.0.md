**一、总体定位**

**1.1 系统正式定位**

本系统正式定义为：

**招标文件智能审阅中枢**

它不是一个“解读结果展示页”，也不是一个“AI 总结工具”，而是：

**招标要求资产生产中心 + 投标编制上游中枢**

它的核心使命只有一句话：

**把非结构化招标文件，转化为可复用、可校验、可追溯、可联动、可驱动后续投标文件自动生成的标准化招标要求资产。**

---

**1.2 本中枢服务的下游对象**

它未来直接服务这几类模块：

- 投标文件框架识别模块

- 模板识别与变量抽取模块

- 材料清单与任务分派模块

- 风险审阅模块

- 评分拆解模块

- 技术偏离表生成模块

- 标书正文生成模块

- 企业文档规范资产池

- 生成装配规则资产池

但注意：

**当前这个中枢只负责“招标侧资产”**

不负责：

- 企业样式规范维护

- 最终 Word 版式装配

- 最终整本标书导出成稿

**二、一步到位的核心设计原则**

**原则 1：不是结果展示模型，而是资产中枢模型**

旧思路是“解析出一堆结果给人看”。\
新思路是“生产一批标准资产，给人看，也给系统用”。

**原则 2：不是单一 Interpretation 大对象，而是主对象群**

不再用一个大 JSON 包所有结果，而是拆成多个领域主对象。

**原则 3：内容、结构、模板、变量、风险、材料、复核必须分对象建模**

否则后续无法联动生成。

**原则 4：所有关键资产都必须可溯源**

必须保留：

- 原文锚点

- 来源页码

- 来源文件

- 解析批次

- 置信度

- 规则命中

- 人工确认状态

**原则 5：所有对象都要预留 AI 任务与人工复核双轨承载位**

因为你要的是 AI Coding 一步到位，不代表系统不需要治理。\
系统必须支持：

- AI 自动抽取

- 规则自动校验

- 人工最小确认

- 留痕复盘

**原则 6：对象全盘设计，不按首期裁剪**

字段一步到位设计，后续只是“启用哪些”，不是“再补模型”。

**六、和现有技术栈的映射结论**

这套设计完全可以放进你现有栈里，不需要换栈。

**前端**

- Next.js App Router

- React 19

- shadcn/ui

- TanStack Query

- TanStack Table

- Zustand 仅做轻状态

**后端**

- Route Handlers

- Service 分层

- 任务编排独立目录

- 规则引擎独立目录

- AI Provider 适配层独立目录

**数据层**

- PostgreSQL 16 做主存储

- Drizzle ORM 做 schema 与查询

- JSONB 只承载灵活字段，不承载主结构

- 所有核心对象尽量结构化建表

**文档处理**

你现有依赖可以继续保留：

- PDF：pdf-parse

- DOC：word-extractor

- DOCX：mammoth

- XLS/XLSX：xlsx

- 导出：docx / pdfmake

这和你现在的技术文档是一致的，只是业务模型换了。
