**020.招标文件智能审阅中枢数据库逻辑关系、主外键与索引设计总表-20260403-V1.0**

**一、编制目的**

《020.招标文件智能审阅中枢数据库逻辑关系、主外键与索引设计总表-20260403-V1.0》用于在《010.招标文件智能审阅中枢核心对象模型与主表逻辑总表-20260403-V1.0》已经明确“招标文件智能审阅中枢”核心对象体系、主表清单与总体关系的基础上，进一步将各主表之间的数据库逻辑关系、主键与外键设计、唯一性约束、典型索引策略、查询路径和数据治理要求正式固化下来，为后续 Drizzle Schema 设计、PostgreSQL 表结构落地、接口开发、页面查询、AI 任务写入、规则引擎执行和人工复核留痕提供统一底板。

本文件不是传统意义上的数据库物理实施手册，也不是面向 DBA 的细粒度运维文档，而是“招标文件智能审阅中枢”在 AI Coding 场景下的一份目标级数据库逻辑总装稿。其目标不是分阶段、按期次逐步修补，而是在保持现有技术栈不变的前提下，一次性把中枢级数据库逻辑关系定准、定稳、定清楚，使后续所有实现都围绕统一结构展开，而不再回到结果型单表、临时 JSON 聚合或页面驱动建模的旧思路。当前既有技术栈仍继续采用 Next.js、React、Node.js、PostgreSQL、Drizzle ORM 及现有文档解析依赖体系。

同时，本文件延续你前面已经确立的中枢型设计思想：模型接入层、任务编排层、规则层和业务应用层必须解耦，真正可沉淀的长期资产必须沉淀在平台自身，而不是绑死在底座模型或临时实现里。数据库设计也必须体现这一点，即：原文结构、要求资产、风险治理、模板变量、规则命中、复核留痕和输出快照必须分层建模，而不能揉成一张“大结果表”。

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“数据库逻辑总控章”。其上承核心对象模型，其下直接约束以下内容：

第一，约束 Drizzle Schema 的主键、外键、索引、唯一约束与枚举结构。\
第二，约束后续 API 查询路径和页面取数路径。\
第三，约束 AI 任务写入路径、规则命中路径、人工复核路径和下游快照输出路径。\
第四，约束后续与企业文档规范资产池、生成装配规则资产池之间的接口口径。

如果没有本文件，后续实现极易出现以下问题：一是表与表之间虽然都有，但关系链不清，导致查询混乱；二是外键缺失或过多，导致一致性不足或性能劣化；三是索引设计只围绕页面临时需求，不围绕中枢核心链路；四是 AI 任务、规则命中、复核留痕和快照输出没有统一主线，最终造成系统虽有数据，但难以稳定复用。

因此，本文件要解决的不是“有没有表”，而是“这套表如何成为一套真正可运行的中枢数据库骨架”。

**三、总体数据库设计原则**

**3.1 目标形态一次性设计原则**

数据库逻辑关系不按首期、二期、后续扩展分段设计，而按目标形态一次性建好。后续差异仅体现为功能启用顺序、页面开放顺序、任务启用顺序，不再体现为主表逻辑反复改道。

**3.2 主对象优先、桥表清晰原则**

凡核心业务对象，必须有独立主表；凡多对多关系，必须通过桥表明确表达；凡不同层级的信息，不得通过一个聚合 JSON 表“糊”在一起。

**3.3 原文溯源必达原则**

数据库逻辑必须确保：要求、风险、评分、技术参数、模板、变量、材料等关键对象，尽量都能追溯到 source_segment，至少能追溯到 source_document 和 document_page。否则页面高亮、跳转原文、复核确认和规则解释都无法真正成立。

**3.4 查询链路稳定原则**

索引设计必须围绕中枢的核心查询路径展开，而不是围绕零散页面做局部优化。查询优先级应围绕以下几条主线：项目查版本、版本查文件、版本查要求、版本查风险、版本查框架、版本查模板、对象查复核、版本查快照。

**3.5 强一致与高弹性平衡原则**

核心结构字段必须结构化建模；复杂原始内容、表格结构、规则命中详情和输出快照可用 JSONB 承载。不能因为图省事把核心结构塞进 JSONB，也不能为了纯粹“关系型洁癖”把所有复杂嵌套内容打散得过细。

**3.6 审计留痕统一原则**

所有关键对象的人工确认、争议处理、结果覆盖、规则触发和下游输出，都必须可追溯。相关留痕统一由 review_task、object_change_log 和 asset_export_snapshot 这三类表承载。

---

**四、数据库总体逻辑分层**

结合对象模型与中枢架构，建议数据库逻辑分为六层。

**4.1 项目与版本主控层**

以 tender_project 为根，以 tender_project_version 为版本分界，统一管理同一项目的多文件、多批次、多结果。

**4.2 原文结构层**

由 source_document、document_page、source_segment、document_section_node 组成，是所有结构化抽取、溯源定位、章节树构建的原始数据基础。

**4.3 招标要求资产层**

由 tender_requirement 及其资格、商务、技术、时间、金额、提交等专门化对象组成，承接“招标要求本身”。

**4.4 风险与规则治理层**

由 risk_item、conflict_item、rule_definition、rule_hit_record、confidence_assessment 组成，承接“自动判断与自动校验”。

**4.5 框架、模板与材料层**

由 bid_framework_node、framework_requirement_binding、bid_template、template_variable、submission_material 等组成，承接“后续标书编制所需的上游骨架”。

**4.6 任务、复核与输出层**

由 document_parse_batch、ai_task_run、review_task、object_change_log、asset_export_snapshot 组成，承接“运行过程、人工治理和对下游交付”。

---

**五、主外键设计总原则**

**5.1 主键设计原则**

全部业务主表统一使用单字段主键 id。\
建议实现层使用 bigint 自增或统一字符串型 ID，但在一个中枢内不应混用两套主键风格而无规则。若当前项目偏向 Drizzle + PostgreSQL 的稳定实现，我建议核心业务表优先采用 bigint 主键，便于排序、索引和排查；对需要外部公开暴露的对象，可在表中增加独立的 biz_id 或 public_id 字段。

**5.2 外键设计原则**

外键统一采用“表意明确、层级单向”的原则。\
凡上层对象对下层对象存在明确归属关系，应设置外键；凡仅用于灵活引用、可跨对象类型变化的场景，优先采用“对象类型 + 对象 ID”的多态引用，而不强上数据库级外键。

**5.3 唯一性设计原则**

凡业务上具有天然唯一性的对象，应优先通过唯一约束明确表达，而不是依赖代码逻辑兜底。例如：

- 同一项目下版本号应唯一。

- 同一版本下同一文件校验值可唯一。

- 同一版本下同一框架节点路径可唯一。

- 同一模板下同名变量应唯一。

- 同一解析批次下同类任务名可唯一。

**5.4 删除策略原则**

建议全系统以**逻辑删除优先**为原则，重要业务主表统一增加 is_deleted、deleted_at、deleted_by 等字段或采用统一软删除机制。\
对核心对象不建议直接物理删除，尤其是：项目、版本、文件、要求、风险、模板、复核、快照、日志。

**九、Drizzle Schema 设计建议**

从 AI Coding 落地角度，Drizzle Schema 建议按对象域拆文件，不建议把全部 38 张表堆进一个 schema 文件。建议目录结构如下：

src/db/schema/tender-center/

project.ts

source.ts

section.ts

requirement.ts

risk.ts

scoring.ts

technical.ts

template.ts

material.ts

ai.ts

review.ts

snapshot.ts

enums.ts

relations.ts

其中：

- enums.ts 统一承载状态类、类型类枚举。

- relations.ts 统一承载 Drizzle relations 定义。

- 每个领域文件只定义本域主表和桥表。

- 多态引用对象不在 Drizzle relations 中强行做伪外键关系，而由 service 层统一处理。

这样既符合你现在的 AI Coding 节奏，也避免后续 schema 文件失控。

**十、数据一致性与性能平衡建议**

**10.1 哪些关系建议强外键**

建议强外键的，主要是层级稳定、删除逻辑清晰的关系，例如：

- 项目与版本

- 版本与文件

- 文件与页

- 页与分片

- 方案与评分项

- 模板与变量

- 参数组与参数项

**10.2 哪些关系建议弱外键或多态引用**

建议用多态引用的，主要是横跨多对象类型、后续灵活性高的场景，例如：

- review_task.target_object_type + target_object_id

- object_change_log.target_object_type + target_object_id

- rule_hit_record.target_object_type + target_object_id

- response_task_item.source_object_type + source_object_id

这样可以减少数据库层面的僵硬耦合。

**10.3 哪些字段建议 JSONB**

推荐 JSONB 的字段包括：

- source_segment.bbox_json

- rule_definition.expression_json

- rule_hit_record.hit_detail_json

- bid_template.template_table_json

- template_block.block_table_json

- form_table_structure 中若需要复杂合并单元格信息可补 JSONB

- ai_task_run.input_ref_json / output_ref_json

- review_task.final_value_json

- object_change_log.before_json / after_json

- asset_export_snapshot.snapshot_json

**10.4 哪些字段必须结构化**

以下字段不应偷懒放 JSONB：

- 风险等级

- 要求类型

- 时间值

- 金额值

- 必填标记

- 必选/可选标记

- 模板类型

- 评分值

- 技术星号标记

- 允许偏离标记

- 复核状态

**十一、对后续实现的直接约束**

基于本文件，后续 AI Coding 必须遵守以下规则：

第一，所有接口和 service 的主查询路径必须围绕本文件确定的主外键关系展开，不允许为了省事直接扫全表或拼大 JSON。\
第二，所有页面查询必须优先使用对象表和桥表，不得临时从日志表、快照表反向拼业务主数据。\
第三，所有复核、确认、关闭、覆盖动作都必须写入 review_task 和 object_change_log。\
第四，所有对下游模块的交付都应优先通过 asset_export_snapshot 输出，不应让下游模块跨表直接侵入读取上游业务库。\
第五，规则层和 AI 层的结果都必须能回挂到批次、对象和版本，不能只保存在临时内存或日志文件中。\
第六，涉及全文搜索、原文定位和招标内容模糊检索的场景，必须优先基于 source_segment、tender_requirement、bid_template 等对象设计全文索引，而不是后期靠页面端搜索兜底。
