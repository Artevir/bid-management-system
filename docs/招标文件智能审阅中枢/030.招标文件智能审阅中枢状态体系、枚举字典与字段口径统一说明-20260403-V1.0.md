**030.招标文件智能审阅中枢状态体系、枚举字典与字段口径统一说明-20260403-V1.0**

**一、编制目的**

《030.招标文件智能审阅中枢状态体系、枚举字典与字段口径统一说明-20260403-V1.0》用于在《010.招标文件智能审阅中枢核心对象模型与主表逻辑总表-20260403-V1.0》已明确核心对象体系、《020.招标文件智能审阅中枢数据库逻辑关系、主外键与索引设计总表-20260403-V1.0》已明确主外键关系与数据库逻辑的基础上，进一步统一“招标文件智能审阅中枢”全部核心对象在状态字段、枚举字典、统一字段命名和业务口径层面的标准定义，为后续 Drizzle Schema、TypeScript 类型、API DTO、页面状态展示、规则引擎判断、AI 任务写入、人工复核处理和资产输出快照提供一致、稳定、可长期复用的语义底板。

本文件的重点，不是简单罗列若干状态值，也不是零散补充几个前后端字段，而是要一次性解决以下四个根本问题：

第一，系统中不同对象的状态命名如何统一，避免“同一含义不同写法、不同含义同一写法”的混乱局面。\
第二，系统中高频出现的类型字段、等级字段、标识字段和模式字段，如何统一枚举口径，避免后续 AI Coding 在不同文件、不同页面、不同 service 中各自定义。\
第三，核心业务字段如“项目名称”“招标人”“预算金额”“开标时间”“模板类型”“风险等级”“复核状态”等，如何形成平台级统一口径，避免同一字段在不同对象和不同页面中语义漂移。\
第四，状态体系如何服务于“AI 自动抽取—规则自动校验—人工最小复核—资产稳定输出”的中枢主链路，而不是退化成页面展示时临时拼凑的前端标签集合。

因此，本文件既是数据库层的字典规范文件，也是应用层、规则层、AI 任务层和页面层的统一语义说明书。

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“全局统一字典与状态规范章”。其上承对象模型和数据库逻辑，其下直接约束以下内容：

一是约束 PostgreSQL 枚举字段、字符串状态字段和 JSONB 内嵌枚举值的标准写法。\
二是约束 Drizzle Schema 中 enum 定义、TypeScript 联合类型和 DTO 字段注释。\
三是约束前端页面状态标签、筛选项、统计口径和颜色语义。\
四是约束规则引擎、AI 抽取任务、人工复核和资产导出中的枚举值与状态值。\
五是约束后续与企业文档规范资产池、生成装配规则资产池之间的跨模块数据交换字段。

如果没有本文件，后续 AI Coding 极易出现以下问题：\
第一，status、reviewStatus、parseStatus、resultStatus 混用，导致状态体系失控。\
第二，不同文件中对“高风险”“必须项”“固定模板”“自动生成”“人工处理”等概念出现多套字典。\
第三，同一字段在不同主表中命名相近但含义不同，例如 title、name、label、content、text 交叉滥用。\
第四，规则判断、页面筛选、导出快照和下游接入之间的状态值不一致，后期联动成本急剧上升。

因此，本文件必须一次性把“状态、字典、字段口径”这三件事收紧、收死、收成统一标准。

**三、总体设计原则**

**3.1 统一语义优先原则**

所有状态值、枚举值和字段口径，优先追求全系统统一语义，不允许因某个页面展示方便、某个任务实现方便而临时改口。

**3.2 对象分层、状态分层原则**

不同对象可以有不同状态，但状态必须按层分类，例如：项目级状态、文件级状态、解析级状态、对象级状态、复核级状态、输出级状态，不得把不同层级状态压缩成同一个 status 字段。

**3.3 枚举值稳定优先原则**

数据库层、后端层和前端层的枚举值，应尽量以稳定英文代码为主，以中文说明为辅。中文仅用于展示口径，不建议作为数据库存储值，以避免后续迁移、比较、过滤和国际化兼容问题。

**3.4 字段名简洁、含义单一原则**

一个字段只表达一个含义；一个含义尽量只保留一个标准字段名。\
例如，标题用 title，名称用 name，正文内容用 content，原文文本用 raw_text，归一化文本用 normalized_text，不得在不同对象中随意切换。

**3.5 状态可计算、可筛选、可统计原则**

所有核心状态字段必须能支持以下能力：\
可筛选、可聚合、可排序、可统计、可驱动页面颜色语义、可驱动规则和任务流转。\
因此，状态字段不能设计得过于自由，也不能大量使用不可控文本描述代替枚举。

**3.6 一步到位原则**

本文件一次性覆盖目标形态，不按一期、二期拆枚举，不为“将来再看”预留模糊口径。后续新增对象可以扩展字典，但不得推翻本文件已明确的统一规则。

**四、状态体系总体分层**

建议将全中枢状态体系分为七大类：

1.  项目级状态

2.  文件处理级状态

3.  解析批次级状态

4.  业务对象级状态

5.  风险与冲突处理级状态

6.  复核与治理级状态

7.  资产输出级状态

这七层状态体系共同服务于完整主链路：

**项目建立 → 文件接入 → 结构解析 → AI 抽取 → 规则校验 → 人工复核 → 资产确认 → 快照输出**

**十三、字段命名口径统一说明**

这里开始统一最关键的字段命名和含义。

---

**13.1 title、name、label、content、text 的统一用法**

**title**

用于“对象的标题型展示文本”，常见于：

- 风险标题

- 澄清问题标题

- 要求标题

**name**

用于“对象的名称型标识”，常见于：

- 模板名称

- 材料名称

- 规则名称

- 节点名称

**label**

用于“界面标签或变量展示标签”，不建议在主业务对象中泛用。

**content**

用于“业务含义完整的正文内容”，常见于：

- 要求内容

- 问题内容

- 风险说明

**text**

用于“原文文本或文本值”，常见于：

- amount_text

- time_text

- raw_text

- requirement_text

结论：\
以后不得混用“标题”和“名称”，不得把正文统一叫 text，也不得把原始文本统一叫 content。

---

**13.2 raw_text 与 normalized_text**

**raw_text**

表示原文提取后的原始文本，不做清洗或仅做最小清洗。

**normalized_text**

表示用于搜索、规则判断、对比分析的归一化文本。

规则：\
凡涉及原文分片、要求正文、模板正文等需要后续搜索或规则处理的对象，应尽量同时保留原始文本和归一化文本的能力，但不要求所有表都直接存两份字段，可通过上游对象复用。

---

**13.3 source\_\* 系列字段**

**source_document_id**

来源原始文件

**source_page_no**

来源页码

**source_section_id**

来源章节节点

**source_segment_id**

来源最小原文分片

统一原则：\
如果一个对象只能保留一个溯源字段，优先保留 source_segment_id。\
如果对象是聚合对象，也应尽量保留 source_section_id 和 source_page_no 以便页面快速跳转。

---

**13.4 \*\_flag 系列字段**

布尔型标识统一使用 \*\_flag 命名，例如：

- star_flag

- required_flag

- editable_flag

- fixed_format_flag

- original_format_required_flag

- signature_required_flag

- seal_required_flag

- allow_deviation_flag

- hard_constraint_flag

禁止混用：

- is_required

- need_seal

- must_sign

- can_edit

除非在非常明确的通用布尔含义上采用 is\_\*，但本中枢建议以 \*\_flag 为统一口径。

---

**13.5 \*\_level 系列字段**

等级型字段统一使用 \*\_level，例如：

- risk_level

- importance_level

- conflict_level

- priority_level

- severity_level

禁止用：

- risk_grade

- level_type

- level_rank\
  等平行写法。

---

**13.6 \*\_status 系列字段**

状态型字段统一使用 \*\_status。\
如果是对象共通状态，优先用：

- parse_status

- review_status

- resolution_status

- asset_status

- snapshot_status

禁止在同一对象上同时存在多个语义重叠的 status 字段而无前缀说明。

下面把几个必须统一的业务字段口径写死。

---

**14.1 项目名称 project_name**

定义：\
招标项目在招标文件中出现的正式项目名称。

规则：\
优先取封面、招标公告主标题或项目基本信息章节中的标准名称；\
如果多个位置存在差异，以正式标题优先，其他值进入冲突识别，不得直接覆盖。

---

**14.2 项目编号 project_code**

定义：\
招标项目的唯一编号、采购编号或招标编号。

规则：\
只保留最正式的一项作为主值；\
出现多个编号时，应在冲突项或附加字段中保留候选值，不得简单拼接成一个字符串。

---

**14.3 招标人 tenderer_name**

定义：\
发起招标或采购的正式主体名称。

规则：\
与 tender_agent_name 严格区分，不得把代理机构写入招标人字段。

---

**14.4 招标代理机构 tender_agent_name**

定义：\
负责组织代理招标活动的中介或代理机构名称。

规则：\
无代理机构时可为空；\
不得以“采购单位”混写。

---

**14.5 预算金额 project_budget_amount**

定义：\
项目预算金额。

规则：\
统一保留原文字段和数值字段的双承载能力：\
显示层可用文本，规则和排序优先用数值字段。

---

**14.6 最高限价 max_price_amount**

定义：\
投标报价不得超过的金额上限。

规则：\
与预算金额严格区分；\
预算不等于限价，不得二者混用。

---

**14.7 投标截止时间 bid_submission_deadline**

定义：\
递交投标文件的最终截止时间。

规则：\
若仅识别出日期未识别出具体时间，必须在置信度或复核状态中体现不完整性。

---

**14.8 开标时间 bid_opening_time**

定义：\
正式开标的时间点。

规则：\
不得与投标截止时间混淆；\
若文本中出现“同投标截止时间”，可在解析层生成关联，但对象层仍应独立表示。

---

**14.9 模板原样响应要求 original_format_required_flag**

定义：\
招标文件是否明确要求模板“必须使用本格式”“不得修改格式”或同等含义。

规则：\
这是后续生成装配模块的关键输入，必须单独建布尔标识，不得只写在备注里。

---

**14.10 不允许负偏离 negative_deviation_forbidden_flag**

定义：\
技术参数或响应条款是否明确禁止负偏离。

规则：\
该字段必须结构化存在，不能只在技术要求正文中由后续模块自己重新猜测。

---

**十五、前后端展示口径统一说明**

**15.1 数据库存储值与前端展示值分离**

例如数据库存 high，前端展示为“高风险”；\
数据库存 required，前端展示为“必须”；\
数据库存 auto_fill，前端展示为“自动填充”。

禁止前端直接把中文展示值回写数据库。

**15.2 页面颜色语义建议统一**

- low / info：灰蓝或浅色

- medium / warning：黄色

- high：橙色

- critical：红色

- confirmed：绿色

- modified：蓝色

- rejected：灰红或深灰

**15.3 筛选项必须直接映射标准枚举**

所有页面的筛选条件、统计看板、列表标签、批量操作条件，都必须直接从本文件确定的枚举值生成，不能页面自己再造一套“简化标签”。

---

**十六、对 Drizzle Schema、TypeScript 与 API 的直接约束**

**16.1 Drizzle Schema 约束**

- 所有高频状态字段和类型字段优先定义为数据库 enum 或受控字符串。

- 枚举集中放在 enums.ts。

- 主表不得直接使用自由文本 status 替代标准状态枚举。

**16.2 TypeScript 类型约束**

- 所有对象 DTO 应显式使用联合类型或 enum 类型。

- 不允许在前端组件内直接写死 'draft' \| 'confirmed' \| 'rejected' 这类零散字符串。

**16.3 API 约束**

- 返回值中的状态、类型、等级字段统一返回数据库标准值。

- 中文展示文本由前端映射层或 shared dictionary 层统一处理。

- 下游模块消费 asset_export_snapshot 时，也必须继承本文件的字典体系。
