**040.招标文件智能审阅中枢标准接口清单、输入输出结构与对象视图映射总表-20260403-V1.0**

**一、编制目的**

《040.招标文件智能审阅中枢标准接口清单、输入输出结构与对象视图映射总表-20260403-V1.0》用于在《010.招标文件智能审阅中枢核心对象模型与主表逻辑总表-20260403-V1.0》已明确核心对象体系、《020.招标文件智能审阅中枢数据库逻辑关系、主外键与索引设计总表-20260403-V1.0》已明确数据库逻辑关系、《030.招标文件智能审阅中枢状态体系、枚举字典与字段口径统一说明-20260403-V1.0》已明确状态字典与字段口径的基础上，进一步将“招标文件智能审阅中枢”在系统实现层所需的标准接口体系、统一输入输出结构、对象视图映射关系以及前后端交互边界一次性固化下来，为后续 AI Coding 直接开展 Next.js Route Handler、Service 层编排、Drizzle 查询、前端页面对接和下游资产输出提供统一实施依据。

本文件的重点，不是简单罗列一批 REST 接口地址，也不是临时为页面拼几个查询 API，而是要一次性解决以下五个关键问题：

第一，系统应当以哪些标准接口作为“项目入口、解析入口、对象查询入口、治理入口、导出入口”的统一边界。\
第二，不同对象在接口层的输入结构、输出结构、列表结构、详情结构和视图聚合结构应如何统一。\
第三，页面不再围绕旧版“单篇解读详情页”组织，而应围绕对象视图组织，因此页面与对象、对象与接口、接口与输出 DTO 之间的映射关系必须提前定准。\
第四，AI 任务、规则命中、人工复核和下游快照输出，应如何通过接口体系进入主业务链，而不是散落在多个页面或多个脚本逻辑中。\
第五，后续企业文档规范资产池和生成装配规则资产池虽然暂不纳入当前中枢范围，但当前中枢必须通过稳定接口输出标准资产，保证后续模块能直接对接，而不需要反向侵入读取业务主表。

因此，本文件既是接口规范文件，也是前后端边界说明文件，更是页面视图与对象视图的一体化对照总表。

---

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“应用实现总装章”。其上承对象模型、数据库逻辑和字段口径，其下直接约束以下实现内容：

一是约束 Next.js App Router 下的页面结构和 Route Handler 接口分组。\
二是约束后端 Service 层的输入输出 DTO、分页结构、筛选结构和视图聚合结构。\
三是约束前端页面的数据获取方式、对象视图划分方式和交互动作调用方式。\
四是约束 AI 任务层、规则层、复核层和导出层在接口边界上的统一进入方式。\
五是约束未来下游模块对“招标要求资产”的消费方式。

如果没有本文件，后续 AI Coding 极易出现以下问题：\
第一，页面一旦增加，接口便跟着散乱生长，最终形成“每页一套私有口径”。\
第二，列表接口、详情接口、聚合接口和导出接口缺少统一 DTO，后续前后端反复返工。\
第三，接口直接把数据库表结构暴露给前端，导致前端强耦合数据库。\
第四，资产输出接口与页面查询接口混用，最终难以稳定服务后续标书生成模块。

因此，本文件必须一次性把“接口层长什么样、视图层怎么拿数据、对象层怎么聚合输出”定成统一标准。

**三、总体设计原则**

**3.1 对象优先原则**

接口不是围绕旧版页面临时拼接，而是围绕核心对象体系设计。\
所有接口分组必须以对象域或中枢主链路为主，而不是以“某页面叫啥”来命名。

**3.2 视图与数据分离原则**

页面可以组合多个对象，但接口输出应尽量保持对象语义清晰。\
也就是说，页面层允许“总览视图”“风险视图”“模板视图”做聚合，但对象层仍应稳定表达为要求、风险、评分、技术、框架、模板、变量、材料等标准 DTO。

**3.3 输入输出标准化原则**

所有列表查询、详情查询、批量操作、复核动作、导出动作，都必须采用统一的：

- 请求头口径

- 分页结构

- 筛选结构

- 排序结构

- 响应包裹结构

- 错误返回结构

**3.4 读写分离原则**

查询接口与动作接口分离。\
查询接口只负责读取；\
动作接口只负责触发解析、发起复核、确认对象、关闭风险、导出快照等动作。\
禁止一个接口既承担大查询又承担对象修改。

**3.5 页面视图稳定原则**

页面不应直接读取底层多张表拼装数据，而应通过标准视图接口获取中枢视图数据。\
前端可以使用 Query 做缓存，但不应自行承担对象拼装职责。

**3.6 下游友好原则**

除了页面型接口，还必须保留“标准资产输出接口”，供后续企业文档规范资产池和生成装配模块消费。\
当前技术栈继续基于 Next.js、React、Node.js、PostgreSQL 与 Drizzle ORM，不改栈，只重建业务接口体系。

- 要求查询

- 风险查询

- 评分查询

- 技术参数查询

- 框架查询

- 模板与变量查询

- 材料清单查询

**4.4 治理与动作接口层**

负责：

- 风险关闭

负责：

- 查看批次

- 标准资产导出

- 下游模块消费接口

**十三、对后续 AI Coding 的直接约束**

基于本文件，后续 AI Coding 必须遵守以下约束：

第一，所有页面优先消费标准视图接口，不得前端自行跨接口拼装核心业务对象。\
第二，所有接口必须优先返回 DTO，不得直接把 Drizzle 原始表记录透传给前端。\
第三，所有动作接口必须进入统一治理链，触发 review_task、object_change_log 或 asset_export_snapshot 写入。\
第四，所有导出型接口必须以快照为基础，不得临时实时扫描多表拼装“假快照”。\
第五，所有对象详情接口必须具备原文溯源信息，不得输出脱离原文依据的黑盒结论。\
第六，后续企业文档规范资产池和生成装配模块接入时，应优先消费 asset_export_snapshot 或标准资产导出接口，而不应直接侵入中枢数据库主表。

**十四、正式结论**

综上所述，“招标文件智能审阅中枢”的接口体系，不应按页面零散生长，也不应围绕旧版结果型模块临时扩展，而应围绕“项目与文件接入、原文结构查询、要求资产查询、风险与冲突治理、评分与技术查询、框架与模板查询、材料与任务查询、复核与快照输出”八大主线，构建统一的标准接口清单、输入输出结构和对象视图映射体系。

本文件一经确认，后续 Route Handler、Service 层、前端 Query 调用、DTO 设计、批量动作、导出快照和下游模块对接，均应以本文件为统一上位依据，不再允许各页面、各模块、各开发任务自行发明平行接口口径。
