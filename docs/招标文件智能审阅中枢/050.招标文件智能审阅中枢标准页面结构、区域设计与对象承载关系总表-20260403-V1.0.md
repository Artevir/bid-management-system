**050.招标文件智能审阅中枢标准页面结构、区域设计与对象承载关系总表-20260403-V1.0**

**一、编制目的**

《050.招标文件智能审阅中枢标准页面结构、区域设计与对象承载关系总表-20260403-V1.0》用于在《010.招标文件智能审阅中枢核心对象模型与主表逻辑总表-20260403-V1.0》已明确核心对象体系、《020.招标文件智能审阅中枢数据库逻辑关系、主外键与索引设计总表-20260403-V1.0》已明确数据库逻辑关系、《030.招标文件智能审阅中枢状态体系、枚举字典与字段口径统一说明-20260403-V1.0》已明确状态字典与字段口径、《040.招标文件智能审阅中枢标准接口清单、输入输出结构与对象视图映射总表-20260403-V1.0》已明确接口体系和对象视图映射的基础上，进一步将“招标文件智能审阅中枢”在前端页面层的标准页面结构、页面区域设计、对象承载方式、区域间关系和交互承接逻辑一次性固化下来，为后续 Next.js 页面开发、React 组件拆分、shadcn/ui 组件使用、页面查询策略、原文联动设计、人工复核交互和下游资产输出提供统一页面底板。

本文件的核心任务，不是简单列出几个页面名称，也不是围绕传统“列表页—详情页—导出页”的轻量模块思路做补充，而是要正式回答以下六个问题：

第一，“招标文件智能审阅中枢”作为一个中枢级模块，到底应由哪些标准页面组成，这些页面之间的层级关系如何。\
第二，每个页面的主要服务对象是什么，解决什么核心问题，承载哪些核心对象。\
第三，每个页面内部应由哪些固定区域构成，各区域在信息密度、阅读顺序、操作顺序和治理责任上如何分工。\
第四，哪些区域承载原文定位，哪些区域承载 AI 输出，哪些区域承载规则结果，哪些区域承载人工确认与动作。\
第五，页面之间如何形成“总览—专题—对象—原文—复核—输出”的中枢浏览链，而不是零散页面拼盘。\
第六，如何保证页面层既能服务领导快速阅读，也能服务售前、商务、技术、投标经理和复核人员深度处理。

因此，本文件既是页面结构说明书，也是对象承载说明书，更是前端实现与交互治理的一体化设计总表。

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“页面与交互总装章”。其上承对象模型、数据库逻辑、字段字典和接口体系，其下直接约束以下实现内容：

一是约束 App Router 下页面结构与路由结构。\
二是约束每个页面的区域组成、主次层级和中枢浏览路径。\
三是约束对象视图如何在页面中承载，避免前端随意堆字段。\
四是约束原文联动、风险提示、人工复核和导出动作在页面上的固定承载位。\
五是约束组件层的拆分思路，确保后续 AI Coding 生成的页面代码具备统一骨架，而不是各页各画一套。

如果没有本文件，后续 AI Coding 极易出现以下问题：\
第一，虽然对象模型和接口都已存在，但页面设计仍然回到“有什么数据就往一个详情页上堆什么”的旧思路。\
第二，不同页面的信息密度和阅读顺序没有统一规则，导致中枢体验失衡。\
第三，原文定位、风险提示、模板变量、材料清单、人工复核等关键能力没有固定区域承载，最终只能零散穿插。\
第四，页面与对象之间缺少稳定映射，后续组件难复用，页面越做越乱。\
第五，前端为赶进度自行拼装对象视图，反向侵入数据库结构或临时拼接接口结果，最终破坏中枢统一性。

因此，本文件必须一次性把“中枢级页面长什么样、每页放什么、各区域怎么分工、对象如何落位”定成统一标准。

**三、总体设计原则**

**3.1 中枢优先原则**

本模块不是普通详情模块，也不是一个“AI 解读结果页”，而是“招标要求资产生产中心”的前端入口。因此，页面设计必须首先体现中枢属性，即：统一入口、统一视图、统一解释、统一风险提示、统一复核和统一输出，而不是若干独立页面的松散集合。

**3.2 页面围绕对象，不围绕长文原则**

所有页面都应以核心对象视图为主，而不是以“解析后长段文本展示”为主。即使有摘要区，也只能作为辅助区，不能替代对象视图本身。

**3.3 阅读与治理双轨原则**

页面既要让用户快速阅读，也要让用户完成治理动作。因此，每页都应同时考虑以下两类区域：

- 阅读区：用于看结果、看解释、看风险

- 治理区：用于确认、修改、驳回、分派、导出

**3.4 原文溯源内嵌原则**

凡关键对象页面，原则上都要有固定的原文依据承载位或原文联动入口，不能让用户跳出中枢后再靠自己去翻 PDF。原文溯源不是附加能力，而是页面标准组件。

**3.5 总览—专题—对象—原文—复核链条原则**

页面之间必须形成可重复的标准浏览链路：

- 先从总览页看到项目概况与关键问题

- 再进入专题页查看某一类对象

- 再进入对象详情或对象弹层

- 再定位原文依据

- 再发起复核或治理动作

**3.6 固定区域优先原则**

每类页面尽量采用稳定区域骨架，而不是每页单独发明布局。这样便于组件复用、风格统一和 AI Coding 稳定生成。

**3.7 后续联动友好原则**

虽然当前中枢不负责企业文档规范和最终装配生成，但页面设计必须为后续模块联动预留“框架资产”“模板资产”“变量资产”“材料资产”的可视化承载位和输出入口。

**五、页面总体导航结构**

建议采用“一级导航 + 项目上下文 + 二级专题页签”的中枢型结构，而不建议采用过深树形菜单。

**5.1 一级入口建议**

- 招标项目列表

- 当前项目中枢

- 快照与导出

- 系统治理（后续扩展）

**5.2 项目内二级专题页签建议**

进入某个项目后，统一显示以下页签：

- 总览

- 要求

- 风险

- 评分

- 技术

- 框架

- 模板

- 材料

- 复核

**5.3 固定顶部上下文区**

所有项目内页面顶部应统一显示一个“项目上下文头部条”，至少包括：

- 项目名称

- 项目编号

- 当前版本

- 解析状态

- 复核状态

- 待复核数量

- 高风险数量

- 最近解析批次时间

- 快照输出按钮

这样可以保证用户在任何专题页中都不会丢失上下文。

**六、标准页面区域骨架**

为提升统一性，建议所有专题页尽量复用以下标准区域骨架，只是不同页面的区域内容不同。

**6.1 页面公共区域结构**

**区域 A：页面头部区**

用于展示当前页面标题、副标题、页面统计和关键操作。

**区域 B：筛选与操作条区**

用于承载搜索、筛选、排序、批量操作、导出等操作。

**区域 C：主内容区**

用于承载对象列表、树形结构、卡片、表格等核心视图。

**区域 D：对象详情区**

用于展示当前选中对象的详细内容、元信息和治理动作。

**区域 E：原文依据区**

用于展示来源页码、原文分片、原文高亮和上下文片段。

**区域 F：治理动作区**

用于执行确认、修改、驳回、复核、关闭、导出等动作。

并不是每个页面都同时以同样大小展示 A-F 六个区域，但页面设计时必须按这六类区域进行映射，避免遗漏关键承载位。

- 表格：TanStack Table

- 顶部：项目上下文头部

- 右侧或底部：原文/详情/治理抽屉区

**二十、对后续 AI Coding 的直接约束**

基于本文件，后续 AI Coding 必须遵守以下规则：

第一，页面必须围绕标准页面体系建设，不允许再回到旧版“解析详情页”模式。\
第二，所有专题页必须至少具备：头部区、筛选区、主内容区、详情区、原文依据区、动作区六类承载位。\
第三，所有关键对象必须有固定主承载页，不能因为开发方便随意换页。\
第四，原文联动必须作为统一能力建设，不允许每个页面各自临时做一套 PDF 定位逻辑。\
第五，治理动作必须集中在动作区或复核页承载，不能散落在表格按钮中而无统一留痕。\
第六，页面所需数据必须优先通过 040 文件中定义的标准视图接口获取，不允许前端跨接口拼接核心业务对象。\
第七，组件命名和区域命名应尽量沿用本文件定义，避免后续组件树混乱。

**二十一、正式结论**

综上所述，“招标文件智能审阅中枢”的页面设计，不应按传统详情模块思路零散展开，而应围绕中枢总览、要求、风险、评分、技术、框架、模板、材料、复核和原文定位十类标准页面统一展开，并以固定区域骨架、固定对象承载关系和固定原文联动能力作为页面实现底板。

本文件一经确认，后续前端页面开发、组件拆分、Query 接口对接、状态标签展示、原文联动实现和治理动作承载，均应以本文件为统一上位依据，不再允许各页面、各组件、各开发任务自行发明平行页面骨架和承载逻辑。
