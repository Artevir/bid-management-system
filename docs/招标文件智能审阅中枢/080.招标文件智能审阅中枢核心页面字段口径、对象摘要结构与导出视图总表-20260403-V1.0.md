本文件的核心任务，不是简单做一份“字段清单”，而是要正式解决以下六个问题：

第一，不同页面中哪些字段属于“必显字段”，哪些字段属于“详情字段”，哪些字段属于“治理字段”，哪些字段属于“溯源字段”。\
第二，同一种对象在列表摘要、详情面板、原文联动区和导出视图中的字段组织方式应如何统一，避免每个页面、每个接口、每次导出都重新拼字段。\
第三，系统中最核心的对象，如招标要求、风险、评分项、技术参数、框架节点、模板、变量、材料和复核任务，应该分别采用什么样的对象摘要结构，才能兼顾阅读效率与治理效率。\
第四，导出不再是旧版模块那种“把解析结果整体扁平导出”，而应围绕对象视图、专题视图和标准资产视图展开，那么导出层应如何设计。\
第五，页面字段显示必须服务于后续投标文件自动生成的上游目标，因此哪些字段必须显式展示给用户确认，哪些字段可以隐藏在详情或导出层，必须提前定准。\
第六，如何确保前端字段、后端 DTO 字段、导出字段和快照字段之间保持一致，不再出现页面字段一套、接口字段一套、导出字段又一套的混乱局面。

因此，本文件既是页面字段规范文件，也是对象摘要设计文件，更是标准导出视图设计文件。

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“页面字段与导出视图总控章”。其上承对象模型、接口体系、页面结构和规则边界，其下直接约束以下实现内容：

一是约束前端列表页、详情页、抽屉区、原文区和导出区的字段组织方式。\
二是约束标准 DTO 在页面呈现时的字段主次关系。\
三是约束页面表格列、详情面板字段组、卡片摘要字段组和导出字段组的统一来源。\
四是约束对象摘要结构，确保所有核心对象都能在“列表—详情—导出”三种视图中稳定复用。\
五是约束未来快照输出和下游模块消费字段时，不得脱离本文件重新定义视图结构。

如果没有本文件，后续 AI Coding 极易出现以下问题：\
第一，页面字段越做越多，但没有统一层级，导致信息堆叠。\
第二，列表页和详情页字段重复、冲突或缺失，用户在不同页面中看到的同一对象不是同一口径。\
第三，导出字段为了“方便打印”临时拼装，最终与页面确认字段不一致，影响后续自动生成标书。\
第四，关键对象没有统一摘要结构，导致组件无法复用，页面越做越散。\
第五，前端组件为了省事直接透传原始数据库字段，而不是使用标准对象摘要视图，最终反向侵入数据库模型。

因此，本文件必须一次性把“页面字段怎么放、对象摘要怎么定、导出怎么组织”收成统一标准。

**三、总体设计原则**

**3.1 页面字段四分法原则**

所有页面字段统一分为四类：

- **摘要字段**：用于列表、卡片和概览，强调短、准、可快速识别。

- **详情字段**：用于详情面板和对象查看，强调完整、可解释。

- **治理字段**：用于复核、风险、状态、置信度、规则命中、确认边界。

- **溯源字段**：用于页码、章节、原文分片、来源文件、来源批次。

以后任何页面设计、DTO 设计和导出设计，都必须先判断字段属于哪一类，不允许无序堆放。

**3.2 列表轻、详情稳、导出全原则**

列表页字段必须保持克制，不追求字段全；\
详情面板字段必须稳定，覆盖对象的业务主语义；\
导出视图字段必须完整，至少覆盖页面确认所需的全部关键字段。

**3.3 摘要结构统一原则**

同类对象无论出现在总览页、专题页、抽屉、关联对象区还是导出模块中，都必须先有一个统一的对象摘要结构。\
摘要结构不是页面自己发明的小卡片，而是平台级对象视图标准。

**3.4 页面字段不直接等于数据库字段原则**

数据库字段是底层结构，页面字段是对象视图。\
页面应消费 DTO 或 View Model，而不是直接消费数据库表结构。\
同理，导出视图也应基于对象视图，不应直接扫表导出。

**3.5 字段命名稳定原则**

页面字段命名必须继承 030 文件中已经确定的字段口径，不得为了“页面好看”临时改名。\
例如：

- 项目名称统一叫 projectName

- 招标人统一叫 tendererName

- 风险等级统一叫 riskLevel

- 复核状态统一叫 reviewStatus

展示中文可以变，但数据键和语义不得乱。

**3.6 导出服务下游原则**

当前导出不仅服务人工查看，也服务后续模块联动。因此导出视图至少分为：

- 页面导出视图

- 专题导出视图

- 标准资产导出视图\
  三类，而不能只做“当前页面另存为 Excel”。

**四、页面字段总体分层模型**

建议全中枢页面字段统一采用“五层字段模型”。

**4.1 上下文字段层**

用于描述当前页面所处项目上下文、版本上下文和批次上下文。

典型字段：

- projectId

- versionId

- projectName

- projectCode

- versionNo

- versionType

- parseStatus

- reviewStatus

- assetStatus

- latestBatchId

**4.2 摘要识别字段层**

用于一眼识别对象是什么、属于哪类、重不重要。

典型字段：

- requirementType

- riskType

- itemName

- templateType

- materialType

- frameworkTitle

- scoreValue

- starFlag

- requiredFlag

**4.3 核心业务字段层**

用于表达对象本身最关键的业务含义。

典型字段：

- content

- criteriaText

- specRequirement

- templateText

- sourceReason

- finalResolution

- proofMaterialHint

**4.4 治理字段层**

用于表达对象当前治理状态和治理优先级。

典型字段：

- reviewStatus

- riskLevel

- conflictLevel

- resolutionStatus

- confidenceScore

- importanceLevel

- requiredType

- generationMode

**4.5 溯源字段层**

用于支持跳回原文和解释对象从哪里来。

典型字段：

- sourceDocumentId

- sourceDocumentName

- sourcePageNo

- sourceSectionTitle

- sourceSegmentId

- batchId

这五层字段模型以后应适用于全部对象视图。

**五、核心对象摘要结构总表**

下面把核心对象的标准摘要结构一次性定下来。

---

**5.1 招标要求对象摘要结构 TenderRequirementSummary**

**定位**\
用于要求页列表、总览重点要求区、风险关联要求区、材料来源区。

**标准字段建议**

- id

- requirementCode

- requirementType

- requirementSubtype

- title

- contentSummary

- sourceSectionTitle

- sourcePageNo

- importanceLevel

- riskLevel

- confidenceScore

- reviewStatus

- isConflicted

- isTemplateRelated

**字段说明**

- title 优先用于展示要求标题；无标题时可用类型名 + 摘要生成。

- contentSummary 只保留正文的短摘要，不直接塞完整文本。

- isTemplateRelated 必须显式展示，因为这关系到后续标书生成联动。

---

**5.2 风险对象摘要结构 RiskItemSummary**

**定位**\
用于总览重点风险区、风险页列表、复核页关联区。

**标准字段建议**

- id

- riskType

- riskTitle

- riskDescriptionSummary

- riskLevel

- reviewStatus

- resolutionStatus

- relatedRequirementId

- sourcePageNo

- hitRuleCode

- confidenceScore

**字段说明**

- riskTitle 必须适合做列表短标题。

- riskDescriptionSummary 用于说明具体风险点。

- hitRuleCode 在风险页列表中可以作为辅助列显示，不建议在总览卡片中强显。

---

**5.3 冲突对象摘要结构 ConflictItemSummary**

**字段说明**

**八、导出视图总体分层**

导出视图不应只有一种。建议统一分为三层。

**8.1 页面导出视图**

用途：\
用户在当前页面按当前筛选条件导出，用于人工查看、内部沟通、临时整理。

特点：

- 保留列表主字段

- 字段相对轻量

- 与页面筛选条件一致

- 强调当前专题视角

例如：

- 当前风险页导出高风险清单

- 当前模板页导出固定格式模板清单

**8.2 专题导出视图**

用途：\
面向业务专题输出，既给人看，也为下游模块做准备。

特点：

- 比页面导出更完整

- 会增加详情字段

- 强调一个主题下的完整性

例如：

- 全量评分细则导出

- 全量技术参数导出

- 全量材料清单导出

- 全量模板与变量导出

**8.3 标准资产导出视图**

用途：\
供下游模块正式消费，是后续企业文档规范和生成装配模块的上游输入。

特点：

- 基于快照

- 字段最稳定

- 语义最完整

- 不随页面临时变化而变化

例如：

- requirements_snapshot

- framework_snapshot

- templates_snapshot

- materials_snapshot

- full_snapshot

**十四、对后续 AI Coding 的直接约束**

基于本文件，后续 AI Coding 必须遵守以下规则：

第一，所有页面必须基于“对象摘要结构”组织列表和卡片，不得临时拼接字符串作为摘要。\
第二，所有详情面板必须按“基础信息—业务内容—治理状态—关联对象—溯源信息”五组逻辑展开，不得随意堆字段。\
第三，所有导出视图必须先定义为稳定 View Model，再进入 Excel、Word、JSON、TXT、PDF 等格式适配。\
第四，页面显示字段和导出字段必须复用同一对象视图口径，不能页面一套、导出一套。\
第五，任何字段若需新增到列表页，必须先判断其属于摘要字段、治理字段还是溯源字段，不得直接往列表上加。\
第六，涉及后续自动生成标书的关键字段，如框架节点标题、模板原样响应标识、模板变量、材料清单、关键时间金额字段，必须在页面中显式可确认，不得隐藏在 JSON 或折叠深处。\
第七，后续企业文档规范资产池和生成装配模块消费快照时，必须优先使用本文件定义的标准资产导出视图，而不是自行扫库。

**十五、正式结论**

综上所述，“招标文件智能审阅中枢”的页面字段、对象摘要和导出视图，必须作为一个整体统一设计：页面字段按摘要、详情、治理、溯源四类分层；对象摘要按核心对象统一建模；导出视图按页面导出、专题导出和标准资产导出三层展开。只有这样，系统才能真正从“页面展示型模块”升级为“对象驱动、治理友好、导出稳定、可供下游模块复用”的中枢级能力模块。
