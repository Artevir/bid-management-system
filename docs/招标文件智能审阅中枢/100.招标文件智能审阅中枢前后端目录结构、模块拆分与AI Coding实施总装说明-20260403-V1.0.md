本文件的核心任务，不是简单列一个目录树，也不是泛泛而谈“前后端分离”或“模块化开发”，而是要正式解决以下八个问题：

第一，既然本次不是旧模块改造，而是在现有技术栈约束下进行全新重建，那么新的工程目录结构应如何组织，才能既符合当前技术栈，又不再继承旧模块包袱。\
第二，前端页面层、后端接口层、服务层、任务编排层、规则层、导出层、数据库层和共享类型层应如何拆分，才能让 AI Coding 可并行、可收敛、可合并。\
第三，哪些目录属于“对象域实现”，哪些目录属于“平台通用能力”，哪些目录属于“中枢专题能力”，边界如何划清。\
第四，如何在 Next.js App Router + React + Node.js + PostgreSQL + Drizzle ORM 的既有技术栈下，把“模型接入层—任务编排层—规则层—应用层”真正工程化落地，而不是停留在概念层。\
第五，AI Coding 不是传统长周期软件开发，因此不能按一期二期慢慢补，但也不能放任目录和代码无序生长，那么模块拆分和代码组织应如何既支持一步到位设计，又支持快速生成。\
第六，后续不同 AI 任务、不同页面、不同对象域由不同提示词或不同智能体生成代码时，如何保证代码仍然能落在统一骨架之下，而不会变成拼接式工程。\
第七，后续的数据库 schema、DTO、页面组件、任务执行器、复核动作、快照导出等不同子模块，如何通过目录和共享层实现稳定协同。\
第八，如何把这份目录和模块说明，直接变成后续 AI Coding 的任务拆解底板，而不是一份无法执行的“架构文稿”。

因此，本文件既是目录结构说明书，也是模块拆分说明书，更是 AI Coding 实施总装说明书。

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“工程实现总控章”。其上承对象模型、数据库设计、接口设计、页面设计、动作体系、规则体系和任务编排体系，其下直接约束以下实现内容：

一是约束前后端目录结构和命名方式。\
二是约束领域模块、专题模块、共享模块和基础设施模块的拆分边界。\
三是约束 Route Handler、Service、Repository、Orchestrator、Rule Engine、Exporter、View Mapper 等工程层角色。\
四是约束不同 AI Coding 任务应分别产出哪些目录、文件、类型和测试支撑代码。\
五是约束后续如果扩展企业文档规范资产池、生成装配规则资产池或标书生成模块，应如何在现有工程骨架上平滑接入，而不是再次推翻。

如果没有本文件，后续 AI Coding 极易出现以下问题：\
第一，页面代码、接口代码、任务代码和规则代码混写在一起，最终目录虽然能跑，但完全不可维护。\
第二，不同对象域没有独立边界，导致一个需求改动牵连全局。\
第三，前端组件与后端对象没有统一的 shared types，最终 DTO 和页面 props 到处漂移。\
第四，导出、快照、规则、AI 任务、复核逻辑都直接堆在 service 里，逐步演化成“大一统巨石 service”。\
第五，AI Coding 虽快，但多人、多轮、多提示词生成的代码无法自然收敛到统一工程骨架。\
第六，后续一旦加入更多行业规则或下游模块，目录层没有预留可扩展位，只能继续缝缝补补。

因此，本文件的目标，是一次性把“工程上怎么搭、模块上怎么拆、代码上怎么落、AI Coding 怎么组织”收成统一标准。

**三、总体设计原则**

**3.1 技术栈继承、业务工程重建原则**

本次不是换栈，而是在现有技术栈保持不变的前提下全新重建业务工程。\
继续沿用：

- Next.js 15.x

- React 19.x

- Node.js 20.x

- PostgreSQL 16.x

- Drizzle ORM

- 现有文档解析依赖链

- 现有导出依赖链

但旧模块目录、旧 service 设计、旧 interpretation 思路、旧数据库表和旧页面路径，不纳入新中枢正式实现基础。现有技术文档中的依赖和版本仍可作为底座参考。

**3.2 领域优先拆分原则**

目录与模块拆分必须优先围绕领域对象和中枢主链路展开，而不是围绕“谁先写哪页”“哪个接口方便先写”。\
目录骨架应优先体现：

- 招标项目与文件接入

- 原文结构

- 要求资产

- 风险治理

- 评分与技术

- 框架与模板

- 材料与任务

- AI任务与批次

- 复核与快照

**3.3 前后端同构口径原则**

前端页面、后端接口、Service 输出、DTO、View Model、导出视图、快照视图应共享一套对象语义和命名，不允许前端自己再造一套“页面对象”，后端自己再造一套“接口对象”。

**3.4 共享层集中原则**

枚举、DTO、View Model、查询参数、分页结构、动作结构、导出结构和状态映射，应统一收在共享层，避免前端和后端各写一套。

**3.5 AI Coding 友好原则**

目录必须做到：

- 层次清楚

- 职责单一

- 文件名可预测

- 提示词好下发

- 结果好合并

- 错误好定位

这意味着目录结构应服务 AI Coding，而不是只服务传统人工开发阅读。

**3.6 Orchestrator 独立原则**

任务编排层必须独立于普通业务 service。\
不能把批次、任务依赖、失败重试、局部重跑和规则衔接塞进页面 API 或普通 CRUD service 中。

**3.7 Exporter 独立原则**

导出和快照不应附着在页面逻辑或对象 service 中，而应单独形成导出层与快照层，以支持：

- 页面导出

- 专题导出

- 标准资产快照导出

**3.8 可扩展原则**

当前只做“招标文件智能审阅中枢”，但目录必须为后续这些模块预留自然扩展位：

- 企业文档规范资产池

- 生成装配规则资产池

- 投标文件生成模块

- 偏离表生成模块

- 材料归档映射模块

- 行业规则包模块

db/

其中：

- app/：页面与 Route Handler 入口层

- styles/：样式扩展层（若已有则沿用）

作为领域目录名。\

**11.2 目录职责说明**

**providers/**

承载不同模型提供方接入，如：

- deepseekProvider.ts

- qwenProvider.ts

- openaiCompatibleProvider.ts

**prompts/**

承载任务级提示模板配置，不再散落于 service 中。

例如：

- extractBasicInfoPrompt.ts

- extractRequirementsPrompt.ts

- extractScoringPrompt.ts

- detectTemplatesPrompt.ts

**tasks/**

承载 AI 任务级调用封装，不含编排，只负责“单任务如何调用模型并返回结构结果”。

例如：

- runExtractBasicInfoTask.ts

- runExtractScoringTask.ts

- runDetectTemplatesTask.ts

**adapters/**

承载模型返回结构的统一适配层。

**logs/**

承载 AI 调用日志写入辅助函数，与 ai_task_run 表配合。

**11.3 原则**

- AI 调用逻辑不进入 Route Handler

- AI 调用逻辑不散落在普通业务 service

- Prompt 不写死在页面和接口里

- 任务类型与提示模板必须绑定

这与前面确立的“模型接入层—任务编排层—规则层—业务应用层”是一致的。

**十七、AI Coding 实施组织建议**

这部分非常关键，因为你明确强调这是 AI Coding，不走传统长周期分期实施路线。

**17.1 实施原则**

不是按“一期、二期、分步上线”的传统软件项目节奏做，而是：

**以目标架构一次性定准，以模块域并行生成，以统一骨架统一收敛。**

也就是说：

- 架构一次性设计到位

- 代码可以分模块并行生成

- 但所有模块都必须落在本文件定义的目录与职责边界上

**17.2 AI Coding 的推荐任务拆分方式**

建议按“对象域 + 技术层”拆分，而不是按页面拆。

**第一组：数据库与共享层**

- schema

- enums

- dto

- view-models

- queries

- actions

**第二组：接入与结构层**

- ingestion

- structure

- source-view support

**第三组：对象域 service**

- requirements

- risks

- scoring

- technical

- framework

- templates

- materials

- review

- snapshots

**第四组：AI 与规则编排层**

- ai/providers

- ai/prompts

- orchestrator

- rules

**第五组：前端页面与组件层**

- overview

- requirements

- risks

- scoring

- technical

- framework

- templates

- materials

- review

- common components

**第六组：导出层**

- exports/json

- exports/excel

- exports/word

- exports/txt

- exports/pdf

**17.3 合并顺序建议**

虽然你不强调传统分期，但工程合并仍要讲顺序。\
推荐合并顺序如下：

1.  shared + db schema

2.  repository + query + mapper

3.  ai + orchestrator + rules

4.  domain services

5.  api route handlers

6.  frontend pages + components

7.  exports + snapshots

8.  polish + validation

注意，这不是“分期实施”，而是“代码合并顺序”，目的是避免 AI Coding 并行时互相踩接口。

**十八、AI Coding 提示词下发建议**

为了让后续生成代码更稳定，建议以后下发任务时按以下格式组织：

**18.1 提示词必须固定说明的内容**

- 本次任务所属文档编号和模块

- 所属目录

- 依赖哪些 shared 类型

- 依赖哪些 schema

- 输出哪些文件

- 禁止改动哪些模块

- 必须遵守哪些状态与字段口径文件

**18.2 单次任务的推荐范围**

每次 AI Coding 任务尽量控制在：

- 一个对象域

- 或一个页面

- 或一个导出器

- 或一个任务执行器\
  不要一次让 AI 生成整个中枢全部代码，否则最容易偏离骨架。

**18.3 必须显式引用的上位文档**

后续每次 AI Coding 提示词都应至少引用：

- 010 对象模型

- 020 数据库关系

- 030 状态与字段口径

- 040 接口与 DTO

- 050 页面骨架

- 060 动作体系

- 070 规则边界

- 090 任务编排

- 本文件 100 目录与模块总装说明

这样后续所有生成代码才会自动收敛到统一体系里。

**十九、代码治理与命名统一要求**

**19.1 文件命名统一要求**

- Route Handler：route.ts

- 页面：page.tsx

- Service：xxxService.ts

- Repository：xxxRepo.ts

- Query：xxxQueries.ts

- Mapper：xxxMapper.ts

- DTO：xxxDto.ts

- View Model：xxxViewModel.ts

- Exporter：xxxExporter.ts

- Task Runner：runXxxTask.ts

**19.2 禁止的做法**

- 在 page.tsx 中直接写数据库查询

- 在 route.ts 中直接写大模型调用

- 在 service 中直接拼导出文件

- 在前端组件中直接拼装 DTO

- 在 rules 中直接写页面颜色或 UI 逻辑

- 在 shared 中放数据库访问代码

**19.3 日志与错误处理建议**

- AI 任务错误统一进入 ai_task_run.errorMessage

- 关键动作错误统一返回标准错误码

- 导出失败不覆盖已有快照

- 重试逻辑不写在页面层

**二十一、对后续 AI Coding 的直接约束**

基于本文件，后续 AI Coding 必须遵守以下规则：

第一，所有新代码必须落在本文件规定的目录骨架内，不允许自由生长新平行目录。\
第二，所有对象域逻辑必须进入对应模块，不允许跨模块乱放。\
第三，所有共享类型必须优先进入 src/shared/tender-center/，不允许前后端各自拷贝一份。\
第四，所有模型调用必须进入 src/lib/tender-center/ai/ 和 orchestrator/，不允许任何页面、接口、普通 service 自行直接调模型。\
第五，所有导出逻辑必须进入 exports/ 或 snapshots/，不允许页面端直接拼文件。\
第六，所有数据库结构必须进入 src/db/schema/tender-center/，不允许散落 schema。\
第七，任何后续扩展模块必须以本文件的目录哲学为基础，即：对象域清楚、共享层集中、编排层独立、规则层独立、导出层独立。\
第八，后续如 AI Coding 生成的代码与本文件目录边界冲突，应以本文件为准进行重构，而不是为了迁就生成结果改架构。
