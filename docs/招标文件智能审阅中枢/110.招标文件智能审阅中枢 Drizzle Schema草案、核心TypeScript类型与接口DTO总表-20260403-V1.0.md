**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“Schema 与类型落地章”。其上承对象模型、数据库逻辑、字典规范、接口体系、页面体系、动作体系、规则体系和任务编排体系，其下直接约束以下实现内容：

一是约束 src/db/schema/tender-center/ 目录下的 Drizzle schema 文件拆分方式。\
二是约束 src/shared/tender-center/ 下的枚举、领域类型、DTO、View Model 和动作参数结构。\
三是约束 Route Handler、Service、Mapper、Query 层所使用的标准输入输出类型。\
四是约束 AI 任务执行记录、规则命中、复核任务、快照构建等运行期对象的数据结构。\
五是约束后续 AI Coding 不得跳过 shared types 直接在页面、接口或 service 中临时手写重复类型。

如果没有本文件，后续 AI Coding 极易出现以下问题：\
第一，数据库表虽然能建起来，但 TypeScript 层仍然各写各的对象类型，最后前后端对不上。\
第二，Drizzle schema 没有统一基础列和工具函数，所有表重复写一遍，容易漂移。\
第三，DTO 没有统一层级，页面为了赶工直接用表结构类型渲染，最终耦合数据库。\
第四，导出视图和快照视图没有专门类型，最后只能用 any 或临时对象兜底。\
第五，任务编排、规则命中、复核动作和页面视图之间的类型接口不统一，导致后续合并困难。

因此，本文件必须把“schema怎么写、类型怎么分层、DTO怎么固化”一次性定清楚。

**三、总体设计原则**

**3.1 Schema 与类型分层原则**

全中枢类型体系必须至少分成四层：

- **Schema层**：数据库建表结构，对应 Drizzle 表定义。

- **Domain层**：领域对象类型，对应业务语义，不直接暴露数据库实现细节。

- **DTO层**：接口输入输出结构，面向 Route Handler 和页面。

- **View层**：页面聚合视图、导出视图、快照视图。

四层不可混用。

**3.2 Schema 不等于 DTO 原则**

数据库表结构是底层真相，DTO 是对外口径。\
例如：

- 表里可能有 source_segment_id

- 但页面 DTO 展示的是 sourcePageNo、sourceSectionTitle、sourceDocumentName

因此，任何页面都不应直接消费 Drizzle 表类型。

**3.3 枚举集中定义原则**

凡在 030 文档中已经明确的状态体系、类型字典、等级字典，后续必须统一进入 shared enums，并在需要时同步映射到 PostgreSQL enum。\
禁止：

- 页面自己写 'confirmed' \| 'draft'

- service 自己再写一套 risk type 字符串

- schema 里用 text 随便塞状态值而没有约束

**3.4 公共列统一抽象原则**

所有主表原则上都要继承统一的公共列策略，包括：

- 主键

- 时间戳

- 软删除

- 创建人/更新人（视对象需要）

- 备注与扩展位（仅在必要对象中保留）

**3.5 JSONB 节制使用原则**

JSONB 必须用于：

- bbox

- 规则表达式

- 快照内容

- 导出内容

- 复杂表格结构

- 任务入参与任务出参引用

但禁止用 JSONB 懒惰替代本应结构化的核心业务字段。

**3.6 可直接 AI Coding 原则**

本文件中的 schema 草案和类型草案应直接转成：

- Drizzle table 定义

- shared enum

- DTO 类型

- View Model 类型

- mapper 输入输出签名

也就是说，本文件必须是“可直接落码”的。

**5.1 公共列策略**

建议统一抽出三组公共列：

- updatedAt

**B. 软删除列**

- deletedAt

**C. 备注与扩展列**

- note（按需）

- extraJson（谨慎使用）

**5.2 common.ts 草案建议**

import { bigint, boolean, jsonb, text, timestamp } from 'drizzle-orm/pg-core';

export const baseId = {

id: bigint('id', { mode: 'number' }).primaryKey().generatedAlwaysAsIdentity(),

};

export const baseTimestamps = {

createdAt: timestamp('created_at', { withTimezone: true }).defaultNow().notNull(),

updatedAt: timestamp('updated_at', { withTimezone: true }).defaultNow().notNull(),

};

export const softDeleteColumns = {

isDeleted: boolean('is_deleted').default(false).notNull(),

deletedAt: timestamp('deleted_at', { withTimezone: true }),

deletedBy: text('deleted_by'),

};

export const noteColumns = {

note: text('note'),

extraJson: jsonb('extra_json'),

};

**5.3 使用原则**

- 核心主表默认使用 baseId + baseTimestamps + softDeleteColumns

- 运行记录表如 ai_task_run、object_change_log 可根据需要不加 updatedAt

- 快照表不建议软删除，而建议通过状态控制有效性

**二十、对后续 AI Coding 的直接约束**

基于本文件，后续 AI Coding 必须遵守以下规则：

第一，任何建表代码都必须以本文件确定的 schema 文件拆分方式为准，不允许把所有表重新塞进一个大文件。\
第二，任何数据库表都必须先确定共享枚举和共享类型，再开始写 Drizzle 表定义，不允许先写 text 字段凑合。\
第三，任何接口返回类型都必须优先使用 shared dto，不允许 Route Handler 临时手写匿名对象。\
第四，任何页面渲染都必须优先消费 DTO 或 View Model，不允许直接拿 \$inferSelect 类型对象渲染。\
第五，任何导出逻辑都必须先建立 ExportView 类型，再做 xlsx/docx/pdf 适配，不允许导出器直接扫描数据库对象拼字段。\
第六，任何 AI 任务、规则命中、复核动作和快照构建都必须引用本文件中的标准任务类型、对象类型和 DTO 类型，不允许自造平行结构。\
第七，后续新增类型时，必须优先判断属于 enum、domain、dto、view-model、query、action、export 中哪一层，不允许所有类型都堆到一个 types.ts 里。\
第八，凡与 030 文档中字段口径和状态口径冲突的 schema/type 设计，一律视为无效设计，应按 030 文档修正。

**二十一、正式结论**

综上所述，“招标文件智能审阅中枢”的 Drizzle Schema、核心 TypeScript 类型和接口 DTO，必须围绕“schema层—domain层—dto层—view层”四层结构统一落地，并与前面已经确立的对象模型、数据库关系、状态字典、页面视图、动作体系、规则边界和任务编排体系完全对齐。只有这样，后续 AI Coding 才能真正做到一次性目标架构下的快速生成，而不是生成出一堆能跑但无法收敛、无法治理、无法复用的代码碎片。

本文件一经确认，后续数据库 schema 初始化、shared types 建立、DTO 定义、mapper 编写、query 层封装、route handler 开发、页面消费和导出模块实现，均应以本文件为统一上位依据，不再允许数据库层、接口层、页面层、导出层各自形成平行类型体系。
