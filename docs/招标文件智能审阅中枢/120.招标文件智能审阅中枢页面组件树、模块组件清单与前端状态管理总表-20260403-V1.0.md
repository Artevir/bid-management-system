第二，哪些组件属于布局组件，哪些属于专题页面组件，哪些属于对象级组件，哪些属于跨页面通用组件，边界如何划清。\

第四，前端状态到底应如何分层管理，哪些状态走 URL 查询参数，哪些状态走 TanStack Query，哪些状态走 Zustand，哪些状态只保留在本地组件。\
第五，如何避免出现“前端组件树混乱、状态来源不清、一个筛选条件改动导致多个页面状态错乱”的情况。\
第六，如何在现有技术栈下，让前端组件体系既能支撑中枢级复杂页面，又能保持 AI Coding 的可生成性和可收敛性。当前既有前端技术底座继续采用 Next.js App Router、React 19、TypeScript、Tailwind、shadcn/ui、Zustand、TanStack Query、TanStack Table、React Hook Form 与 Zod，不做换栈。\
第七，如何让后续多个 AI Coding 任务分别生成不同页面、不同组件和不同状态 store 时，最终仍能合并为统一的前端工程，而不会变成平行页面和重复组件的堆叠。\
第八，如何为后续企业文档规范资产池、生成装配规则资产池和标书生成模块预留前端扩展位，而不污染当前中枢组件骨架。

因此，本文件既是组件树说明书，也是前端状态治理说明书，更是中枢级前端实现总装稿。

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“前端组件与状态总控章”。其上承对象模型、接口体系、页面骨架、动作体系和共享类型，其下直接约束以下实现内容：

一是约束 src/components/tender-center/ 下的目录拆分和组件职责。\
二是约束每个页面的组件树结构和页面装配方式。\
三是约束页面状态、列表筛选状态、对象选中状态、原文联动状态和复核交互状态的统一管理方式。\
四是约束前端不得绕过标准 DTO 和 View Model 直接操作数据库结构字段。\
五是约束组件命名、事件命名、Store 命名和 Query Key 命名，保证后续 AI Coding 生成代码时可预测、可复用、可合并。

如果没有本文件，后续 AI Coding 极易出现以下问题：\
第一，页面虽然存在，但组件树是平铺式的，公共区域和专题区域没有稳定骨架。\
第二，筛选栏、详情抽屉、原文联动区、复核动作区等公共能力在不同页面各写一套。\
第三，状态管理无统一边界，URL、Query、Store、本地 state 四层混用失控。\
第四，页面组件与共享 DTO 脱节，导致 props 结构漂移、测试困难、组件难以复用。\
第五，前端为了方便临时拼接复杂对象，反向侵入查询层和接口层，破坏整体架构。

因此，本文件的目标，是把前端的“树”和“状态”一次性定稳。

**三、总体设计原则**

**3.1 页面先骨架后内容原则**

每个页面必须先确定固定骨架，再填入对象内容。\
中枢页面不是自由排版页面，而是“统一骨架 + 专题内容 + 原文联动 + 动作区”的组合体。

**3.2 组件分层原则**

组件统一分为四层：

- 布局骨架组件

- 专题页面组件

- 对象视图组件

- 通用交互组件

任何组件都必须明确属于哪一层，不允许无边界“万能组件”。

**3.3 状态四分法原则**

前端状态统一分成四层：

- URL 状态

- 服务端数据状态

- 页面级共享交互状态

- 组件局部状态

后续任何新增状态都必须先判断归属，而不是默认先上 Zustand 或默认写在组件本地。

**3.4 Query 优先、Store 克制原则**

服务端返回的数据、分页、筛选结果和详情数据，优先由 TanStack Query 管理。\
Zustand 只用于跨组件共享但不值得进 URL 的临时交互状态，例如：当前选中对象、原文面板开关、右侧抽屉模式、当前高亮分片。\
不得用 Zustand 替代服务端数据管理。

**3.5 原文联动统一原则**

原文联动必须抽象成统一能力，不允许每个页面各自维护一套 selectedSegmentId、selectedPageNo、openSourceDrawer 的零散实现。

**3.6 表格与详情双区固定原则**

中枢中的大部分专题页都应采用“左侧列表/表格 + 右侧详情/原文/动作”的标准双区模式。\
组件树必须围绕这一模式组织。

**3.7 AI Coding 可收敛原则**

目录命名、组件命名、Store 命名、Hook 命名、Query Key 命名都应具备强可预测性，使不同提示词和不同智能体生成的代码能自动落到统一位置并保持同风格。

**6.2 核心组件职责说明**

**TenderCenterPageShell**

定位：项目内页面统一壳组件。\
职责：

- 承载顶部上下文区

- 承载页签导航

- 统一内容区宽度和边距

- 统一空态、加载态、错误态位置

**ProjectContextHeader**

定位：所有项目内页面顶部上下文头部。\
职责：

- 展示项目名称、项目编号、版本号、状态

- 展示批次状态、待复核数量

- 提供重新解析、生成快照等项目级动作入口

**ProjectTabsNav**

定位：项目内二级专题页签导航。\
职责：

- overview / requirements / risks / scoring / technical / framework / templates / materials / review 统一入口

**PageTitleBar**

定位：专题页标题栏。\
职责：

- 展示当前页面标题、副标题、统计摘要

- 承载页面级导出按钮或批量操作入口

**StandardSplitLayout**

定位：标准左右分栏布局。\
职责：

- 左区：表格/树/列表

- 右区：详情/原文/动作区

**StandardRightDrawerLayout**

定位：右侧抽屉模式布局。\
职责：

- 用于移动端或中等复杂场景，把详情与原文合并到右侧抽屉

**StandardBottomPanelLayout**

定位：底部扩展面板布局。\
职责：

- 用于显示变更日志、原文上下文、双来源对照

**九、前端状态管理总体分层**

这是本文件最关键的部分之一。

建议把前端状态统一分成四层。

**9.1 第一层：URL 状态**

适用于：

- 当前项目 ID、版本 ID

- 当前页签

- 搜索词

- 筛选条件

- 排序条件

- 分页条件

- tab 切换

- 是否只看高风险、只看固定格式等专题过滤条件

原则：

- 所有影响“刷新后仍应保留”的页面筛选条件，优先进入 URL

- 这样可以支持分享链接、返回保留、浏览器前进后退

**9.2 第二层：TanStack Query 状态**

适用于：

- 项目总览数据

- 要求列表与详情

- 风险列表与详情

- 评分项与详情

- 技术参数与详情

- 框架树与节点详情

- 模板详情和变量列表

- 材料详情

- 复核任务和变更日志

- 原文分片详情

- 快照列表

原则：

- 服务端数据不进 Zustand

- 服务端列表不靠本地 state 缓存

- 所有 Query 必须有明确 queryKey

**9.3 第三层：Zustand 页面共享交互状态**

适用于：

- 当前选中的对象 ID

- 详情抽屉开关

- 原文面板开关

- 当前高亮分片 ID

- 树节点展开状态

- 双原文对照模式

- 模板预览模式

- 跨多个子组件共享的复核面板状态

原则：

- 只放“跨多个组件共享，但不值得进入 URL”的状态

- 不放服务端数据实体

- 不放表格全量数据

- 不放导出结果

**9.4 第四层：组件局部状态**

适用于：

- Tooltip 开关

- 临时输入框

- 表格列显示开关

- 面板折叠开关

- 动画状态

- 局部 Tab 状态

原则：

- 只在组件内部使用

- 不跨页面、不跨区域共享

**十二、表格、树、抽屉、原文联动的状态治理规则**

**12.1 表格状态治理**

表格应统一管理以下状态：

- pagination

- sorting

- columnVisibility

- rowSelection

其中：

- pagination 和 sorting 建议同步 URL

- rowSelection 建议仅本地或页面 store 维护

- columnVisibility 建议本地组件维护或后续走用户偏好配置

**12.2 树状态治理**

框架树统一管理：

- expandedNodeIds

- selectedNodeId

- activeNodePath

其中：

- selectedNodeId 走页面 store

- expandedNodeIds 走页面 store

- activeNodePath 可由 selectedNodeId 推导

**12.3 抽屉状态治理**

统一约定：

- 详情抽屉是否打开：走页面 store

- 原文抽屉是否打开：走 useSourceViewStore

- 批量操作弹窗：局部 state

- 导出弹窗：局部 state

**12.4 原文联动治理**

任何页面触发“定位原文”时，统一调用：

- openSourcePanel

- setSourceDocumentId

- setSourcePageNo

- setHighlightedSegmentIds

不允许每页自己发明原文联动逻辑。

**十六、对后续 AI Coding 的直接约束**

基于本文件，后续 AI Coding 必须遵守以下规则：

第一，任何页面开发都必须先按本文件给出的页面组件树组织根组件和一级子组件，不允许直接在 page.tsx 中堆所有逻辑。\
第二，任何跨页面通用能力必须优先进入 common/、layout/ 或 source-view/，不得在各页面复制粘贴实现。\
第三，任何服务端数据必须优先通过 Query Hook 获取，不允许页面或组件直接 fetch 并手写缓存。\
第四，任何页面共享交互状态必须先判断是否适合进 Zustand，不得因为方便把所有状态都塞进 Store。\
第五，任何筛选条件若需要刷新保留、可分享链接、支持前进后退，必须进入 URL 查询参数。\
第六，任何对象详情区、原文联动区、动作区都必须使用标准组件骨架，不允许每页自定义一套详情结构。\
第七，任何新页面、新专题、新组件如果与现有组件职责重叠，应先扩展现有组件，而不是新建平行组件。\
第八，后续如扩展企业文档规范资产池和生成装配模块，也必须遵循本文件的组件分层和状态分层哲学，不得重新引入一套页面状态体系。

**十七、正式结论**

综上所述，“招标文件智能审阅中枢”的前端实现，必须围绕“统一页面骨架、统一组件分层、统一状态分层、统一原文联动、统一动作承载”的原则展开，并通过布局组件、专题组件、对象组件和通用交互组件四级体系，配合 URL 状态、TanStack Query 状态、Zustand 交互状态和组件局部状态四层状态治理机制，形成既适合中枢级复杂界面、又适合 AI Coding 快速生成和统一收敛的前端工程底板。
