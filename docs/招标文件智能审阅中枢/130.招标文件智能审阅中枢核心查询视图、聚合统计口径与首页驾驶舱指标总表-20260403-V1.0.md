第一，当前中枢中哪些查询属于“对象级明细查询”，哪些属于“聚合统计查询”，哪些属于“驾驶舱视图查询”，边界如何划分。\
第二，统计口径必须统一，否则总览页、要求页、风险页、复核页看到的数字会互相打架。那么哪些指标以项目为维度，哪些以版本为维度，哪些以当前筛选条件为维度，必须提前定死。\
第三，首页驾驶舱不是简单展示“解析了多少条”，而是要为投标经理、售前负责人、技术负责人和复核人员提供真正有用的决策视图，那么首页该展示哪些核心指标、哪些异常指标、哪些待办指标。\
第四，专题页中的统计卡与首页统计卡如何统一来源，避免 Overview 一套算式、专题页另一套算式。\
第五，批次层、对象层、规则层、复核层和快照层的统计关系如何打通，使首页驾驶舱既能看“当前项目总体状态”，也能看“当前版本是否可用”“当前批次是否完成”。\
第六，如何在 Query 层中构建稳定的聚合查询，而不是后续每个页面自己 count 一遍、group 一遍。\
第七，哪些统计适合实时计算，哪些统计应通过聚合视图或快照视图复用，避免中枢越来越重时查询失控。\
第八，如何让这些指标未来自然延伸到后续企业文档规范资产池、生成装配规则资产池和标书生成模块，而不是只服务当前的“解读页面”。

因此，本文件既是查询视图设计文件，也是统计口径统一文件，更是首页驾驶舱指标设计文件。

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“查询视图与驾驶舱总控章”。其上承对象模型、数据库关系、接口体系、页面结构、任务编排和前端状态管理，其下直接约束以下实现内容：

一是约束 queries/ 层的查询分类和命名方式。\
二是约束首页总览页、专题页统计卡、批次摘要面板和快照状态区的统计来源。\
三是约束不同页面中相同指标的统一计算口径。\
四是约束聚合查询输出的 View Model 结构。\
五是约束后续导出和快照中涉及统计汇总时的字段口径。\
六是约束前端不得自行在页面端拼统计数字，而应消费标准聚合视图。

如果没有本文件，后续 AI Coding 极易出现以下问题：\
第一，Overview 页和各专题页自己算自己的数字，后面必然出现对不上。\
第二，很多 count 看起来简单，但到底按项目、版本、批次、筛选条件还是复核状态统计，语义不清。\
第三，批次状态、对象状态、风险状态、复核状态和快照状态彼此割裂，首页无法形成真正的“驾驶舱”。\
第四，统计逻辑散落在 route、service、page、component 中，最终性能差且难维护。\
第五，后续一旦增加更多对象域或导出视图，统计口径无法复用，只能继续补丁。

因此，本文件的目标，是一次性把“查什么、怎么算、展示到哪里、由谁提供”定清楚。

**三、总体设计原则**

**3.1 版本优先原则**

除项目列表页外，绝大多数中枢统计口径必须以“当前项目的当前版本”为主统计维度，而不是按全项目混算。\
原因在于：招标文件、答疑、补遗和补充版本可能不断变化，真正供投标使用的是“当前有效版本资产”。

**3.2 查询分层原则**

查询统一分成三层：

- **对象级查询**：取明细

- **聚合级查询**：取统计

- **驾驶舱级查询**：取跨对象综合视图

页面不应跨层乱用。

**3.3 口径先于数字原则**

一个数字有没有价值，不取决于它大不大，而取决于它口径是否稳定。\
因此，所有统计卡、驾驶舱指标、趋势字段都必须先有文字定义，再有 SQL 或 Drizzle Query 实现。

**3.4 状态驱动统计原则**

中枢中的绝大多数统计都不是静态计数，而是“对象数量 + 状态分布”的组合。\
因此，Query 设计必须优先围绕：

- parseStatus

- reviewStatus

- resolutionStatus

- riskLevel

- conflictLevel

- requiredType

- generationMode

- snapshotStatus\
  这些核心状态组织。

**3.5 可解释原则**

首页驾驶舱中的每一个数字，都应能追溯到下一级专题页或对象列表。\
也就是说，指标不能只是“孤立数字”，而应能下钻到对应列表。

**3.6 性能与一致性平衡原则**

高频首页指标和专题统计应优先通过标准聚合查询统一提供；\
对极复杂统计，可以采用 Query 聚合层或后续物化视图/快照复用；\
但当前口径必须先统一，不允许为了性能先乱简化。

**3.7 查询视图独立原则**

聚合统计查询不应散落在 Service 和页面中，而应尽量统一进入 queries/ 层和 View Model 层。

**六、首页驾驶舱总体结构**

首页驾驶舱建议固定由六个指标区组成。

**6.1 项目基础信息区**

用于回答：

- 当前看的是哪个项目

- 当前使用哪个版本

- 项目基本规模和当前状态如何

**6.2 资产生成状态区**

用于回答：

- 对象抽取生成得怎么样

- 关键专题有没有生成完

- 哪些资产域还不完整

**6.3 风险与冲突告警区**

用于回答：

- 当前有哪些高优先级风险

- 有没有重大冲突

- 是否存在阻塞投标的红线问题

**6.4 复核与待办区**

用于回答：

- 还有多少对象待复核

- 强制复核项还有多少

- 当前最应该先处理什么

**6.5 批次与运行状态区**

用于回答：

- 最近一次解析批次是否成功

- 是否存在部分成功

- AI 任务是否有失败、重试、降级

**6.6 快照与输出区**

用于回答：

- 当前版本是否可生成正式快照

- 最近一次快照何时生成

- 当前快照是否已发布、是否可供下游使用

**八、首页驾驶舱指标定义细则**

**8.1 requirementCount**

定义：当前版本下 tender_requirement 的有效记录总数，不含逻辑删除和已废弃记录。\
用途：衡量要求资产总规模。

**8.2 highRiskCount**

定义：当前版本下 risk_item 中 riskLevel in ('high','critical') 的对象数。\
用途：衡量当前风险压力。

**8.3 criticalRiskCount**

定义：当前版本下 risk_item.riskLevel='critical' 的对象数。\
用途：首页红色告警核心指标。

**8.4 conflictCount**

定义：当前版本下全部 conflict_item 数量。\
用途：衡量内容不一致问题数量。

**8.5 unresolvedConflictCount**

定义：当前版本下 conflict_item.reviewStatus not in ('resolved','closed') 的数量。\
用途：衡量当前未定案冲突压力。

**8.6 pendingReviewCount**

定义：当前版本下 review_task.reviewStatus in ('pending_assign','assigned','reviewing') 的数量。\
用途：衡量人工待办压力。

**8.7 forceReviewCount**

定义：当前版本下由强制复核规则生成的复核任务数。\
建议实现：在 review_reason 或 extraJson 中区分是否为强制复核来源。\
用途：衡量真正不能放过的关键对象压力。

**8.8 reviewedCount**

定义：当前版本下核心对象中 reviewStatus='confirmed' 的总数。\
用途：衡量治理完成度。

**8.9 modifiedCount**

定义：当前版本下核心对象中 reviewStatus='modified' 的总数。\
用途：衡量人工纠偏量。

**8.10 assetReadyFlag**

定义：当前版本在满足以下条件时取 true：

- 主链路批次成功或部分成功但核心对象齐全

- 无 unresolved critical risk 阻塞

- 无 unresolved major conflict 阻塞

- 强制复核对象已全部处理或有明确结论

- 至少生成过一份合规快照候选\
  用途：是否可进入下游使用的判断信号。

**8.11 blockingIssueCount**

定义：所有阻塞快照发布或下游消费的问题总数，建议计算来源包括：

- criticalRiskCount

- unresolvedConflictCount 中的重大冲突

- forceReviewCount 中未处理项

- 缺失关键框架节点或关键模板识别失败项\
  用途：首页中最关键的“是否还能往下走”指标。

**10.1 要求页统计口径**

| **指标**                        | **统计口径**                        |
| ------------------------------- | ----------------------------------- |
| totalRequirementCount           | 当前版本全部要求数                  |
| highRiskRequirementCount        | riskLevel 为 high/critical 的要求数 |
| conflictedRequirementCount      | isConflicted = true 的要求数        |

**十三、查询口径统一规则**

**13.1 全量统计口径**

所谓“总数”，默认指：

- 当前项目当前版本

- 非逻辑删除

- 非无效化

- 非已废弃

除非页面显式声明“含历史版本”或“含全部批次”。

**13.2 当前筛选统计口径**

专题页顶部统计默认建议分两类：

- 全量统计：针对当前版本全量对象

- 当前筛选统计：针对当前页面筛选条件下的对象

若同时展示，必须明确标识，不能让用户误以为列表只有 20 条但顶部说 100 条是在“出 bug”。

**13.3 当前批次统计口径**

只有当页面明确进入批次视角时，才按 document_parse_batch 维度统计。\
首页和大多数专题页，优先按“当前版本已生效对象”统计，而不是按最近批次临时对象统计。

**13.4 风险阻塞口径**

“阻塞问题”不是所有风险的简单加总，而是指满足下列任一条件的对象：

- critical 风险

- 未定案重大冲突

- 强制复核未完成对象

- 快照发布前的关键框架/模板缺失问题

后续 Query 层必须按这一定义实现，而不能用 “highRiskCount + unresolvedConflictCount” 机械替代。

**十四、首页驾驶舱重点对象清单口径**

首页除了数字，还必须展示少量重点对象清单。

**14.1 Top Risks**

建议取：

- riskLevel 优先

- reviewStatus 未完成优先

- 最近新增优先\
  默认展示前 5 条。

**14.2 Top Conflicts**

建议取：

- unresolved 优先

- conflictLevel 优先\
  默认展示前 5 条。

**14.3 Pending Reviews**

建议取：

- 强制复核优先

- 高风险相关复核优先

- 最近分派优先\
  默认展示前 8 条。

**14.4 Key Time Nodes**

建议取：

- bid_submission_deadline

- bid_opening_time

- bond_deadline

- question_deadline\
  默认展示关键 3~5 条。

**十五、Query 实现建议**

**15.1 Overview 查询建议**

建议把首页总览查询拆成以下子查询，再在 Query 层聚合：

- getProjectMeta

- getRiskStats

- getReviewStats

- getBatchStats

- getSnapshotStats

- getTopRisks

- getTopConflicts

- getPendingReviews

- getKeyTimeNodes

- getKeyMoneyTerms

**15.2 专题统计查询建议**

每个专题页建议至少拆成：

- listQuery

- detailQuery

- statsQuery

例如：

- listRequirements

- getRequirementDetail

- getRequirementStats

这样前后端结构也更统一。

**15.3 Query 命名建议**

统一使用：

- getXxxStats

- getXxxSummary

- getXxxOverview

- listXxx

- getXxxDetail

避免混用：

- countXxx

- fetchXxxData

- loadXxx

- getAllXxxForDashboard\
  这类风格不一的命名。

**十六、前端消费规则**

**16.1 首页消费规则**

首页总览页优先消费 dashboardQueries.ts 中的 getDashboardOverviewView 或等价 Query Hook。\
不允许首页分别调用十几个接口自己拼接统计数字。

**16.2 专题页消费规则**

专题页应分别消费：

- statsQuery

- listQuery

- detailQuery

不建议一个专题页只调 listQuery 后在前端自行 count 统计卡。

**16.3 导出前统计预览**

导出弹窗如果要展示“当前将导出多少条”，应尽量复用 statsQuery 或当前 listQuery 的分页外总数字段，不应另写一套临时统计。

**十七、与快照和导出的衔接口径**

**17.1 快照不等于驾驶舱**

驾驶舱是当前状态视图；\
快照是标准资产输出视图。\
二者可以复用部分统计口径，但不能混为一体。

**17.2 快照统计建议**

快照中建议附带以下统计字段：

- requirementCount

- riskCount

- conflictCount

- scoringItemCount

- technicalSpecCount

- frameworkNodeCount

- templateCount

- materialCount

- reviewedCount

- pendingReviewCount

- blockingIssueCount

这样后续下游模块在接快照时，也能快速知道上游资产健康度。

**17.3 导出统计建议**

导出文件头部若需显示摘要，可复用对应专题页统计字段，不应重新计算另一套数字。

**十八、性能与落地建议**

**18.1 当前阶段建议**

在测试阶段和首轮实现阶段，优先使用标准 Query 聚合实现，不急着引入物化视图。\
先把口径定准，比过早做性能微优化更重要。

**18.2 后续可优化方向**

若后续数据量上来，可考虑对以下查询做优化：

- 首页总览聚合

- 风险统计聚合

- 复核任务聚合

- 批次摘要聚合

- 模板与变量统计聚合

优化方式可包括：

- 查询结果缓存

- 快照复用

- 聚合中间表

- 物化视图\
  但优化前后口径不得改变。

**18.3 查询层原则**

即使未来做缓存或物化视图，也必须保持：

- Query 方法签名不变

- View Model 结构不变

- 页面消费方式不变

---

**十九、对后续 AI Coding 的直接约束**

基于本文件，后续 AI Coding 必须遵守以下规则：

第一，所有聚合统计都必须先在 Query 层定义标准方法，再供页面消费，不允许页面自行统计。\
第二，所有首页驾驶舱数字都必须有本文件中的正式定义，不能临时想到什么就展示什么。\
第三，所有专题页统计卡都必须与首页和对象口径一致，不允许同名指标不同算法。\
第四，所有“阻塞问题”“待复核”“高风险”“原样响应模板”等核心指标都必须由规则层和状态层共同驱动，不能只看单表字段。\
第五，所有 Query 输出都必须进入 shared view-model 或 dto，不允许 route 直接返回匿名统计对象。\
第六，所有后续新增驾驶舱指标，必须先明确统计口径、统计维度和下钻路径，再进入实现。\
第七，快照与导出复用统计字段时，必须优先继承本文件定义，不允许另立统计口径。\
第八，若后续为了性能引入缓存、聚合表或物化视图，也只能替换实现，不得改变本文件定义的查询视图语义。

---

**二十、正式结论**

综上所述，“招标文件智能审阅中枢”的查询视图、聚合统计口径与首页驾驶舱指标，必须作为一个整体统一设计：对象级查询负责明细，专题统计查询负责局部聚合，驾驶舱查询负责项目级综合视图，快照统计负责下游可用性摘要。首页驾驶舱不应只是若干漂亮数字的拼盘，而应真正成为项目当前状态、风险压力、复核压力、批次健康度和快照可用性的统一判断入口。

本文件一经确认，后续 Query 层开发、Overview 页面建设、专题页统计卡建设、导出摘要生成和快照统计扩展，均应以本文件为统一上位依据，不再允许页面端、服务端和导出端各自发展出平行统计口径。
