第七，如何让整个中枢在状态推进上体现“AI自动抽取—规则约束判断—人工最小确认—标准资产输出”的主链路，而不是仅仅做成几个 CRUD 状态字段。\
第八，如何让后续扩展企业文档规范资产池、生成装配规则资产池和标书生成模块时，依然可以复用本文件中已经固化的状态推进哲学，而不需要另起一套。

因此，本文件既是对象状态流转总表，也是批次推进规则总表，更是页面联动规则总表。

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“状态流转与联动总控章”。其上承对象模型、动作体系、规则体系、任务编排和查询统计，其下直接约束以下实现内容：

一是约束所有核心对象的状态流转路径、禁止跳转路径和异常回退路径。\
二是约束 document_parse_batch 及其子任务的状态推进逻辑。\
三是约束 review_task、risk_item、conflict_item、asset_export_snapshot 等治理对象与业务对象的联动关系。\
四是约束前端页面中 Badge、按钮、统计卡、筛选项、抽屉提示和操作入口的可用条件。\
五是约束后续 AI Coding 不得在 service 或页面中私自增减状态流转路径，所有状态变化都必须回到本文件定义。

如果没有本文件，后续 AI Coding 极易出现以下问题：\
第一，状态虽然有定义，但没有流转规则，最终项目里到处都是“直接改 status”。\
第二，前端按钮是否可点、页面是否显示待办、快照是否可发布，全部依赖临时判断，久而久之一定混乱。\
第三，批次部分成功、局部重跑、风险关闭、复核提交和快照发布之间没有清晰依赖，导致对象状态互相打架。\
第四，统计卡数字虽然能显示，但底层状态与页面状态不一致，用户会觉得系统“不可信”。\
第五，后续想接入更多模块时，发现上游根本没有统一状态推进底座，只能重新补一套。

因此，本文件的目标，是把“谁能变、怎么变、何时变、变了后页面怎么响应”一次性定稳。

**三、总体设计原则**

**3.1 分层流转原则**

状态流转必须严格分层，至少分为：

- 项目层状态流转

- 版本层状态流转

- 文件与批次层状态流转

- 业务对象层状态流转

- 治理对象层状态流转

- 快照输出层状态流转

- 页面展示层状态映射

不同层级不得混用一个状态值。

**3.2 动作驱动原则**

任何关键状态变化都必须由明确动作或明确任务结果驱动，不允许“代码顺手就改了状态”。\
状态推进的合法来源只有三类：

- 系统任务推进

- 规则判断推进

- 人工治理动作推进

**3.3 主链优先原则**

全中枢的状态推进必须服务于一条主链：

**项目建立 → 文件接入 → 批次解析 → 对象生成 → 规则校验 → 复核分流 → 关键对象确认 → 快照构建 → 快照发布**

所有对象状态流转都应尽量能映射回这条主链中的某个环节。

**3.4 最小必要人工原则**

不是所有状态变化都需要人工，但涉及关键时间、关键金额、高风险、冲突、关键框架、关键模板和正式快照发布时，必须经过人工确认节点后再推进。

**3.5 异常可回退原则**

任何关键状态都必须允许在必要时回到合理上一步，而不是只能不断向前推。\
例如：

- 风险关闭后可重新打开

- 冲突定案后可回退

- 快照发布后可失效化

- 对象确认后可进入重新复核

**3.6 页面统一映射原则**

页面上看到的颜色、标签、可点击动作、统计归类，都必须由底层状态统一映射，不允许页面自己另发明一套“显示状态”。

**3.7 口径稳定原则**

状态流转规则一经确认，后续实现只能优化触发条件和性能，不应随意改变主流转路径和状态意义。凡涉及主流转路径变更，必须修订本文件。

---

**四、状态流转总体分层**

建议将状态流转统一拆成七个层级：

1.  项目与版本状态流转

2.  文件与批次状态流转

3.  原文结构对象状态流转

4.  核心业务对象状态流转

5.  风险与冲突治理状态流转

6.  复核任务状态流转

7.  快照与输出状态流转

页面层不单独定义新状态，而是对以上七层做统一映射。

---

**五、项目与版本状态流转规则**

**5.1 项目层核心状态**

建议项目层保留三组核心状态：

- parseStatus

- reviewStatus

- assetStatus

**5.2 项目解析状态流转规则**

**流转路径**

not_started → ingesting → parsing → partially_parsed / parsed → archived

**异常路径**

- parsing → parse_failed

- parse_failed → parsing（重跑后）

- partially_parsed → parsing（补跑后）

**触发条件**

- 创建项目后默认 not_started

- 上传首个文件并开始接入后进入 ingesting

- 创建首个解析批次并启动主链任务后进入 parsing

- 主链完成且存在部分专题失败时进入 partially_parsed

- 主链和核心专题全部完成时进入 parsed

- 项目不再使用时可进入 archived

**5.3 项目复核状态流转规则**

**流转路径**

not_reviewed → reviewing → partially_reviewed / reviewed → closed

**异常回退**

- reviewed → reviewing

- closed → reviewing

**判定原则**

- 只要存在待复核对象，则不应进入 reviewed

- 全部强制复核对象处理完成，但普通建议复核对象仍存在时，可进入 partially_reviewed

- 全部关键对象完成确认或有明确结论时，进入 reviewed

**5.4 项目资产状态流转规则**

**流转路径**

not_generated → generating → generated → partially_confirmed / confirmed → exported / invalidated

**判定原则**

- 对象生成完成但未完成关键确认时，最多进入 generated 或 partially_confirmed

- 只有达到可输出条件时，才能进入 confirmed

- 生成快照且正式用于下游时，进入 exported

- 若因重跑、规则更新、关键冲突回退导致资产不再可信，则进入 invalidated

**六、版本层状态流转规则**

**6.1 版本层设计原则**

版本是中枢真正的工作边界。\
绝大多数对象、统计、快照、复核、导出都以版本为主语，因此版本状态应比项目状态更精细。

**6.2 版本核心状态建议**

建议版本层同样保留：

- parseStatus

- reviewStatus

- assetStatus

- isCurrent

**6.3 当前版本切换规则**

- 新建版本时，若显式指定为当前版本，则旧版本 isCurrent=false

- 当前版本的快照、统计和页面默认视图优先展示

- 旧版本对象不得直接覆盖当前版本对象

**6.4 版本回退规则**

若新版本解析失败或关键对象异常严重，可：

- 保持旧版本 isCurrent=true

- 新版本停留在 parse_failed 或 partially_parsed

- 不允许把失败版本强行升为当前版本

**七、文件与批次状态流转规则**

**7.1 文件接入状态流转**

适用对象：

- source_document

**流转路径**

uploaded → queued → ingesting → ingested

**异常路径**

- ingesting → ingest_failed

- ingest_failed → queued（重试后）

**补充说明**

- 文件接入完成不等于文本提取完成

- 文件接入完成后，文本提取与结构识别状态单独推进

**7.2 文本提取状态流转**

适用对象：

- source_document.textExtractStatus

**流转路径**

not_started → extracting → extracted

**异常路径**

- extracting → extract_failed

- extract_failed → extracting

**7.3 结构识别状态流转**

适用对象：

- source_document.structureExtractStatus

**流转路径**

not_started → structuring → structured

**异常路径**

- structuring → structure_failed

- structure_failed → structuring

**7.4 批次状态流转**

适用对象：

- document_parse_batch

**主流转路径**

queued → running → partially_succeeded / succeeded

**异常路径**

- running → failed

- failed → running（重新执行）

- partially_succeeded → running（补跑专题任务）

- queued → cancelled

**7.5 批次状态判定规则**

**queued**

批次已创建，尚未进入主任务执行。

**running**

至少一个任务已开始。

**partially_succeeded**

满足以下条件：

- 主链任务完成

- 核心对象已生成

- 但一个或多个非阻塞专题任务失败或降级

**succeeded**

满足以下条件：

- 主链任务完成

- 核心专题任务完成

- 规则校验与复核分流完成

**failed**

满足以下条件：

- 主链关键任务失败

- 无法产出有效对象集

- 无法进入复核分流

**7.6 批次与版本联动规则**

- 当前有效批次 succeeded 时，版本 parseStatus 可推进至 parsed

- 当前有效批次 partially_succeeded 时，版本 parseStatus 推进至 partially_parsed

- 当前有效批次 failed 且无其他有效批次可接替时，版本 parseStatus 推进至 parse_failed

**八、原文结构对象状态流转规则**

**8.1 适用对象**

- document_page

- source_segment

- document_section_node

- attachment_requirement_node

**8.2 设计原则**

原文结构对象一般不需要复杂人工确认状态，但应有最基本的有效性与可用性状态。

**8.3 建议状态**

- valid

- invalid

- superseded

**8.4 流转规则**

- 正常提取成功后默认 valid

- 若发现结构识别错误、页内容重复、分片失真、章节树错误，可转为 invalid

- 若新批次生成的新结构对象取代旧结构对象，可将旧对象标记为 superseded

**8.5 页面联动规则**

页面默认只消费 valid 对象，除非进入排错模式或批次对比模式。

**九、核心业务对象状态流转规则**

**9.1 适用对象**

- tender_requirement

- qualification_requirement

- commercial_requirement

- technical_requirement

- time_node

- money_term

- submission_requirement

- scoring_item

- technical_spec_item

- bid_framework_node

- bid_template

- template_variable

- submission_material

- response_task_item

- clarification_candidate

**9.2 通用复核状态流转主路径**

建议统一使用：

draft → pending_review → reviewing → confirmed / modified / rejected → closed

**9.3 各状态定义**

**draft**

AI 或规则已生成对象，但尚未进入正式复核或尚未完成基础校验。

**pending_review**

对象已被系统识别为需要进入复核队列，但尚未正式分派或尚未开始处理。

**reviewing**

对象已进入人工处理过程。

**confirmed**

对象当前值已被人工确认，或符合自动通过条件且已进入正式可用状态。

**modified**

对象经过人工修正后确认，属于正式可用状态，但必须保留“曾经被修改”的治理痕迹。

**rejected**

对象当前结果被明确否定，不应作为正式资产继续使用。

**closed**

对象已完成处理并不再活跃，常用于任务型对象或临时治理对象。

**9.4 自动推进规则**

以下对象在满足条件时可从 draft 直接进入 confirmed：

- 一般提交要求

- 普通说明性要求

- 普通低风险模板变量

- 普通材料提示项

前提必须同时满足：

- 高置信度

- 无冲突

- 未命中高风险规则

- 非强制人工确认对象

**9.5 强制人工确认对象流转规则**

以下对象不得从 draft 直接自动进入 confirmed：

- 关键时间节点

- 关键金额条款

- 高风险对象

- 冲突关联对象

- 一级框架节点

- 原样响应模板

- 固定格式模板

- 不允许负偏离技术项

- 高分值评分项

- 必须材料中的关键签章项

这些对象必须：\
draft → pending_review / reviewing → confirmed / modified / rejected

**9.6 驳回后的处理规则**

对象进入 rejected 后：

- 默认不进入页面默认主列表

- 可在“已驳回”筛选中查看

- 若后续重抽取生成新对象，不应直接复活旧对象，而应新增对象并通过关联关系区分

**9.7 重新打开规则**

已 confirmed 或 modified 的对象，在以下场景允许重新进入 reviewing：

- 关联冲突新出现

- 规则版本升级命中新的高风险

- 人工发现明确错误

- 上游原文版本切换

**十、风险与冲突治理状态流转规则**

**10.1 风险对象状态流转**

适用对象：

- risk_item

建议拆分两组状态：

- reviewStatus

- resolutionStatus

**10.2 风险复核状态流转**

draft / pending_review → reviewing → confirmed / modified / rejected

适用逻辑与通用对象一致。

**10.3 风险处理状态流转**

open → in_progress → accepted / mitigated / clarified / closed

**说明**

- open：风险已识别，尚未处理

- in_progress：已有处理动作进行中

- accepted：风险被接受但尚未消除

- mitigated：风险已缓释

- clarified：通过澄清或解释获得收敛

- closed：风险已正式关闭

**10.4 风险关闭规则**

风险进入 closed 前，必须满足以下其一：

- 已有人工确认结论

- 已生成澄清结果并明确可关闭

- 已通过修改对象或补充资料消除风险

- 已决定接受风险且不再继续处理

**10.5 风险重开规则**

风险已 closed 后，如出现：

- 新版本补遗推翻原判断

- 新冲突出现

- 关联对象被重新驳回\
  则允许：\
  closed → open

---

**10.6 冲突对象状态流转**

适用对象：

- conflict_item

建议主路径：

detected → pending_review → reviewing → resolved / ignored → closed

**各状态定义**

- detected：系统已识别冲突

- pending_review：等待定案

- reviewing：人工处理中

- resolved：已形成最终结论

- ignored：确认为可忽略冲突

- closed：处理结束，不再活跃

**10.7 冲突定案规则**

冲突进入 resolved 必须满足：

- 有 finalResolution

- 有定案动作记录

- 若影响关键对象，必须同时更新相关对象或触发其重新复核

**10.8 冲突回退规则**

若定案被证明错误，可：\
resolved → reviewing\
并保留原定案历史，不允许覆盖抹除。

---

**十一、复核任务状态流转规则**

**11.1 适用对象**

- review_task

**11.2 主流转路径**

pending_assign → assigned → reviewing → confirmed / modified / rejected → closed

**11.3 各状态定义**

**pending_assign**

任务已生成，但未明确责任人。

**assigned**

任务已分派，但尚未正式处理。

**reviewing**

复核人已开始处理。

**confirmed**

复核人确认对象原值可用。

**modified**

复核人修改后确认。

**rejected**

复核人明确驳回。

**closed**

复核任务生命周期结束。

**11.4 任务与对象联动规则**

- review_task.confirmed 应推动目标对象进入 confirmed

- review_task.modified 应推动目标对象进入 modified

- review_task.rejected 应推动目标对象进入 rejected

- 对象已被其他路径重新处理时，可将历史复核任务转为 closed

**11.5 重新分派规则**

assigned / reviewing → assigned

并更新：

- assignedTo

- 最近操作人

- 最近更新时间

但不得丢失原处理记录。

---

**十二、快照与输出状态流转规则**

**12.1 适用对象**

- asset_export_snapshot

**12.2 主流转路径**

generating → generated → published / invalidated

**12.3 各状态定义**

**generating**

快照构建中。

**generated**

快照已形成，但尚未正式用于下游。

**published**

快照已被正式认可，可供下游模块消费。

**invalidated**

快照因上游对象变化、规则变化或人工回退而失效。

**12.4 快照生成前置条件**

快照进入 generated 前应满足：

- 当前版本主链批次至少 partially_succeeded

- 核心对象已形成

- 关键统计可计算

- 快照结构校验通过

**12.5 快照发布前置条件**

快照进入 published 前应满足：

- 强制人工确认对象已处理

- blockingIssueCount = 0

- 当前版本 assetReadyFlag = true

- 发布动作已被人工触发并留痕

**12.6 快照失效规则**

以下任一情况可导致：\
published / generated → invalidated

- 当前版本关键对象重新进入 reviewing

- 新冲突出现且未解决

- 新 critical 风险出现

- 新批次产出取代旧快照依据

- 人工主动执行回退发布动作

---

**十三、批次推进规则与对象联动规则**

**13.1 主链任务推进对对象状态的影响**

**接入与结构阶段完成后**

- source_document、document_page、source_segment、document_section_node 默认进入 valid

**对象抽取阶段完成后**

- 新生成业务对象默认进入 draft

- 若规则判断可自动通过，则可直接进入 confirmed

- 若命中复核规则，则进入 pending_review

**规则校验阶段完成后**

- 生成 risk_item、conflict_item

- 更新目标对象的 riskLevel、isConflicted 等字段

- 必要时把对象从 draft 推到 pending_review

**复核分流阶段完成后**

- 生成 review_task

- 更新项目与版本的 reviewStatus

**13.2 批次补跑规则**

若批次 partially_succeeded，补跑时：

- 不应整体重置所有对象状态

- 应只对受影响对象域更新状态

- 若新结果替代旧结果，旧对象可转为 superseded 或 invalid

**13.3 批次重跑规则**

若整批重跑：

- 新建新批次

- 原批次状态不变

- 新批次成为当前有效批次后，版本统计与页面默认切换至新批次结果

- 旧快照可保持存在，但若引用对象已失效，应标记 invalidated

**十六、统计口径与状态联动规则**

**16.1 首页统计必须响应以下状态变化**

- 对象 confirmed / modified / rejected

- 风险 open / closed / mitigated / clarified

- 冲突 resolved / closed

- 复核任务 pending / reviewing / confirmed / modified / rejected

- 快照 generated / published / invalidated

**16.2 统计刷新触发点**

以下动作完成后应触发 Overview 和专题统计刷新：

- 创建新批次

- 批次状态变化

- 提交复核

- 关闭风险

- 定案冲突

- 生成快照

- 发布快照

- 回退快照

- 重新打开风险/冲突/对象

- 页面切换

- 原文定位

- 展开/折叠抽屉

- 列表局部排序

- 仅查看详情

**十七、状态冲突处理规则**

**17.1 禁止出现的典型矛盾状态**

以下组合视为不合法：

1.  对象 reviewStatus='confirmed'，但存在活跃 review_task.reviewStatus='reviewing'

2.  风险 resolutionStatus='closed'，但对应对象仍处于 pending_review 且未有结论

3.  冲突 resolved，但 finalResolution 为空

4.  快照 published，但 assetReadyFlag=false

5.  批次 failed，但版本 parseStatus='parsed'

6.  版本 assetStatus='exported'，但存在未处理的阻塞问题

**17.2 处理原则**

发现上述矛盾时：

- 优先触发系统修正任务或治理提醒

- 必要时自动回退更高层状态

- 页面应显示一致性异常提示，不允许静默忽略

**十八、建议固化的状态流转校验规则**

建议在规则层和动作层增加以下校验：

**18.1 对象确认前校验**

- 是否命中强制人工确认对象类型

- 是否存在未解决冲突

- 是否存在 critical 风险关联

- 是否缺失原文依据

**18.2 快照生成前校验**

- 是否存在至少一条有效要求

- 是否存在至少一棵有效框架树或明确的缺失说明

- 是否存在模板识别结果或明确无模板说明

- 是否规则校验已完成

- 是否统计视图可计算

**18.3 快照发布前校验**

- blockingIssueCount 是否为 0

- 强制复核任务是否清零

- 当前版本是否为有效当前版本

- 快照状态是否为 generated

**十九、实现层建议**

**19.1 状态推进不应散落**

建议统一在以下层承载：

- 批次推进：orchestrator/status/

- 对象状态推进：review/ 与对象域 service

- 风险/冲突状态推进：risks/

- 快照状态推进：snapshots/publishers/

- 页面状态映射：shared/constants/ + 前端通用 Badge 组件

**19.2 建议增加统一状态推进工具**

例如：

- advanceBatchStatus()

- advanceObjectReviewStatus()

- advanceRiskResolutionStatus()

- advanceConflictStatus()

- advanceSnapshotStatus()

统一由这些工具负责合法性校验和留痕，不建议每个 service 自己写状态跳转。

**19.3 页面层只做映射**

前端页面不得自行判断“这个按钮大概可以点吧”，而应基于后端返回的状态和值，或基于共享状态规则函数计算。\
建议建立统一函数，例如：

- canConfirmObject()

- canModifyObject()

- canCloseRisk()

- canResolveConflict()

- canGenerateSnapshot()

- canPublishSnapshot()

**二十、对后续 AI Coding 的直接约束**

基于本文件，后续 AI Coding 必须遵守以下规则：

第一，任何状态变化都必须明确写出触发动作、前置状态、目标状态和留痕对象，不允许直接改字段。\
第二，任何批次推进都必须遵守本文件定义的主路径和异常路径，不允许自造 done、processing、finished 等平行状态。\
第三，任何对象如属于强制人工确认范围，均不得通过快捷逻辑直接跳到最终可用态。\
第四，任何页面按钮的可用条件都必须建立在本文件状态规则之上，不允许页面自己推断。\
第五，任何统计卡、筛选项和 Badge 展示都必须基于本文件状态定义统一映射，不允许各页自定义口径。\
第六，任何快照发布都必须做发布前校验，不允许“只要能生成就能发布”。\
第七，任何回退动作都必须保留历史，不允许覆盖式抹去原状态。\
第八，后续如新增对象域或新增治理模块，必须先补充其状态流转规则，再进入实现，不允许“先上对象后补状态”。

**二十一、正式结论**

综上所述，“招标文件智能审阅中枢”的对象状态流转、批次推进和页面联动，必须围绕“主链稳定、动作驱动、人工边界明确、异常可回退、页面统一映射”的原则统一建设。项目与版本负责总体状态，文件与批次负责运行状态，业务对象负责资产可用状态，风险与冲突负责治理状态，复核任务负责人工处理状态，快照负责对下游输出状态，页面层只负责统一映射与统一响应。
