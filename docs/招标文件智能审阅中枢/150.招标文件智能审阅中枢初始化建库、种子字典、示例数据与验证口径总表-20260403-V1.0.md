**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“初始化与验证总控章”。其上承对象模型、状态体系、任务编排、页面骨架和查询统计，其下直接约束以下实现内容：

一是约束数据库初始化方式、迁移顺序和重建策略。\
二是约束基础枚举、规则字典、状态映射、页面默认配置和导出配置的种子装载方式。\
三是约束示例项目、示例版本、示例文件、示例要求、示例风险、示例模板、示例材料、示例复核和示例快照的组织方式。\
四是约束验证脚本、验证清单和联调验收口径。\
五是约束后续 AI Coding 不得在模块内部私自塞入平行的测试字典和临时数据。

如果没有本文件，后续 AI Coding 极易出现以下问题：\
第一，Schema 能建，但数据库初始为空，页面大面积空态，无法验证中枢全链路。\
第二，状态、枚举、规则和页面筛选项没有统一种子来源，前后端只能靠硬编码兜底。\
第三，示例数据覆盖不足，很多页面虽然开发完成，但一打开没有可验证对象。\
第四，联调时每个人各造一份测试数据，最终页面、规则、快照、导出结果全对不上。\
第五，环境重建后不知道该看哪些对象、点哪些按钮、哪些结果算通过，只能凭感觉验证。

因此，本文件的目标，是把“怎么起库、怎么装字典、怎么装演示数据、怎么做验证”一次性定稳。

**三、总体设计原则**

**3.1 绿色建库原则**

当前处于测试与重构阶段，旧代码和旧数据库均不保留。因此，初始化策略必须坚持“绿色建库”原则，即：

- 不做旧库迁移兼容

- 不做旧数据清洗承接

- 直接按新中枢结构建库

- 可随时一键重建

**3.2 结构先于数据原则**

初始化顺序必须是：

**Schema → 基础字典 → 规则种子 → 示例业务数据 → 验证任务**

不允许先塞业务数据，再临时补字典或临时改表结构。

**3.3 种子分层原则**

全部初始化数据统一分为四层：

- 基础字典种子

- 规则参数种子

- 示例业务数据

- 验证基准数据

四层不可混装。

**3.4 示例数据覆盖主链原则**

示例数据必须覆盖中枢主链：

**项目建立 → 文件接入 → 批次解析 → 对象生成 → 风险识别 → 复核分流 → 对象确认 → 快照生成**

如果某类数据不能帮助验证主链路，就不应优先进入首批示例集。

**3.5 演示与真实隔离原则**

示例项目、示例批次、示例快照必须有明确标识，不得与真实业务项目混淆。\
建议通过：

- 独立命名

- 明确 demo_flag

- 独立项目编号前缀\
  等方式隔离。

**3.6 可重复执行原则**

建库、种子装载、示例数据装载和验证脚本必须可重复执行。\
重复执行后，环境结果要么完全一致，要么按明确规则覆盖，不能每次跑完都长得不一样。

**3.7 验证先口径后页面原则**

不是看到页面有数据就算成功，而是必须先有验证口径，然后再看页面、接口、任务和导出是否符合口径。

**九、示例业务数据总体设计原则**

示例业务数据不能只是“每张表来一条”，而必须支撑中枢完整验证。建议示例数据设计遵循以下原则：

**9.1 一主两辅原则**

建议初始化至少准备三类示例项目：

**示例项目 A：标准完整型**

特点：

- 结构完整

- 要求齐全

- 评分、技术、框架、模板、材料全都有

- 适合验证主链路

**示例项目 B：高风险冲突型**

特点：

- 存在多个高风险项

- 存在时间冲突、金额冲突

- 存在原样响应模板

- 适合验证风险、冲突、复核、阻塞逻辑

**示例项目 C：部分成功与补跑型**

特点：

- 主链成功

- 部分专题任务故意缺失

- 至少一个批次 partially_succeeded

- 适合验证批次补跑、页面提示和快照边界

**9.2 版本覆盖原则**

每个示例项目至少准备：

- 1 个主版本\
  如需增强验证，再增加：

- 1 个补遗版本或答疑版本

**9.3 状态覆盖原则**

示例数据必须覆盖以下状态至少一次：

- draft

- pending_review

- reviewing

- confirmed

- modified

- rejected

- open

- mitigated

- resolved

- generated

- published

- invalidated

- partially_succeeded

- failed

否则很多页面和动作无法验证。

  - 批次1：partially_succeeded

- 批次2 中：
  - 补齐模板变量和材料清单

- Overview 可看到最近批次状态变化

- 页面可提示“曾存在部分成功批次”

**十六、关键验证场景建议**

除了表格化验证，还应准备三条固定“演示/验收路线”。

**16.1 路线一：标准完整项目验收路线**

**16.2 路线二：高风险冲突项目验收路线**

**二十、对后续 AI Coding 的直接约束**

基于本文件，后续 AI Coding 必须遵守以下规则：

第一，任何新环境初始化都必须先执行 schema，再执行种子，再执行示例数据，再执行验证，不允许跳步。\
第二，任何页面、接口、规则、任务所依赖的字典与默认配置，都必须来自统一种子，不允许写在代码常量里偷偷兜底。\
第三，任何示例数据都必须服务于主链路验证，不允许为了某个页面好看临时乱加演示数据。\
第四，任何验证结果都必须有明确口径，不允许只凭“页面好像能看”就算通过。\
第五，任何新增对象域、新增专题页、新增快照类型、新增规则包，都必须同步补充对应的种子、示例数据和验证项。\
第六，任何重建脚本、清理脚本和装载脚本都必须可重复执行，不允许一次性脚本。\
第七，后续若接入企业文档规范资产池和生成装配规则资产池，也必须沿用本文件的初始化哲学：结构先行、字典统一、示例可复现、验证有口径。\
第八，若示例数据与真实业务数据共库存在，必须提供明确的 demo 标识与筛选隔离能力，不允许混淆。

**二十一、正式结论**

综上所述，“招标文件智能审阅中枢”的初始化建库、种子字典、示例数据和验证口径，必须作为一个整体统一设计：数据库结构要可绿色重建，字典与规则要可统一装载，示例项目要能覆盖主链路与关键状态，验证清单要能直接判断系统是否真正可用。只有这样，后续 AI Coding 才能在一个稳定、可重建、可演示、可验收的基础环境上快速推进，而不是每次都从零猜测“现在这套库到底算不算搭好了”。

本文件一经确认，后续建库脚本、种子脚本、示例数据脚本、验证脚本、联调验收和环境重建，均应以本文件为统一上位依据，不再允许各模块各自准备平行字典、平行示例和临时验证口径。
