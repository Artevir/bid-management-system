本文件的核心任务，不是再重复前面各专题文档的内容，也不是简单列几个测试步骤，而是要正式回答以下八个关键问题：

第一，整套“招标文件智能审阅中枢”从技术联调到业务联调，应按什么顺序推进，才能最快发现关键问题。\
第二，哪些验证属于开发自测，哪些属于模块联调，哪些属于整体验收，必须分层。\
第三，演示不是打开几个页面看一看，而是要围绕主链路把“文件进入—对象生成—风险识别—人工确认—快照输出”走通，那么标准演示路径应如何设计。\
第四，验收不能只看页面是否能打开，而必须看对象是否可追溯、状态是否一致、按钮是否合规、统计是否可信、快照是否可用，这些口径必须统一。\
第五，如何把前面所有文档真正收束成一份可执行的验收标准，而不是停留在设计稿层。\
第六，哪些问题属于可接受缺陷，哪些问题属于阻塞上线缺陷，必须明确分级。\
第七，如何让后续每次代码大调整、规则调整、模型调整或页面重构之后，都能用这份文档快速复验整套系统是否仍然成立。\
第八，如何为未来企业文档规范资产池、生成装配规则资产池和标书生成模块提供可复用的联调与验收方法论，而不只是当前模块的一次性文档。

因此，本文件既是联调清单，也是演示脚本，更是整体验收总表。

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“联调与验收总控章”。其上承前面全部设计与实现文档，其下直接约束以下内容：

一是约束开发完成后的联调顺序。\
二是约束演示环境、示例项目和示例数据的使用方式。\
三是约束功能验收、数据验收、状态验收、页面验收和输出验收的统一口径。\
四是约束问题分级、缺陷关闭和是否具备进入下一阶段的判断标准。\
五是约束后续所有回归验证都应优先复用本文件，而不是每次重新发明一套验收表。

如果没有本文件，后续 AI Coding 极易出现以下问题：\
第一，各模块都说“自己能跑”，但整条主链路并没有真正打通。\
第二，演示时靠临场点击，路径不稳定，无法体现中枢真正价值。\
第三，验收标准模糊，最后只能靠主观感觉判断“差不多”。\
第四，统计、状态、原文联动、复核、快照等关键能力是否一致，没有统一检查表。\
第五，后续一旦模型、规则或页面改动，很难快速判断系统到底有没有被改坏。

因此，本文件的目标，是把“联什么、怎么演、验收到什么程度算通过”一次性定稳。

**三、总体设计原则**

**3.1 主链路优先原则**

整体验收必须优先围绕主链路展开，不允许只做局部页面验收。\
主链路固定为：

**项目建立 → 文件接入 → 批次解析 → 对象生成 → 风险/冲突识别 → 复核分流 → 关键对象确认 → 快照生成 → 快照发布**

**3.2 分层联调原则**

联调统一分为四层：

- 结构层联调

- 服务层联调

- 页面层联调

- 业务链路联调

必须逐层推进，不建议直接跳到页面端看结果。

**3.3 可追溯原则**

任何演示与验收结果，都必须能追溯到：

- 示例项目

- 示例版本

- 示例批次

- 具体对象

- 原文依据

- 操作留痕

**3.4 场景化原则**

验收必须建立在场景上，而不是建立在孤立 API 或孤立表格上。\
因此，必须至少围绕：

- 标准完整型

- 高风险冲突型

- 部分成功补跑型\
  三类示例项目开展验收。

**3.5 指标与动作并重原则**

不能只验“看见了什么”，还必须验“能不能操作”和“操作后状态是否正确联动”。

**3.6 阻塞问题优先原则**

对于中枢级模块来说，真正阻塞验收通过的，不是样式细节，而是：

- 原文不可追溯

- 状态不一致

- 关键动作失效

- 快照口径不可信

- 风险与冲突逻辑失真

- 统计口径前后不一致

**3.7 可重复验收原则**

任何验收项都必须尽量做到：

- 可重复执行

- 可明确判断结果

- 可记录通过/失败

- 可用于回归验证

**四、联调整体分层**

建议整套联调按以下五层推进。

**4.1 第一层：初始化与结构联调**

验证：

- 库是否正确建立

- 种子是否正确装载

- 示例数据是否可读

- 核心表与核心关系是否成立

**4.2 第二层：任务与对象联调**

验证：

- 批次是否可执行

- 任务是否可记录

- 对象是否正确生成

- 风险、冲突、复核是否能联动生成

**4.3 第三层：接口与页面联调**

验证：

- 标准接口是否返回正确 DTO

- 页面是否按标准视图消费

- 详情、原文、动作区是否联动正常

**4.4 第四层：治理链路联调**

验证：

- 发起复核

- 提交复核

- 关闭风险

- 冲突定案

- 快照生成与发布

- 状态与统计刷新

**4.5 第五层：整体验收与演示联调**

验证：

- 三条标准演示路径

- 全部关键指标

- 全部关键页面

- 全部关键动作

- 快照与导出可用性

**九、标准演示路径 A 详单**

**A01 打开项目总览**

检查：

- 项目名称、项目编号、当前版本

- requirementCount、riskCount、templateCount 等统计

- 最近批次状态

- 快照状态

通过标准：

- 总览信息完整

- 统计卡有值且合理

**A02 进入要求页**

检查：

- 列表有多类要求

- 筛选可按 requirementType 与 reviewStatus 工作

- 选中要求后详情正确

- 原文页码与原文高亮可联动

通过标准：

- 列表、详情、原文三者一致

**A03 进入评分页**

检查：

- 有评分方案总分

- 有评分细项

- 有证明材料提示

通过标准：

- 评分项数与评分统计一致

**A04 进入技术页**

检查：

- 有参数组与参数项

- 星号项、不允许负偏离项可筛出

通过标准：

- 技术参数关键标识正确

**A05 进入框架页**

检查：

- 有框架树

- 一级节点、必选节点、模板节点可识别

- 节点详情可显示来源页码和关联要求

通过标准：

- 框架树与详情联动正确

**A06 进入模板页**

检查：

- 有模板列表

- 有固定格式模板和变量列表

- 原样响应模板标识正确

通过标准：

- 模板与变量可解释、可追溯

**A07 进入材料页**

检查：

- 材料清单存在

- 必须材料、签字材料、盖章材料可筛选

- 来源原因可见

通过标准：

- 材料与来源对象关联正确

**A08 生成快照**

检查：

- 可以生成 full_snapshot

- 快照状态从 generating → generated

通过标准：

- 快照成功生成且详情可打开

**十、标准演示路径 B 详单**

**B01 打开总览看阻塞状态**

检查：

- criticalRiskCount \> 0

- blockingIssueCount \> 0

- assetReadyFlag = false

通过标准：

- 首页明确提示当前不可直接输出

**B02 进入风险页**

检查：

- 高风险项可见

- 风险标题、风险说明、来源页码、命中规则可见

- 风险处置按钮可用

通过标准：

- 风险对象可信、可解释、可处理

**B03 进入冲突页**

检查：

- 有时间冲突、金额冲突

- candidateA/candidateB 可见

- 双来源原文可对照

通过标准：

- 冲突不是黑盒，用户可看出为什么冲突

**B04 发起复核与定案**

操作：

- 对至少一条冲突执行定案

- 对至少一条高风险项执行关闭或缓释

通过标准：

- 状态正确推进

- 变更日志可查

- 首页指标同步变化

**B05 尝试发布快照**

检查：

- 若仍有阻塞问题，则发布按钮不可用或被阻止

- 若阻塞消除，则可发布

通过标准：

- 快照发布前校验逻辑成立

**十一、标准演示路径 C 详单**

**C01 打开总览看部分成功批次**

检查：

- 最近批次状态为 partially_succeeded

- 页面提示部分专题可能缺失

通过标准：

- 系统没有把“部分成功”伪装成“全部正常”

**C02 检查缺失专题**

例如：

- 模板变量缺失

- 材料清单不完整

通过标准：

- 缺失有明确提示，不是静默空白

**C03 触发补跑或切换有效批次**

操作：

- 执行补跑

- 或查看成功批次切换结果

通过标准：

- 新批次建立成功

- 批次状态推进正确

**C04 再次查看页面**

检查：

- 模板变量补齐

- 材料清单补齐

- 统计卡随之更新

通过标准：

- 补跑结果真正影响页面与统计

**C05 生成新快照**

检查：

- 新快照可以生成

- 旧快照不被无痕覆盖

通过标准：

- 批次与快照之间关系清楚

| 统计可信性   | 首页与专题页统计一致、可下钻、可解释                         |
| 输出可用性   | 快照、导出、发布与失效逻辑成立                               |

**十三、验收通过标准建议**

建议将验收结果统一分为三档。

**13.1 通过**

满足以下条件：

- 三条演示路径全部打通

- 所有阻塞级缺陷已关闭

- 快照生成与发布逻辑成立

- 页面、状态、统计、原文联动一致

- 可以进入下一阶段开发或演示

**13.2 有条件通过**

满足以下条件：

- 主链打通

- 演示路径可走通大部分

- 少量非阻塞缺陷存在

- 不影响中枢核心价值展示

但必须明确遗留项与修复时间点。

**13.3 不通过**

出现以下任一情况建议判不通过：

- 主链未打通

- 原文联动失效

- 风险/冲突逻辑不可信

- 状态与页面严重不一致

- 快照不可用或发布逻辑失控

- 首页统计口径明显错误

**十四、缺陷分级建议**

**14.1 P1 阻塞级缺陷**

示例：

- 页面打不开

- 核心接口报错

- 原文跳转错误

- 关键状态推进错误

- 风险关闭后统计不更新

- 冲突定案后未写入结果

- 快照发布绕过阻塞校验

- 当前版本切换逻辑错误

P1 不关闭，不得通过验收。

**14.2 P2 高优先级缺陷**

示例：

- 统计口径局部不一致

- 某专题筛选失效

- 某类模板变量展示错误

- 变更日志缺一部分字段

- 部分导出字段缺失

P2 原则上应在验收前关闭，至少要有明确修复计划。

**14.3 P3 一般缺陷**

示例：

- 文案问题

- 页面排版细节

- 个别 Badge 颜色不统一

- 非关键提示语不准确

P3 不阻塞整体验收，但应纳入迭代清单。
