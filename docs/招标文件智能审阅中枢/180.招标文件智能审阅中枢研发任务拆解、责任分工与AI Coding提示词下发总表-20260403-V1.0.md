**一、编制目的**

《180.招标文件智能审阅中枢研发任务拆解、责任分工与AI Coding提示词下发总表-20260403-V1.0》用于在《010.招标文件智能审阅中枢核心对象模型与主表逻辑总表-20260403-V1.0》至《170.招标文件智能审阅中枢全套交付件总目录与AI Coding执行顺序总表-20260403-V1.0》已经完成对象模型、数据库逻辑、状态字典、接口体系、页面骨架、动作体系、规则边界、字段视图、任务编排、工程目录、Schema 与类型、前端组件、查询统计、状态流转、初始化方案、联调验收以及整体交付总控设计的基础上，进一步将本次“招标文件智能审阅中枢”真正落成可执行的研发任务包、职责归口、提示词下发方式、交付边界、完成判定和集成约束一次性固化下来，为后续 AI Coding 大规模并行生成、稳定合并和快速收敛提供统一任务总装稿。

本文件的核心任务，不是把 170 的执行包简单换个名字重写一遍，也不是做成一份传统软件项目的人力排期表，而是要正式解决以下八个关键问题：

第一，如何把 170 文档中已经明确的执行包，进一步拆到“可以直接下发给 AI Coding 执行”的研发任务颗粒度。\
第二，如何在不走传统长周期研发、又坚持目标架构一次性到位的前提下，让 Schema、shared types、对象域 service、任务编排、规则、前端页面、导出、初始化和验证能够并行生成但不失控。\
第三，如何明确每个研发任务包的责任归口，即谁对目录、谁对类型、谁对接口、谁对页面、谁对规则、谁对快照和谁对最终集成负责。\
第四，如何规定 AI Coding 的提示词下发模板、下发粒度、输入依据、禁止改动范围和完成判定，防止不同智能体生成出互相冲突的代码。\
第五，如何通过一套“任务包编号—责任角色—提示词包编号—目标目录—完成判定”的统一机制，让整个工程始终处于可控收敛状态。\
第六，如何把设计文稿真正转化为研发执行资产，使后续任何一个模块开发都能明确知道该读哪些上位文档、产出哪些文件、遵守哪些边界。\
第七，如何让本次 AI Coding 不仅能把代码写出来，而且能留下清晰的交付线索、责任线索和回归线索，便于后续二次扩展。\
第八，如何让这套任务拆解和提示词下发方法论，在后续企业文档规范资产池、生成装配规则资产池和标书生成模块中继续复用。

因此，本文件既是研发任务总表，也是责任分工总表，更是 AI Coding 提示词下发总控章。

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“研发执行总控章”。其上承 010–170 全部设计、工程、初始化与验收文档，其下直接约束以下实施内容：

一是约束本次研发任务包的拆分方式、命名方式和执行边界。\
二是约束不同任务包的责任归口、协同关系和集成顺序。\
三是约束 AI Coding 提示词的统一模板、统一输入、统一输出和统一验收方式。\
四是约束任何新增开发任务都必须回到本文件的任务体系中找到归属，而不是临时另起一包。\
五是约束后续所有模块提交、集成、联调和验收，都必须以本文件中的任务包完成情况为基础判断，而不再仅凭“代码差不多写完了”。

如果没有本文件，后续 AI Coding 极易出现以下问题：

第一，虽然执行包已经有了，但真正下发给 AI 的任务颗粒度还是过粗，导致生成结果超范围。\
第二，虽然大家都在写代码，但没有明确责任归口，最终目录、类型、接口和页面互相改来改去。\
第三，不同智能体提示词写法不统一，输出质量高度波动，难以合并。\
第四，任务包之间虽然有依赖，但开发过程中没有显式边界，经常出现前端抢跑、接口乱改、shared types 被反向推翻。\
第五，最后虽然功能看起来都在，但很难说明每个能力到底是谁负责收口、谁负责验收、谁负责回归。

因此，本文件的目标，是把“任务怎么拆、谁负责、怎么下发、做到什么算完成”一次性定稳。

**三、总体设计原则**

**3.1 任务包可直接下发原则**

每一个研发任务包都必须满足以下要求：

- 输入边界明确

- 目标目录明确

- 依赖文档明确

- 产出文件明确

- 禁止改动范围明确

- 完成判定明确

只有满足这六条，才算合格的 AI Coding 任务包。

**3.2 责任归口而非人海分工原则**

本文件中的“责任分工”，不是传统意义上的部门人力排班，而是本次顶尖虚拟研发团队的职责位分工。\
也就是说，责任角色的意义在于：

- 对某类任务包的技术口径负责

- 对某类目录的最终收敛负责

- 对某类提示词的下发与结果审阅负责

- 对某类模块的集成质量负责

**3.3 先总控、后对象、再页面原则**

AI Coding 必须坚持：

- 先定工程骨架、Schema、shared types、状态与查询口径

- 再做对象域 service、任务编排、规则治理

- 再做页面骨架、专题页、联动与导出

- 最后做联调修正与验收收口

不得跳过前置层直接做后层。

**3.4 单任务单目标原则**

每次下发给 AI 的任务，原则上只解决一个明确目标，例如：

- 一个 schema 文件组

- 一个 shared types 文件组

- 一个对象域 service 模块

- 一个专题页

- 一个导出器

- 一个初始化脚本组

不建议一次性下发“把整个中枢都写完”这类大而空的提示词。

**3.5 提示词标准化原则**

所有 AI Coding 提示词都必须采用统一模板，至少包含：

- 任务背景

- 上位依据

- 目标目录

- 输出文件

- 约束事项

- 禁止事项

- 完成判定

- 自测清单

**3.6 收敛优先原则**

任何 AI Coding 结果，如与既定目录、状态、字段、DTO、Query、页面骨架冲突，一律以上位文档和本文件为准进行收敛，不迁就生成结果。

**3.7 集成门槛原则**

任务包写完不等于完成，只有满足：

- 本包自测通过

- 不破坏上位 shared types

- 不越界改动无关目录

- 可被后续任务包直接依赖\
  才算可进入集成。

**五、责任分工总体规则**

**5.1 总控责任**

A01 负责：

- 判断某任务包是否可以下发

- 判断某任务包是否已完成

- 审核跨模块改动是否越界

- 对最终工程收敛结果负责

**5.2 数据底座责任**

A02 与 A03 联合负责：

- Schema 不漂移

- shared enums 不漂移

- DTO 和数据库字段口径不打架

- 所有状态与对象类型统一

**5.3 编排与规则责任**

A04 与 A05 联合负责：

- AI 任务不散落

- 规则逻辑不散落

- 状态流转不失控

- 复核与快照逻辑能闭环

**5.4 前端责任**

A08 与 A09 联合负责：

- 页面组件树不跑偏

- Query 不被页面直连替代

- Store 不被滥用

- 原文联动、详情区、动作区统一

**5.5 输出与验证责任**

A10、A11、A12 联合负责：

- 快照能生成、能发布、能失效

- 导出能复用标准视图

- 初始化环境可重复

- 三条演示路径可跑通

- 验收结果可沉淀

**十、AI Coding 提示词下发总体规则**

**10.1 下发前必须明确的六项内容**

1.  所属任务包编号

3.  目标目录与目标文件

5.  禁止改动范围

缺少任一项，不得下发。

**10.2 单次提示词推荐粒度**

单次提示词尽量控制在以下四类粒度之一：

- 一个 schema 文件组

- 一个对象域 service 文件组

- 一个专题页文件组

- 一个导出或初始化文件组

不建议单次提示词同时跨：

- schema + 页面

- shared types + 风险治理 + 前端页面

- 导出 + 原文联动 + 复核动作\
  这类多层混合任务。

**10.3 提示词下发禁令**

任何 AI Coding 提示词中都必须明确禁止以下行为：

- 擅自新建平行目录

- 擅自推翻 shared types

- 擅自修改无关对象域

- 在页面里直接写数据库访问

- 在 route handler 里直接写模型调用

- 用 any 替代已存在共享类型

- 为了跑通而临时硬编码状态和字典
