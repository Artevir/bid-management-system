**一、编制目的**

《200.招标文件智能审阅中枢可直接下发的第二批AI Coding提示词包（AI接入、任务编排、Ingestion、Requirements、Rules）-20260403-V1.0》用于在《190.招标文件智能审阅中枢可直接下发的首批AI Coding提示词包（Schema、Shared Types、Query、Bootstrap）-20260403-V1.0》已经完成首批底座层代码生成前提下，进一步把“招标文件智能审阅中枢”真正进入主链路执行的第二批关键提示词包一次性收成可直接复制下发的正式执行稿，为后续 AI Coding 从“底层结构已稳”推进到“主链任务可跑、对象可生、规则可判、复核可分流”提供统一作战底稿。

本文件聚焦第二批五类核心任务包：

- AI 接入与任务注册提示词包

- 任务编排提示词包

- Ingestion 与 Structure 提示词包

- Requirements 对象域提示词包

- Risks / Rules / Review Routing 提示词包

这五类任务包之所以被列为第二批，是因为它们共同构成“招标文件智能审阅中枢”真正的主执行链。首批四包解决的是“底座能不能立住”的问题；第二批五包解决的是“主链能不能开始跑起来”的问题。只有当这五包按统一边界落成，后续评分、技术、框架、模板、材料、快照、前端页面、导出与联调，才不至于变成空壳工程。

因此，本文件不是一般性的“继续写几个提示词”，而是本次中枢重构从静态底座正式进入动态执行层的关键分水岭文档。

---

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“第二批主链提示词总装章”。其上承 180 文档中的任务包体系和 190 文档中的首批底座提示词，其下直接约束以下执行内容：

一是约束第二批只围绕主链任务、对象生成和规则治理展开，不得过早扩展到复杂页面与导出。\
二是约束第二批必须建立在首批已通过的 Schema、shared types、Query、Bootstrap 基线之上。\
三是约束第二批各提示词包的改动目录、输出文件、禁止事项和完成判定。\
四是约束第二批不得反向推翻首批底座成果。\
五是约束后续第三批页面包、第四批专题包、第五批快照与验收包，都必须建立在本文件的第二批主链成果之上。

如果没有本文件，后续虽然已经具备首批底座能力，但真正进入主链编码时仍然容易出现以下问题：

第一，不知道第二批应该先做 AI 接入，还是先做文件接入，还是先做规则。\
第二，不同智能体可能同时去碰 orchestrator、requirements、rules，结果边界混乱。\
第三，AI 接入和任务编排很容易被写散，重新回到“service 里直接调模型”的旧路。\
第四，requirements 对象域可能在规则和复核分流尚未定稳时就先行发散。\
第五，第二批如果不先把主链跑通，后面的页面包就会缺乏真实可验证数据来源。

因此，本文件的目标，是把“第二批先发什么、怎么发、做到什么算主链成立”一次性定稳。

**三、第二批提示词包选择原则**

第二批提示词包必须同时满足以下五个条件：

**3.1 必须直接服务主链路**

也就是说，这些任务包都要能映射到以下主链中的明确环节：

**文件进入 → 文本提取 → 结构切片 → 任务注册 → 批次编排 → 要求抽取 → 风险与规则判断 → 复核分流**

**3.2 必须建立在首批成果上**

第二批不负责重新定义 Schema、DTO、Query 基线，而是必须直接复用首批成果。\
若第二批需要改 Schema 或 shared types，应先回到首批相关包修正，而不能在第二批里偷偷改。

**3.3 必须可独立审查**

每个提示词包都必须可以独立审查以下内容：

- 是否落在正确目录

- 是否遵守既定类型

- 是否没有越界

- 是否真的推进了主链

- 是否具备下一包依赖价值

**3.4 必须为第三批页面与专题包提供真实对象来源**

第二批做完后，至少要能让后续页面包消费：

- 批次状态

- 文本结构

- 要求对象

- 风险对象

- 复核任务

**3.5 必须优先保证收敛，不追求全覆盖**

第二批的目标不是把所有对象域全部做完，而是先让最关键主链成立。因此，评分、技术、框架、模板、材料等专题对象域可放在后续批次，不能挤占第二批主链任务的焦点。

**五、第二批提示词包执行顺序建议**

第二批虽然有五包，但不建议并行无序推进，建议按以下顺序下发和合并：

**第一步：PP060**

先把 AI providers、prompts、adapters、task registry 建立起来。

**第二步：PP065**

在 AI 接入基线已存在的前提下，建立 batch、definitions、runners、retry、status、hooks。

**第三步：PP070**

在 task registry 和 orchestrator 基础上，打通 ingestion 和 structure 的主链前半段。

**第四步：PP080**

在结构对象已具备的前提下，生成 requirements 主对象域。

**第五步：PP110**

在 requirements 对象已存在的前提下，做风险识别、规则命中、置信度和复核分流。

这样顺序最稳，因为：

- PP065 明显依赖 PP060

- PP070 依赖 PP065 的批次与任务执行壳

- PP080 依赖 PP070 的结构切片与章节树

- PP110 依赖 PP080 的对象域结果
