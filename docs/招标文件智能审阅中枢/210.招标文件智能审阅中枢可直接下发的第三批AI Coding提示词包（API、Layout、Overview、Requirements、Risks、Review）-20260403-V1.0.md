**210.招标文件智能审阅中枢可直接下发的第三批AI Coding提示词包（API、Layout、Overview、Requirements、Risks、Review）-20260403-V1.0**

**一、编制目的**

《210.招标文件智能审阅中枢可直接下发的第三批AI Coding提示词包（API、Layout、Overview、Requirements、Risks、Review）-20260403-V1.0》用于在《190.招标文件智能审阅中枢可直接下发的首批AI Coding提示词包（Schema、Shared Types、Query、Bootstrap）-20260403-V1.0》已完成底座层代码生成、《200.招标文件智能审阅中枢可直接下发的第二批AI Coding提示词包（AI接入、任务编排、Ingestion、Requirements、Rules）-20260403-V1.0》已完成主链执行层代码生成的前提下，进一步把“招标文件智能审阅中枢”进入前后端可见、可点、可联调、可演示阶段所需的第三批关键提示词包一次性收成可直接复制下发的正式执行稿，为后续真正实现标准接口开放、统一页面骨架建立、首页驾驶舱可见、要求页可用、风险治理页可用、复核页可用提供统一作战底稿。

本文件聚焦第三批六类核心任务包：

- API Route Handler 提示词包

- 前端 Layout 与 Common 基础提示词包

- Overview 页面提示词包

- Requirements 页面提示词包

- Risks 页面提示词包

- Review 页面提示词包

这六类任务包之所以被列为第三批，是因为它们共同构成“中枢从后台主链走向前台主界面”的第一阶段闭环。首批四包解决的是“底座是否稳”，第二批五包解决的是“主链是否能跑”，第三批六包解决的是“用户是否能真正看到、点到、确认到、治理到”。如果第三批不及时跟上，前两批就仍然只是后端工程；而如果第三批抢在前两批之前推进，就会再次回到“页面先出来、底层不稳定”的老路。

因此，本文件不是一般性的“再补几个页面提示词”，而是本次中枢重构从“后端主链成立”正式进入“前后端闭环成立”的关键分水岭文档。

---

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“第三批前后端闭环提示词总装章”。其上承 180 文档中的任务包体系、190 文档中的首批底座提示词和 200 文档中的第二批主链提示词，其下直接约束以下执行内容：

一是约束第三批只围绕 API 开放、页面骨架、总览页、要求页、风险页和复核页展开，不得过早扩展到全部专题页。\
二是约束第三批必须建立在首批底座和第二批主链已经成立的前提下。\
三是约束第三批各提示词包的改动目录、输出文件、禁止事项和完成判定。\
四是约束第三批不得反向推翻首批 shared types、第二批 orchestrator 或对象域逻辑。\
五是约束后续第四批专题页面包、第五批快照导出包、第六批联调验收包，都必须建立在本文件的第三批可见闭环成果之上。

如果没有本文件，后续虽然已经具备底座和主链，但真正进入用户可见层时仍然容易出现以下问题：

第一，不知道第三批应该先做 API，还是先做 Layout，还是先做 Requirements 页面。\
第二，API 很容易被写成 route 里直查数据库，破坏 Query 和 DTO 分层。\
第三，页面骨架若不先定，Overview、Requirements、Risks、Review 很容易各做各的。\
第四，原文联动、详情区、动作区和统计卡很容易在不同页面重复造轮子。\
第五，如果先做全部专题页而不是先做总览、要求、风险、复核这四个关键页面，联调效率会很低。

因此，本文件的目标，是把“第三批先发什么、怎么发、做到什么算前后端闭环成立”一次性定稳。

**三、第三批提示词包选择原则**

第三批提示词包必须同时满足以下五个条件：

**3.1 必须直接服务首轮可见闭环**

第三批必须优先打通以下用户可见链路：

**项目总览 → 要求列表与详情 → 原文定位 → 风险与冲突 → 复核任务与提交**

**3.2 必须建立在第二批真实数据源之上**

第三批页面和接口必须直接消费：

- batch / overview 聚合视图

- requirements 对象

- risk / conflict / reviewTask 治理对象

- sourceSegment / page 原文对象

而不是靠假数据先画页面。

**3.3 必须优先复用通用骨架**

第三批中所有页面都必须建立在统一的：

- layout

- common components

- query hooks

- state stores\
  之上，不允许各页各写各的壳。

**3.4 必须优先打通治理主链而非专题全覆盖**

第三批不追求把 scoring、technical、framework、templates、materials 一次做完，而是优先把：

- Overview

- Requirements

- Risks

- Review\
  四大关键可见页打通。

**3.5 必须便于快速联调**

第三批完成后，应当能够直接支撑 160 文档中的两条关键演示路径：

- 标准完整链路

- 高风险冲突治理链路

**五、第三批提示词包执行顺序建议**

建议第三批按以下顺序下发和合并：

**第一步：PP140**

先把 API 接口开放出来。

**第二步：PP150**

再把 layout/common 统一壳和基础组件搭起来。

**第三步：PP160A**

先做 Overview，总览页最容易验证全链是否通了。

**第四步：PP160B**

再做 Requirements + SourceView，这一页最能验证“对象—详情—原文”链路。

**第五步：PP170A**

再做 Risks，验证治理对象和双原文对照。

**第六步：PP170B**

最后做 Review，验证复核任务、动作提交和状态联动。

这样顺序最稳，因为：

- 页面必须依赖 API

- 专题页必须依赖 layout/common

- Review 又明显依赖 Risks 的治理语义和动作模式

- Requirements + SourceView 是第三批中最重要的中间桥梁

**六、第三批提示词包统一使用说明**

**6.1 一次只下发一个包**

不要把 PP140、PP150、PP160A、PP160B、PP170A、PP170B 混成一个大提示词。

**6.2 每包生成后先做“目录审查 + 组件树审查 + 状态审查”**

第三批最容易出问题的地方不只是代码是否能跑，而是：

- 组件树是否跑偏

- Query Hook 是否乱写

- Zustand 是否滥用

- 页面是否绕过 DTO / View Model

**6.3 第三批禁止抢跑复杂专题页**

第三批只做：

- Overview

- Requirements

- Risks

- Review\
  不提前进入 scoring、technical、framework、templates、materials。

**6.4 第三批必须显式复用第二批主链对象**

不能为了页面好看临时造假数据。

**6.5 每包必须输出“文件清单 + 自测清单 + 依赖说明”**

否则不进入合并。
