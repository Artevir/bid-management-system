**一、编制目的**

《230.招标文件智能审阅中枢可直接下发的第四批AI Coding提示词包（Technical）-20260403-V1.0》用于在《190.招标文件智能审阅中枢可直接下发的首批AI Coding提示词包（Schema、Shared Types、Query、Bootstrap）-20260403-V1.0》已完成底座层、《200.招标文件智能审阅中枢可直接下发的第二批AI Coding提示词包（AI接入、任务编排、Ingestion、Requirements、Rules）-20260403-V1.0》已完成主链层、《210.招标文件智能审阅中枢可直接下发的第三批AI Coding提示词包（API、Layout、Overview、Requirements、Risks、Review）-20260403-V1.0》已完成第一阶段前后端闭环、《220.招标文件智能审阅中枢可直接下发的第四批AI Coding提示词包（Scoring）-20260403-V1.0》已完成首个专题页样板的前提下，进一步把第四批专题页扩展中的 **Technical 专题包** 单独收成一份可直接复制下发的正式提示词执行稿，为后续真正落成技术参数专题页、参数组与参数项联动、关键参数识别、负偏离限制识别、原文溯源和动作入口提供统一作战底稿。

本文件只处理第四批中的 **Technical**，不混入 Framework、Templates、Materials。这样做的目的，是延续 Scoring 专题页的单专题稳步推进思路，把 Technical 专题页做成第二个“可复用的专题样板”，让页面骨架、状态管理、原文联动、动作区组织方式继续收敛，而不是在一个大提示词里把多个专题页一起下发，导致边界失控、对象语义互相污染、参数字段和页面字段口径跑偏。

因此，本文件不是“第四批继续补几个页面”的普通说明，而是 **Technical 专题页的直接作战文件**。

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“第四批 Technical 专题提示词总装章”。其上承 180 文档中的任务包体系，以及 190、200、210、220 四批提示词成果，其下直接约束以下内容：

一是约束 Technical 专题页的改动目录、组件边界、状态边界和提示词正文。\
二是约束 Technical 页面必须建立在既有 API、layout/common、Query、shared dto 和技术参数对象域之上。\
三是约束 Technical 页面不得反向推翻前面四批已经确立的页面骨架、Query 口径和状态口径。\
四是约束技术参数专题页的重点，不是做复杂炫技，而是把“参数组、参数项、星号项、不允许负偏离项、详情解释、来源页码、原文定位、动作入口”做成稳定页面。\
五是约束后续 Framework、Templates、Materials 专题页，也应沿用本文件对专题页提示词的组织方式。

如果没有本文件，后续虽然已有主链、治理链和 Scoring 专题页样板，但在扩展 Technical 专题页时仍然容易出现以下问题：

第一，技术页容易把参数组和参数项混成一张平铺大表，丧失“组—项”层级。\
第二，星号项、允许偏离项、不允许负偏离项等关键标识很容易只在后端有字段，前端没有可用表达。\
第三，技术页容易为了赶进度复用 Scoring 页外壳，但没有真正适配技术参数对象和参数语义。\
第四，技术页很容易只显示参数名称和值，不显示约束性质、单位、值类型和来源依据。\
第五，后续再做 Framework、Templates 等专题页时，无法复用技术页沉淀出的“专题页第二范式”。

因此，本文件的目标，是把“Technical 包怎么发、怎么做、做到什么算专题页成立”一次性定稳。
