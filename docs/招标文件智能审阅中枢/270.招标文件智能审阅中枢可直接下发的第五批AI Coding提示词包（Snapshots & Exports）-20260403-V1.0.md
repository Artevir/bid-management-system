**270.招标文件智能审阅中枢可直接下发的第五批AI Coding提示词包（Snapshots & Exports）-20260403-V1.0**

**一、编制目的**

《270.招标文件智能审阅中枢可直接下发的第五批AI Coding提示词包（Snapshots & Exports）-20260403-V1.0》用于在《190.招标文件智能审阅中枢可直接下发的首批AI Coding提示词包（Schema、Shared Types、Query、Bootstrap）-20260403-V1.0》已完成底座层、《200.招标文件智能审阅中枢可直接下发的第二批AI Coding提示词包（AI接入、任务编排、Ingestion、Requirements、Rules）-20260403-V1.0》已完成主链层、《210.招标文件智能审阅中枢可直接下发的第三批AI Coding提示词包（API、Layout、Overview、Requirements、Risks、Review）-20260403-V1.0》已完成第一阶段前后端闭环、《220.招标文件智能审阅中枢可直接下发的第四批AI Coding提示词包（Scoring）-20260403-V1.0》至《260.招标文件智能审阅中枢可直接下发的第四批AI Coding提示词包（Materials）-20260403-V1.0》已完成五类生成前资产专题页样板的前提下，进一步把第五批扩展中的 **Snapshots & Exports** 能力单独收成一份可直接复制下发的正式提示词执行稿，为后续真正落成“资产快照构建、快照发布、快照失效、标准导出渲染、导出注册表、正式输出命名与格式交付”提供统一作战底稿。

本文件处理第五批中的两类核心能力：

- **Snapshots**：把当前版本的可用资产冻结为标准快照，形成后续下游消费的稳定输入。

- **Exports**：把快照或受控视图渲染为正式导出文件，形成可交付、可预览、可归档的输出物。

这两类能力之所以放在第五批，是因为到第四批结束时，招标文件智能审阅中枢的“生成前资产专题页”已经基本齐全，此时最自然、也最必须推进的，不是继续补零散页面，而是把这些资产真正转化为 **可发布、可导出、可供下游使用** 的正式输出物。如果没有快照与导出层，前面再多对象和页面，也仍停留在“系统里能看”的阶段，而不能进入“系统里能交付”的阶段。

因此，本文件不是一般性的“导出功能提示词”，而是本次中枢重构从“资产可见”正式进入“资产可输出”的关键分水岭文档。

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“第五批快照与导出提示词总装章”。其上承 180 文档中的任务包体系，以及 190、200、210、220、230、240、250、260 八批提示词成果，其下直接约束以下内容：

一是约束 Snapshots 与 Exports 两类能力的改动目录、提示词边界和完成判定。\
二是约束快照构建必须建立在既有对象域、统计口径、状态流转和就绪校验基础之上。\
三是约束导出渲染必须建立在快照或受控导出视图之上，不得页面端临时拼数据。\
四是约束正式输出文件命名、格式映射、发布与失效口径。\
五是约束后续快照页、导出中心页、正式标书生成模块，也应沿用本文件对“输出层提示词”的组织方式。

如果没有本文件，后续虽然已有主链、治理链和五个专题页样板，但在扩展 Snapshots 与 Exports 时仍然容易出现以下问题：

第一，快照很容易退化成“把当前数据库对象复制一份”，没有稳定的发布、失效和再构建语义。\
第二，导出很容易退化成“页面上有什么就导什么”，导致口径和版本失控。\
第三，快照发布前的就绪校验、阻塞校验、强制复核校验很容易被绕开。\
第四，导出文件很容易各模块各起名字，不符合统一的正式命名规范。\
第五，后续标书生成模块需要稳定输入时，发现快照和导出层没有抽象清楚，只能重新返工。

因此，本文件的目标，是把“Snapshots & Exports 包怎么发、怎么做、做到什么算输出层成立”一次性定稳。

**三、Snapshots & Exports 批次选择原则**

第五批必须同时满足以下五个条件：

**3.1 必须建立在真实资产对象之上**

快照与导出只能消费已经存在并通过治理链的对象，例如：

- requirement 系列对象

- scoring 对象

- technical 对象

- framework 对象

- template / variable / table structure 对象

- material 对象

- 风险、冲突、复核状态汇总

- overview / topic stats 统计视图

不得引入假数据导出或临时拼装快照。

**3.2 必须优先体现“输出受控”**

这批能力的核心，不是“会导出文件”，而是：

- 何时允许生成快照

- 何时允许发布快照

- 何时必须让快照失效

- 导出基于什么视图

- 导出文件如何命名

- 导出结果如何留痕

**3.3 必须复用统一口径**

快照与导出必须复用：

- shared export views

- 既有查询口径

- 既有状态口径

- 既有统计口径

- 既有正式命名规则

不得另起一套“导出版字段口径”。

**3.4 必须与前面专题资产形成上下游闭环**

Snapshots & Exports 必须把第四批的五类专题资产真正连接起来，使系统从“可看”进入“可交付”。

**3.5 必须便于后续页面与生成模块复用**

第五批做完后，应能自然成为后续：

- 快照页

- 导出中心

- 生成装配规则页

- 标书生成模块

的稳定上游能力。

**四、第五批提示词包总表**

本文件包含两个正式提示词包。

| **提示词包编号** | **对应任务包** | **提示词包名称**   | **目标**                             |
| ---------------- | -------------- | ------------------ | ------------------------------------ |
| PP130A           | WP130          | Snapshots 提示词包 | 建立快照构建、发布、失效与读取能力   |
| PP130B           | WP130          | Exports 提示词包   | 建立导出注册表、渲染器与正式输出能力 |

说明：\
这里把 Snapshots 和 Exports 拆成两个提示词包，是因为：

- Snapshots 更偏向 **版本化冻结、状态推进、发布控制**

- Exports 更偏向 **输出装配、渲染格式、文件命名、交付行为**

虽然都属于输出层，但职责明显不同，拆开后更利于 AI Coding 稳定生成与审阅收敛。

**五、第五批提示词包执行顺序建议**

建议第五批按以下顺序下发和合并：

**第一步：PP130A**

先把快照构建、就绪校验、发布与失效语义建立起来。

**第二步：PP130B**

再在快照层稳定后，建立导出注册表和多格式渲染层。

这样顺序最稳，因为：

- 导出层应优先消费快照，而不是直接消费活数据

- 正式导出文件的稳定性，取决于快照输入的稳定性

- 快照生命周期若不先定，导出会很容易变成“实时拼装输出”，后续很难收敛

**六、第五批提示词包统一使用说明**

**6.1 一次只下发一个包**

不要把 PP130A 和 PP130B 混成一个大提示词。

**6.2 每包生成后先做“目录审查 + 生命周期审查 + 命名审查”**

第五批最容易出问题的地方不只是代码能不能跑，而是：

- 快照生命周期是否成立

- 导出是否真的基于受控视图

- 输出命名是否统一

- 发布与失效是否受控

**6.3 第五批禁止抢跑快照页或导出页**

第五批先做后端输出能力，不直接做页面层。

**6.4 第五批必须显式复用 shared export views**

不能在导出器里临时发明匿名结构。

**6.5 每包必须输出“文件清单 + 自测清单 + 依赖说明”**

否则不进入合并。
