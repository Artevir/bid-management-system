**280.招标文件智能审阅中枢可直接下发的第六批AI Coding提示词包（Snapshot API、Export API、Snapshot Page、Export Center）-20260403-V1.0**

**一、编制目的**

《280.招标文件智能审阅中枢可直接下发的第六批AI Coding提示词包（Snapshot API、Export API、Snapshot Page、Export Center）-20260403-V1.0》用于在《190.招标文件智能审阅中枢可直接下发的首批AI Coding提示词包（Schema、Shared Types、Query、Bootstrap）-20260403-V1.0》已完成底座层、《200.招标文件智能审阅中枢可直接下发的第二批AI Coding提示词包（AI接入、任务编排、Ingestion、Requirements、Rules）-20260403-V1.0》已完成主链层、《210.招标文件智能审阅中枢可直接下发的第三批AI Coding提示词包（API、Layout、Overview、Requirements、Risks、Review）-20260403-V1.0》已完成第一阶段前后端闭环、《220.招标文件智能审阅中枢可直接下发的第四批AI Coding提示词包（Scoring）-20260403-V1.0》至《260.招标文件智能审阅中枢可直接下发的第四批AI Coding提示词包（Materials）-20260403-V1.0》已完成五类生成前资产专题页样板、《270.招标文件智能审阅中枢可直接下发的第五批AI Coding提示词包（Snapshots & Exports）-20260403-V1.0》已完成输出层后端能力的前提下，进一步把第六批“输出层可见闭环”所需的 **Snapshot API、Export API、Snapshot Page、Export Center** 四类能力，正式收成可直接复制下发的提示词执行稿，为后续真正实现“快照可控、导出可调、输出可见、发布可审”的前后端闭环提供统一作战底稿。

本文件聚焦四类核心能力：

- **Snapshot API**：把快照的创建、查询、发布、失效、重建以标准接口形式暴露出去。

- **Export API**：把导出注册表、导出触发、导出结果返回和格式能力以标准接口形式暴露出去。

- **Snapshot Page**：把版本快照、快照状态、就绪校验、发布与失效动作做成前端页面。

- **Export Center**：把可导出的对象、导出类型、来源类型、目标格式、正式文件命名和导出动作做成前端页面。

这四类能力之所以放在第六批，是因为第五批已经把快照与导出后端能力做实，接下来最自然也最必要推进的，不是继续补零散对象页，而是把“输出层能力真正开放到 API 与页面”。只有这样，前面所有专题资产和输出层逻辑，才真正从“代码里存在”进入“用户可见、可控、可操作、可验收”的阶段。

因此，本文件不是一般性的“再补几个接口和页面”，而是本次中枢重构从“后端输出层成立”正式进入“输出层前后端闭环成立”的关键分水岭文档。

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“第六批输出层前后端闭环提示词总装章”。其上承 180 文档中的任务包体系，以及 190、200、210、220、230、240、250、260、270 九批提示词成果，其下直接约束以下内容：

一是约束 Snapshot API、Export API、Snapshot Page、Export Center 四类能力的改动目录、提示词边界和完成判定。\
二是约束第六批必须建立在第五批 Snapshots & Exports 后端能力已存在的前提下，不得反向在页面包中重写快照和导出逻辑。\
三是约束 API 仍然只承担边界职责，页面仍然只承担展示和动作触发职责。\
四是约束输出层页面必须消费标准 API、标准 DTO、标准快照视图和标准导出视图。\
五是约束后续第七批演示、回归和整体验收，也应沿用本文件对“输出层可见闭环”的组织方式。

如果没有本文件，后续虽然已有 Snapshots 与 Exports 后端能力，但真正进入输出层可见闭环时仍然容易出现以下问题：

第一，快照 API 很容易被写成各模块各自调用内部 service，路径不统一。\
第二，导出 API 很容易退化成“某个页面点一下直接拼文件”，无法形成统一出口。\
第三，Snapshot Page 很容易只做成一个列表页，没有发布、失效、重建和校验摘要。\
第四，Export Center 很容易只做一个下载按钮页，没有注册表、来源类型、格式能力和命名预览。\
第五，后续进入演示和验收阶段时，发现输出层虽存在代码，但缺少标准接口和页面抓手，无法真正演示。

因此，本文件的目标，是把“第六批先发什么、怎么发、做到什么算输出层前后端闭环成立”一次性定稳。

**三、第六批提示词包选择原则**

第六批提示词包必须同时满足以下五个条件：

**3.1 必须直接服务输出层可见闭环**

第六批必须优先打通以下链路：

**快照构建结果 → Snapshot API → Snapshot Page**\
**导出注册与渲染结果 → Export API → Export Center**

**3.2 必须建立在第五批成果之上**

第六批不重新定义：

- snapshot builder

- export registry

- export renderer

- shared export views

- snapshot status 语义

而是必须直接复用第五批成果。\
若第六批需要补快照或导出底层逻辑，应先回到第五批相关包修正，而不能在 API 或页面层偷补。

**3.3 必须优先控制边界而非追求炫技**

第六批的关键不是做花哨页面，而是确保：

- API 路径清晰

- 输出视图统一

- 页面动作受控

- 正式命名稳定

- 发布与失效可见

- 导出来源可解释

**3.4 必须便于第七批演示与验收复用**

第六批做完后，应直接支撑：

- 快照构建与发布演示

- 导出能力演示

- 正式输出命名检查

- 阻塞校验与失效演示

**3.5 必须保持“后端输出层”和“前端输出层”分离**

API 只做边界，页面只做消费与动作触发。\
不得在页面里直接调用 builder / renderer。\
不得在 API 里直接重写页面所需视图。

拆开后更利于 AI Coding 稳定生成与审阅收敛。

先把 Snapshot API 打通。

再把 Export API 打通。

在 Snapshot API 已存在的前提下，建立 Snapshot Page。

**第四步：PP160F**

在 Export API 已存在的前提下，建立 Export Center。

这样顺序最稳，因为：

- 页面必须建立在标准 API 之上。

- 快照页和导出中心页虽然都属于输出层，但一个偏生命周期，一个偏输出动作，先分开更利于收敛。

- 第七批的演示与验收，会直接依赖这四包。

**六、第六批提示词包统一使用说明**

**6.5 每包必须输出“文件清单 + 自测清单 + 依赖说明”**
