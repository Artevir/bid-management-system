**一、编制目的**

《290.招标文件智能审阅中枢可直接下发的第七批AI Coding提示词包（Regression、Demo、Acceptance）-20260403-V1.0》用于在《190.招标文件智能审阅中枢可直接下发的首批AI Coding提示词包（Schema、Shared Types、Query、Bootstrap）-20260403-V1.0》已完成底座层、《200.招标文件智能审阅中枢可直接下发的第二批AI Coding提示词包（AI接入、任务编排、Ingestion、Requirements、Rules）-20260403-V1.0》已完成主链层、《210.招标文件智能审阅中枢可直接下发的第三批AI Coding提示词包（API、Layout、Overview、Requirements、Risks、Review）-20260403-V1.0》已完成第一阶段前后端闭环、《220.招标文件智能审阅中枢可直接下发的第四批AI Coding提示词包（Scoring）-20260403-V1.0》至《260.招标文件智能审阅中枢可直接下发的第四批AI Coding提示词包（Materials）-20260403-V1.0》已完成五类专题页样板、《270.招标文件智能审阅中枢可直接下发的第五批AI Coding提示词包（Snapshots & Exports）-20260403-V1.0》已完成输出层后端能力、《280.招标文件智能审阅中枢可直接下发的第六批AI Coding提示词包（Snapshot API、Export API、Snapshot Page、Export Center）-20260403-V1.0》已完成输出层前后端闭环的前提下，进一步把第七批收口阶段的 **Regression、Demo、Acceptance** 三类能力，正式收成可直接复制下发的提示词执行稿，为后续真正实现“可回归、可演示、可验收、可归档”的最后闭环提供统一作战底稿。

本文件聚焦三类核心能力：

- **Regression**：把前面各批能力固化成可重复执行的回归检查清单、验证脚本与结果记录。

- **Demo**：把系统核心价值路径固化成标准演示脚本、演示前检查和演示过程辅助资料。

- **Acceptance**：把整体验收口径、缺陷分级、通过判定、验收结论和归档材料固化为正式收口资产。

这三类能力之所以放在第七批，是因为到第六批结束时，系统已经基本具备：

- 底座层

- 主链层

- 治理层

- 专题页层

- 输出层

- 输出层前后端闭环

此时最自然、也最必要推进的，不再是继续补对象或补页面，而是把整套系统真正收束成 **可验证、可展示、可签收** 的交付形态。如果没有这一批，前面再完整，也仍然停留在“工程已成型”的状态，而不是“交付已成形”的状态。

因此，本文件不是一般性的“补一份测试说明”，而是本次中枢重构从“系统已完成”正式进入“系统可交付”的关键分水岭文档。

**二、文档定位**

本文件在整套“招标文件智能审阅中枢”方案中的定位，应明确为“第七批回归、演示与验收提示词总装章”。其上承 180 文档中的任务包体系，以及 190 至 280 各批提示词成果，其下直接约束以下内容：

一是约束 Regression、Demo、Acceptance 三类能力的改动目录、提示词边界和完成判定。\
二是约束第七批必须建立在前六批系统能力已经存在的前提下，不得在本批提示词中重写前置业务逻辑。\
三是约束 Regression 必须可重复、Demo 必须可复现、Acceptance 必须可归档。\
四是约束演示与验收必须严格围绕 160 文档既有主链路和通过口径，而不是临时发挥。\
五是约束本批形成的结果，应直接成为后续正式交付、回归复验、二次迭代和对外汇报的基础资产。

如果没有本文件，后续虽然已有整套系统能力，但真正进入交付收口时仍然容易出现以下问题：

第一，回归验证只停留在口头层，缺少标准化检查脚本和结果记录。\
第二，演示过程依赖临场发挥，路径不稳定，重点不聚焦。\
第三，验收口径分散在多个文档里，没有收口成可执行材料。\
第四，缺陷分级、通过条件、验收结论和归档材料缺少统一模板。\
第五，后续每次回归或向领导演示时，都要临时重新准备。

因此，本文件的目标，是把“第七批先发什么、怎么发、做到什么算真正交付可收口”一次性定稳。

**三、第七批提示词包选择原则**

第七批提示词包必须同时满足以下五个条件：

**3.1 必须直接服务交付收口**

第七批必须优先打通以下链路：

**回归验证结果 → 演示脚本与演示路径 → 验收结论与归档材料**

**3.2 必须建立在前六批成果之上**

第七批不重新定义：

- Schema

- shared types

- Query 口径

- API 路径

- 页面结构

- 快照与导出能力

而是必须直接复用前六批成果。\
若发现缺陷，应以回归问题形式指出，不得在第七批提示词中越界重写业务逻辑。

**3.3 必须优先保证可重复与可验证**

这一批的关键不是再增加功能，而是确保：

- 回归可重复执行

- 演示可重复复现

- 验收可重复判断

- 结果可留痕归档

**3.4 必须直接服务 160 文档的整体验收口径**

第七批必须直接落地 160 文档中已经定义的：

- 联调验证项

- 三条演示路径

- 缺陷分级

- 通过条件

- 验收清单

**3.5 必须便于后续版本继续复用**

第七批做完后，应直接成为后续：

- 每次版本升级回归

- 面向内部领导演示

- 面向客户方案演示

- 阶段性验收复核

的基础资产。

这里把 WP210 再拆成 Demo 和 Acceptance 两包，是因为：

- **Acceptance** 更偏向“怎么打勾、怎么记缺陷、怎么形成结论”

虽然都属于收口层，但职责和产出物不同，拆开后更利于 AI Coding 稳定生成与审阅收敛。

**五、第七批提示词包执行顺序建议**

建议第七批按以下顺序下发和合并：

**第一步：PP200A**

先把回归验证脚本、验证清单和结果结构固化下来。

**第二步：PP210A**

再把演示脚本和演示资料建立起来，确保演示基于已验证的系统。

最后建立正式验收资产，把缺陷、通过判定和归档一次性收束。

这样顺序最稳，因为：

- 演示必须建立在已通过基础回归验证之上

- 验收必须建立在演示可跑和缺陷可记的基础上

- 最终归档必须等到回归与演示材料已经形成
