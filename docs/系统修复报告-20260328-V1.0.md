# 系统修复与代码优化报告

**版本**: V1.0
**日期**: 2026-03-28

## 修复概览

本次修复主要针对标书管理系统中存在的三个关键问题进行了处理，涵盖了安全风险防范、项目权限校验控制以及代码可维护性优化。

---

### Issue 1: IP 地址获取逻辑安全风险修复
* **所在文件**: `src/app/api/auth/login/route.ts`
* **问题描述**: 原有 IP 地址获取逻辑直接采信 `x-forwarded-for` 头内容，存在被恶意客户端伪造的风险。
* **修复方案**: 改进了解析逻辑，通过分割 `x-forwarded-for` 字符串并提取最左侧的有效 IP 记录（去除多余空格），并补充了对原生 `request.ip` 属性的支持作为更底层的回退手段。这显著增强了获取真实客户端 IP 的可靠性与抗欺骗能力。

### Issue 2: 缺少项目权限验证修复
* **所在文件**: `src/app/api/bid/documents/route.ts`
* **问题描述**: 之前在获取项目标书文档列表以及创建新文档时，缺少对当前用户项目访问权限的检查。
* **修复方案**: 
  - 引入了 `hasProjectPermission` 方法，并封装出 `validateProjectAccess` 函数。
  - 在涉及文档查询和创建操作前，强制执行 `view` 权限的校验；若验证未通过，则抛出统一的 `403 Forbidden` 异常，拒绝越权访问。

### Issue 3: 冗余错误处理优化
* **所在文件**: `src/app/api/bid/documents/route.ts`
* **问题描述**: 路由处理方法（`getDocuments`、`createNewDocument`）内部使用了无意义的 `try-catch` 块结构（仅捕获后原样抛出）。
* **修复方案**: 移除了多余的 `try-catch` 包裹代码。让接口内的业务异常及验证错误自然冒泡，交由 Next.js 与统一的应用级错误处理器负责接管与返回格式化响应，进而提高了代码整洁度与可维护性。

### Issue 4: 项目ID解析缺少数字格式验证修复
* **所在文件**: `src/app/api/bid/documents/route.ts`
* **问题描述**: 获取标书文档列表接口在解析 URL 参数 `projectId` 时，直接使用了 `parseInt` 方法。由于未进行前置格式验证，如果传入包含非数字字符的恶意或错误输入，`parseInt` 将返回 `NaN`，这可能引发未定义的后续数据库查询或程序错误。
* **修复方案**: 增加正则表达式 `/^\d+$/.test(projectId)` 的前置验证，当输入不是全数字字符串时，直接通过应用级别的异常处理器（`AppError.badRequest`）抛出 "项目ID格式错误"，并且在调用 `parseInt` 时补充了 `10` 作为基数。这保证了向后传递的数据准确性和鲁棒性。

### Issue 5: 全局鉴权中间件异常吞没及错误码覆盖 BUG
* **所在文件**: `src/lib/auth/middleware.ts`、`src/lib/auth/jwt.ts`
* **问题描述**: 原 `withAuth` 及 `withResourcePermission` 中间件在执行内部 `handler` 时采用了宽泛的 `try-catch` 包裹，导致接口内部正常抛出的业务异常（如 400 校验错误）被中间件拦截，并被强行篡改为 401（"认证失败"）或 500（"权限检查失败"）返回给前端，掩盖了真实的业务错误。
* **修复方案**: 将 JWT 解析部分与业务 Handler 调用的 `try-catch` 分离。同时在 `middleware.ts` 及 `jwt.ts` 中引入全局的 `handleError` 及 `AppError` 进行统一处理，确保业务异常能正确向外冒泡并返回正确的 HTTP 状态码与提示信息。

### Issue 6: API 路由冗余的错误处理代码优化
* **所在文件**: `src/app/api/projects/route.ts`、`src/app/api/auth/login/route.ts`
* **问题描述**: 多个 API 路由内手动构建了 `NextResponse.json` 进行异常返回或做了冗余的 `try-catch`，导致代码重复度高，不符合系统统一的错误处理规范。
* **修复方案**: 去除了 `src/app/api/projects/route.ts` 中的 `try-catch`，让异常通过权限中间件交由 `handleError` 处理。在 `login` 路由中全面使用 `AppError` 替换手动的 `NextResponse`，并在外层统一通过 `handleError` 输出，极大提升了代码整洁度与规范性。

### Issue 7: 项目创建时软删除冲突验证与批量插入性能优化
* **所在文件**: `src/lib/project/service.ts`
* **问题描述**: `createProject` 接口在校验 `code` 唯一性时，允许复用被“软删除”的项目编码。但在数据库层面 `code` 字段存在全局唯一索引 (`unique`)，若尝试插入会导致数据库报错。此外，该接口在创建默认阶段时使用了 `for` 循环单条插入，导致数据库性能损耗。
* **修复方案**: 修正了项目编码唯一性的判断逻辑，若发现编码对应的是被软删除的项目，明确抛出冲突异常 (`AppError.conflict`)。同时，将 `defaultPhases` 及 `milestones` 的插入改为批量 `db.insert(...).values(Array)` 操作，并包裹在事务中执行，消除了潜在的 N+1 写入性能问题并保证了数据一致性。
