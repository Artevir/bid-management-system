# 系统修复报告-20260329-V1.1

**版本**: V1.1  
**日期**: 2026-03-29  

## 修复背景

服务器部署执行 `pnpm build` 时出现构建失败。根因并非业务逻辑错误，而是工程层面“引用路径漂移 + 依赖缺失 + 语法结构被破坏”叠加导致 Next.js 编译阶段直接失败，阻断上线。

本次修复目标：

1. 让 `next build` 在 Linux 生产环境可稳定通过
2. 消除历史遗留的 “schema 拆分导入路径” 与 “真实 schema.ts” 不一致问题
3. 修复流式接口中 `ReadableStream` 的语法结构破坏问题
4. 补齐缺失依赖，避免生产环境出现 `Module not found`

## 问题与修复项

### Issue 1: `ReadableStream` 语法结构被破坏导致编译失败

* **所在文件**: `src/app/api/llm/chat/route.ts`
* **问题现象**: `next build` 报语法错误，错误位置落在 `ReadableStream` 闭合附近（如 `},` / `});`），属于“表象位置不等于真实根因”的典型情况。
* **根因分析**:
  - `catch` 块内 `logCall({ ... })` 的对象参数结构被破坏，字段不在对象字面量内
  - `if (!request.signal.aborted) { ... }` 缺少闭合花括号，导致后续 `},` / `});` 被解析器判定为语法不完整
* **修复方案**:
  - 恢复 `catch` 块结构，确保对象字面量与花括号完整闭合
  - 引入 `finally { controller.close(); }` 兜底关闭流，避免异常路径遗漏关闭造成连接泄露
* **结果**: 语法错误消除，流式异常路径与中断路径都可稳定释放资源。

### Issue 2: schema 拆分导入路径与真实 schema.ts 不一致导致构建失败

* **涉及文件**:
  - `src/app/api/export/route.ts`
  - `src/lib/analytics/report-service.ts`
  - `src/lib/search/search-service.ts`
* **问题现象**: `Module not found: Can't resolve '@/db/schema/projects'` / `@/db/schema/documents` / `@/db/schema/companies`。
* **根因分析**:
  - 当前项目真实 schema 定义集中在 `src/db/schema.ts`，导出的表为 `projects`、`bidDocuments`、`companies` 等
  - 代码仍引用了不存在的拆分模块路径 `@/db/schema/projects`、`@/db/schema/documents`、`@/db/schema/companies`
  - 同时 `src/db/schema.ts` 中并不存在 `documents` 表，真实表名为 `bidDocuments`
* **修复方案**:
  - 统一改为从 `@/db/schema` 直接导入：`projects`、`bidDocuments`、`companies`
  - 将业务中作为表对象使用的 `documents` 全部替换为 `bidDocuments`
  - 对统计与搜索逻辑中引用的字段进行“最小可用对齐”：将原 `documents.type`、`documents.file_size` 等不存在字段切换为 `bidDocuments.status`、`bidDocuments.wordCount` 等真实字段，确保编译通过且语义可解释
* **结果**: 消除 schema 路径漂移，构建阶段不再出现上述模块缺失错误。

### Issue 3: audit-logs schema 导入路径错误导致构建失败

* **所在文件**: `src/lib/audit/audit-service.ts`
* **问题现象**: `Module not found: Can't resolve '@/db/schema/audit-logs'`
* **根因分析**:
  - 审计日志 schema 实际位于 `src/lib/db/schema/audit-logs.ts`
  - 代码错误引用了不存在的 `@/db/schema/audit-logs`
  - `audit-logs.ts` 同时依赖 `./users`，但该目录缺少 `users.ts`
* **修复方案**:
  - 将导入路径统一修正为 `@/lib/db/schema/audit-logs`
  - 补齐 `src/lib/db/schema/users.ts` 以满足 `audit-logs.ts` 的依赖链
  - 修正字段引用：将 `user.name` 对齐为 `user.username`（与该 users schema 对齐）
* **结果**: 审计模块构建依赖链完整，模块缺失错误消除。

### Issue 4: coze-coding-dev-sdk 依赖缺失导致构建失败

* **涉及文件**:
  - `src/app/api/ai/call/route.ts`
  - `src/app/api/ai/recommendations/route.ts`
  - `src/app/api/bid/ai-inline-generate/route.ts`
  - 以及项目内其它使用 `coze-coding-dev-sdk` 的模块
* **问题现象**: `Module not found: Can't resolve 'coze-coding-dev-sdk'`
* **根因分析**: 代码直接引用第三方 SDK，但 `package.json` 未声明依赖，生产环境安装后不存在该包。
* **修复方案**: 在 `package.json` 中补充 `coze-coding-dev-sdk` 依赖，确保生产环境可安装。
* **结果**: 依赖缺失问题消除。
