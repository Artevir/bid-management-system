# 系统修复报告-20260329-V1.14

**版本**: V1.14  
**日期**: 2026-03-29  

## 修复背景

服务器执行 `pnpm build` 在 TypeScript 类型校验阶段失败。错误来自认证中间件链将 handler 返回类型限制为 `NextResponse`，而业务路由中存在流式响应（原生 `Response`）的合法返回。

## Issue 1: 认证中间件链返回类型过窄，导致流式路由无法通过类型检查

* **涉及文件**:
  - `src/lib/auth/middleware.ts`
  - `src/app/api/bid/chapters/[id]/generate/route.ts`
* **问题现象**:
  - 报错：handler `Promise<Response>` 无法赋值给 `Promise<NextResponse>`
  - 表现为中间件调用处类型不兼容（`Response` 缺少 `NextResponse` 的 cookies / INTERNALS）
* **根因分析**:
  - `withAuth / withPermission / withResourcePermission / withChapterPermission` 等中间件将 handler 类型写死为 `Promise<NextResponse>`
  - 但 `NextResponse` 仅是 `Response` 的子类；对中间件而言应允许更通用的 `Response` 返回
  - 流式 API（SSE/文件下载等）通常返回原生 `Response`，属于 Next.js 支持的标准返回类型
* **修复方案**:
  - 将中间件链的签名统一放宽为 `Promise<Response>`：
    - `withAuth`
    - `withOptionalAuth`
    - `withPermission`
    - `withResourcePermission`
    - `withDocumentPermission`（闭包 handler 类型）
    - `withChapterPermission`（闭包 handler 类型）
    - `withAdmin`
  - 保持现有实现不变：`NextResponse.json(...)` 仍可作为 `Response` 返回
* **修复结果**:
  - 流式路由可直接返回 `createStreamResponse(...)` 的原生 `Response`
  - 非流式路由继续返回 `NextResponse.json(...)` 不受影响
  - 该类型墙问题不再在后续流式 API 中反复出现
