# 系统修复报告-20260329-V1.2

**版本**: V1.2  
**日期**: 2026-03-29  

## 修复背景

在服务器 `pnpm build` 阶段仍存在一批工程级“阻断上线”问题，主要集中在：

1. Next.js 15 路由处理器参数签名差异（动态路由 `params`）
2. 认证中间件导出不完整（历史路由仍在引用 `withAdmin / clearPermissionCache / requireAuth`）
3. 缓存模块导出不一致（大量调用方引用 `@/lib/cache` 的 `cache` / redis 相关导出，但入口文件未提供）

目标是让线上构建不再因这些“接口契约/导出缺失”问题失败，并减少后续维护的迁移成本。

## 修复项

### Issue 1: Next.js 15 动态路由 `params` 签名不兼容导致构建失败

* **问题现象**: 动态路由 API 文件使用 `{ params }: { params: { id: string } }` 形式，在 Next.js 15 的类型约束下出现签名不兼容。
* **修复方案**: 将动态路由处理器统一调整为 `params: Promise<...>` 并使用 `await params` 获取参数，保证类型与运行时兼容。
* **涉及文件**:
  - `src/app/api/admin/rbac/roles/[roleId]/route.ts`
  - `src/app/api/bid/chapters/[id]/generate/route.ts`
  - `src/app/api/bid/chapters/[id]/route.ts`
  - `src/app/api/bid/documents/[id]/route.ts`
  - `src/app/api/tenants/[tenantId]/route.ts`
  - `src/app/api/upload/[sessionId]/route.ts`

### Issue 2: 认证中间件导出缺失导致历史路由无法编译

* **问题现象**: 多个 API 路由仍引用 `@/lib/auth/middleware` 的 `withAdmin`、`clearPermissionCache`、`requireAuth`，但入口文件未提供对应导出，导致构建失败。
* **修复方案**:
  - 在 `src/lib/auth/middleware.ts` 增加 `requireAuth(request)` 兼容导出，返回 `{ user, error }` 结构，满足现有调用方。
  - 增加 `withAdmin(request, handler)` 兼容导出，内部复用 `withAuth` 并以角色等级作为管理员判定依据。
  - 增加 `clearPermissionCache(userId?)` 兼容导出，清理权限相关缓存，避免权限变更后缓存不一致。
* **涉及文件**:
  - `src/lib/auth/middleware.ts`

### Issue 3: 缓存模块入口导出不一致导致大量调用方编译失败

* **问题现象**:
  - 大量模块使用 `import { cache } from '@/lib/cache'`
  - 部分模块使用 `import redis, { RedisKeys, getRedisClient, isUsingMemoryFallback } from '@/lib/cache'`
  - 但 `src/lib/cache/index.ts` 未对外提供这些导出，导致构建失败。
* **修复方案**:
  - 在 `src/lib/cache/index.ts` 统一补齐 redis-client 的导出与默认导出，并提供 `cache` 作为 `redis` 的别名，兼容历史调用方。
  - 保留并继续支持 `memoryCache/getCacheStats/clearAllCache` 等内存缓存能力，避免影响健康检查与离线模块。
* **涉及文件**:
  - `src/lib/cache/index.ts`
