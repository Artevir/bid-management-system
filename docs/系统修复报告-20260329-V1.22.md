# 系统修复报告-20260329-V1.22

**版本**: V1.22  
**日期**: 2026-03-29  

## 修复背景

在 Next.js 15 的类型检查阶段，多个“二进制下载”接口容易因 `NextResponse` 的 body 类型约束而失败（`Buffer` / `Uint8Array` 不能直接作为 `BodyInit` / `BlobPart`）。

## Issue 1: DOCX 导出接口使用 `Uint8Array` 作为 `NextResponse` body 存在类型风险

* **所在文件**: `src/app/api/frameworks/export/route.ts`
* **问题现象**:
  - 文件导出分支使用 `new NextResponse(new Uint8Array(buffer), ...)` 返回二进制
  - 在严格的 `BodyInit` 类型约束环境下容易触发与 `Buffer/Uint8Array` 相关的类型错误
* **根因分析**:
  - `docx` 的 `Packer.toBuffer` 在 Node 环境返回 `Buffer`
  - `Buffer`/`Uint8Array` 在当前 TS lib/NextResponse 类型组合下不一定被视为合法 body
* **修复方案**:
  - 将 `buffer` 拷贝为标准 `Uint8Array`，再包装成 `Blob` 作为响应 body：
    - `const bytes = Uint8Array.from(buffer)`
    - `const body = new Blob([bytes], { type: DOCX_MIME })`
    - `return new NextResponse(body, { headers })`
* **修复结果**:
  - 降低“二进制导出接口”在 `pnpm build` 类型检查阶段的阻断风险
  - 导出下载行为保持不变（Content-Type/Content-Disposition 保持一致）
