# 系统修复报告-20260329-V1.23

**版本**: V1.23  
**日期**: 2026-03-29  

## 修复背景

在 Next.js 15 的 `pnpm build` 类型检查阶段，二进制下载接口容易因 `NextResponse` 的 body 类型约束发生阻断（`Buffer/Uint8Array` 不一定被识别为 `BodyInit`）。为避免后续“一个接口一个接口补洞”的低效模式，本轮对同类导出接口做批量收口。

## Issue 1: 二进制导出接口统一使用 `Blob` 作为响应 body

* **涉及文件**:
  - `src/app/api/bid/export/route.ts`
* **问题现象**:
  - 导出接口直接 `new NextResponse(Uint8Array|Buffer, ...)` 在严格类型约束环境中存在构建阻断风险
* **修复方案**:
  - 对 `Buffer/Uint8Array` 统一拷贝为标准字节数组再包装为 `Blob`：
    - `const bytes = Uint8Array.from(result.content)`
    - `const body = new Blob([bytes], { type: mimeType })`
    - `return new NextResponse(body, { headers })`
  - 对非二进制内容保持原语义（仍可作为 `BlobPart` 放入 `Blob`）
* **修复结果**:
  - 二进制下载接口在类型检查阶段更稳定，不再依赖 `Buffer/Uint8Array` 是否被 `BodyInit` 接受
  - 下载行为保持不变（Content-Type/Content-Disposition 仍由原逻辑控制）
